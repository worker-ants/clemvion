이 페이지에서는 데이터 흐름을 제어하는 로직 노드를 설명해요. 조건 분기부터 반복, 배열 가공, 변수 선언과 변경, 여러 경로를 합치는 Merge까지 다뤄요.

## If/Else

### 언제 써요

- 입력 데이터의 값에 따라 **두 갈래**로 분기하고 싶을 때 써요.
- 여러 조건을 AND/OR로 묶어 최종 분기를 결정할 수 있어요.

### 필드

<FieldTable rows={[
  {name: "conditions", required: true, type: "ConditionGroup[]", description: "평가할 조건 목록이에요. 각 항목은 `field` `operator` `value` 구조를 가져요.", default: "[]"},
  {name: "combineMode", required: false, type: "`and` / `or`", description: "조건 간 결합 방식이에요.", default: "and"},
  {name: "strictComparison", required: false, type: "boolean", description: "true면 타입 강제 변환 없이 엄격하게 비교해요.", default: "false"}
]} />

지원 연산자는 `eq`, `neq`, `gt`, `gte`, `lt`, `lte`, `contains`, `not_contains`, `starts_with`, `ends_with`, `is_empty`, `is_not_empty`, `regex`, `is_null`, `is_type`예요.

### 예시

<Example title="관리자 여부 분기">
```ts
// conditions
[
  { field: "{{ $input.role }}", operator: "eq", value: "admin" },
  { field: "{{ $input.age }}", operator: "gte", value: 18 }
]
// combineMode: "and"
// → 둘 다 참이면 `true` 포트, 아니면 `false` 포트로
```
</Example>

<Callout type="note">
입력 데이터는 어느 쪽 포트로 가든 **변형 없이 그대로** 전달돼요. 값을 가공하고 싶다면 Transform이나 Code 노드를 조합해요.
</Callout>

## Switch

### 언제 써요

- 하나의 값에 따라 **여러 경로**로 분기하고 싶을 때 써요.
- 값 매칭(`value` 모드) 또는 조건식(`expression` 모드) 중 선택할 수 있어요.

### 필드

<FieldTable rows={[
  {name: "mode", required: true, type: "`value` / `expression`", description: "케이스 매칭 방식이에요.", default: "value"},
  {name: "switchValue", required: false, type: "expression", description: "`mode=value`일 때 비교 기준 값이에요.", default: "-"},
  {name: "cases", required: true, type: "CaseDef[]", description: "케이스 목록이에요. 각 케이스는 `label`, `value`(value 모드) 또는 `condition`(expression 모드)을 가져요.", default: "[]"},
  {name: "hasDefault", required: false, type: "boolean", description: "true면 어느 케이스에도 매칭되지 않을 때 `default` 포트로 보내요.", default: "false"},
  {name: "strictComparison", required: false, type: "boolean", description: "true면 타입 강제 변환 없이 엄격 비교해요.", default: "false"}
]} />

### 예시

<Example title="문의 유형별 경로">
```ts
// switchValue: "{{ $input.type }}"
// cases:
[
  { label: "결제", value: "billing" },
  { label: "기술", value: "tech" },
  { label: "일반", value: "general" }

// → 매칭 안되면 `default` 포트
```
</Example>

<Callout type="tip">
케이스 포트 ID는 UUID v4로 고정돼 있어요. 라벨을 바꾸거나 순서를 재정렬해도 연결선이 그대로 유지돼요.
</Callout>

## Loop

### 언제 써요

- 정해진 **횟수**만큼 같은 로직을 반복할 때 써요.
- 반복 중에 특정 조건이 되면 조기 종료하고 싶을 때 `breakCondition`을 써요.

### 필드

<FieldTable rows={[
  {name: "count", required: true, type: "expression", description: "반복 횟수예요. 정수 리터럴이나 표현식 모두 가능해요.", default: "1"},
  {name: "maxIterations", required: false, type: "integer", description: "안전 상한이에요. 초과 시 에러로 중단해요.", default: "1000"},
  {name: "breakCondition", required: false, type: "ConditionGroup", description: "이 조건이 충족되면 반복을 조기 종료해요.", default: "-"}
]} />

Loop는 컨테이너 노드예요. 입력은 `in`, `emit` 두 개, 출력은 `body`, `done` 두 개를 가져요. body 안에서 반복을 수행하고, 반복 결과는 `emit`에 연결된 노드의 출력을 수집해 배열로 `done`에 모아요.

### 예시

<Example title="3번 반복하며 API 호출">
```ts
// count: "3"
// body 안에서 $loop.index (0,1,2), $loop.isFirst, $loop.isLast 사용 가능
// emit에 HTTP Request 1개 연결 → done으로 [결과0, 결과1, 결과2] 배열
```
</Example>

<Callout type="warn">
컨테이너의 `emit` 포트에는 **정확히 1개의 body 노드**만 연결해야 해요. 여러 개 또는 0개를 연결하면 `CONTAINER_MISSING_EMIT` 또는 `CONTAINER_MULTIPLE_EMIT`로 실행 실패해요. body 내부에 순환(back-edge)이나 blocking 노드(Form, 버튼 블로킹, AI 대화)는 허용되지 않아요.
</Callout>

## ForEach

### 언제 써요

- 배열의 **각 항목마다 body 서브그래프**를 순차 실행할 때 써요.
- Map과 실행 모델이 같지만, 시맨틱이 "각 항목에 대한 처리(사이드 이펙트 중심)"에 맞춰져 있어요.

### 필드

<FieldTable rows={[
  {name: "arrayField", required: true, type: "expression", description: "대상 배열 경로예요. dot-path(`items`, `order.items`) 또는 inline 표현식(`{{ $var.list }}`) 모두 허용해요.", default: "-"},
  {name: "errorPolicy", required: false, type: "`stop` / `skip` / `continue`", description: "항목 실행 중 에러가 났을 때의 정책이에요. 결과 배열은 원본 인덱스를 유지해요.", default: "stop"}
]} />

body 내부에서 `$item`, `$itemIndex`, `$item.isFirst`, `$item.isLast`를 참조할 수 있어요.

### 예시

```ts
// body 내부에서 $item.value, $itemIndex 참조
```

## Filter

- Transform의 `array_filter`는 단일 조건의 인라인 연산이에요. 다중 조건과 분기가 필요하면 Filter 노드를 써요.

- 여러 입력이 모두 도착하길 기다리거나, 가장 먼저 도착한 값만 통과시킬 때 유용해요.

### 필드

<Example title="Parallel 분기를 커스텀 전략으로 합치기">
```ts
// Parallel(3)의 각 분기 마지막 노드를 Merge의 in 포트에 연결
```

</Callout>

## Parallel

### 언제 써요

- 여러 작업을 **동시에** 실행해 전체 시간을 줄이고 싶을 때 써요. 예: 여러 API 호출을 동시에 보내기.

### 필드

<FieldTable rows={[
  { name: "Branch Count", required: true, type: "number", description: "동시에 실행할 분기 수예요. 2~16 사이로 지정해요.", default: "2" },
  { name: "Max Concurrency", required: false, type: "number", description: "동시에 실행할 최대 분기 수예요. 0이면 전부 동시에, 1~16이면 그 수만큼만 동시에 실행해요.", default: "0 (무제한)" }
]} />

### 포트

- **입력**: `in` — 모든 분기에 동일한 입력이 복제 전달돼요.
- **분기 출력**: `branch_0`, `branch_1`, … — Branch Count에 따라 동적으로 생성돼요. 각 분기의 시작 노드를 여기에 연결해요.
- **done 출력**: `done` — 모든 분기가 완료된 뒤 결과를 모아서 내보내요. 출력 형태는 `{ branches: [분기0결과, 분기1결과, ...] }` 배열이에요.

### 동작

- 입력이 들어오면 `branch_0`, `branch_1`, … 모든 분기 포트로 **동일한 입력을 동시에** 내보내요.
- `PARALLEL_ENGINE=v1` 환경변수가 설정되면 각 분기가 **진짜 병렬로** 실행돼요. 설정하지 않으면 기존과 동일하게 순차 실행해요.
- Max Concurrency를 지정하면 리소스를 보호하면서 동시 실행 수를 제한할 수 있어요.
- 모든 분기가 완료되면 각 분기의 마지막 노드 출력이 `done` 포트의 `branches` 배열로 수집돼요.

### 사용 패턴

**패턴 A — done 포트로 바로 이어가기 (간단)**

분기 결과를 배열로 그대로 쓸 때 적합해요. Merge 노드가 필요 없어요.

```
Parallel(3)
  ├── branch_0 → [API 호출 A]
  ├── branch_1 → [API 호출 B]
  ├── branch_2 → [API 호출 C]
  └── done → 결과 처리 노드
               ↑ { branches: [A결과, B결과, C결과] }
```

**패턴 B — Merge로 결과 재구성 (고급)**

객체 병합, 첫 도착값만 사용 등 전략이 필요하면 각 분기의 마지막 노드를 Merge에 직접 연결해요.

```
Parallel(2)
  ├── branch_0 → [A] ──┐
  ├── branch_1 → [B] ──┼→ Merge(merge_object) → 처리
  └── done (미연결)
```

두 패턴을 동시에 사용할 수도 있어요 — `done`으로는 전체 결과를, Merge로는 가공된 결과를 받아 서로 다른 후속 처리를 연결할 수 있어요.

### 예시

<Example title="3개 API를 동시 호출하고 결과 수집">
```ts
// Branch Count: 3, Max Concurrency: 0
// branch_0 → HTTP Request (서비스 A)
// branch_1 → HTTP Request (서비스 B)  
// branch_2 → HTTP Request (서비스 C)
// done → Transform
// Transform에서 {{ $input.branches[0].data }} 등으로 각 분기 결과 참조
```
</Example>

<Callout type="warn">
Parallel 분기 안에서는 **Form, 버튼 블로킹, AI 대화(Multi Turn)** 같은 블로킹 노드를 사용할 수 없어요. 순환 연결(back-edge)과 Parallel 안에 또 다른 Parallel을 넣는 중첩도 현재는 지원하지 않아요.
</Callout>

<Callout type="tip">
하나의 출력 포트에서 여러 연결선을 끌어내는 1:N 분기와의 차이는 **의도 표현**이에요. Parallel은 "이 분기들은 동시에 실행한다"라는 의미를 노드 자체로 드러내, 워크플로우를 읽는 사람이 의도를 바로 파악할 수 있어요.
</Callout>

## Background

오래 걸리는 후속 작업(이메일 발송, 외부 API 알림 등)을 메인 흐름에서 분리하고 싶을 때 사용해요. Background 노드는 입력이 도착하면 **`main` 포트로 즉시** 다음 노드를 진행시키고, 동시에 **`background` 포트로 연결된 본문 분기**를 별도 워커(BullMQ 큐 `background-execution`)에서 실행해요.

- 본문 실행 컨텍스트는 enqueue 시점의 변수·노드 출력 **스냅샷**으로 시작해요. 그래서 본문 실행 중 메인이 변수를 바꿔도 본문에는 반영되지 않아요(역도 마찬가지).
- 본문 실행이 실패해도 메인 워크플로우 상태는 그대로 유지돼요. NodeExecution 이력에는 실패가 남아 타임라인에서 확인할 수 있어요.
- 본문 결과는 메인으로 다시 흘러들어오지 않아요(fire-and-forget).

</Callout>

## 팁 & 참고

- 표현식 안에서 배열과 객체를 다루는 방법은 [표현식 치트시트](/docs/03-expression-language/cheatsheet)에서 확인해요.
