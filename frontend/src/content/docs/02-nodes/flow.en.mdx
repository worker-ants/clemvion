This page explains how to call another workflow as a sub-workflow using the **Workflow node**. It's useful when you want to split a reusable sub-flow out for separate management, or break a long pipeline into manageable stages.

## Workflow

### When to use it

- To **reuse common logic** across multiple workflows.
- To **split a long flow into modules** for better readability.
- To **fire and forget** another workflow asynchronously while your main flow continues.

### Ports

One input `in` and one output `out`. In sync mode the sub-workflow's final output is emitted; in async mode, invocation info (`executionId`, etc.) is emitted instead.

### Sync mode behavior

1. Compose the input via `inputMapping`. When empty, the parent input is passed through as-is.
2. Load the target workflow's nodes and connections and **run them inline inside the parent Execution**.
3. Sub-workflow nodes share the parent's `executionId`, node-result cache, and node map.
4. In the run history, sub-workflow nodes appear right after the parent node on the timeline.
5. The sub-workflow's final output leaves through the `out` port.
6. The sub-workflow's trigger node is auto-skipped and simply passes the input data along.

### Async mode behavior

### Examples

<Example title="Sync mode — reuse a shared pipeline">
```ts
// config
{
  "workflowId": "b6c7…-…-…",
  "workflowName": "Data Pipeline",
  "mode": "sync",
  "inputMapping": [
    { "target": "userId", "source": "{{ $input.user.id }}" },
    { "target": "event",  "source": "{{ $input.type }}" }
  ],
  "timeout": 300
}
```
</Example>

<Example title="Async mode — fire a notification">
```ts
// config
{
  "workflowId": "{{ $var.notifyWorkflowId }}",
  "mode": "async",
  "inputMapping": [
    { "target": "channel", "source": "{{ $input.channel }}" }
  ]
}

// out port output
{
  "executionId": "9f2c-…-…",
  "workflowId": "…",
  "status": "started"
}
```
</Example>

<Callout type="warn">
If the target workflow is deleted or deactivated, the settings panel shows a warning, and the canvas summary displays a `⚠ Missing workflow` badge.
</Callout>

<Callout type="tip">
The dropdown only lists workflows in the same workspace and excludes the workflow you're currently editing. Inactive workflows appear with an `(inactive)` label.
</Callout>

## Canvas summary

Displayed as `{workflowName or workflowId} · {mode}`. When a name is available it shows the name; otherwise the ID. Example: `Data Pipeline · sync`.

## Tips & notes

- When you invoke without an input mapping, the parent input is passed through whole. To forward only specific parameters, use `inputMapping`.
- Results and logs of a sub-workflow are also visible in [Run results](/docs/04-run-and-debug/run-results).
- Power-user tip: in sync mode the sub-workflow executes inside the same run, so you can reference its results right from the parent with `{{ $node["Sub Workflow"].output.field }}`, and the run timeline stays a single continuous line — great for debugging.
