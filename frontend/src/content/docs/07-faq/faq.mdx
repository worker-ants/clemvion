이 페이지에서는 사용자분들이 자주 묻는 질문을 한 자리에 모아 답을 드려요. 각 항목의 링크를 타고 가면 더 자세한 설명이 나와요.

## Q1. 워크플로우와 실행 데이터는 어디에 저장돼요?

워크플로우 구조·노드 설정·실행 이력은 제품의 관리형 데이터베이스에 저장돼요. Form 노드 첨부 파일 같은 업로드 자산은 오브젝트 스토리지에, 지식 저장소의 임베딩은 벡터 저장소에 분리되어 들어가요. 셀프 호스팅 환경에서는 같은 역할을 로컬 인프라로 대체해서 배포해요.

## Q2. 실행이 실패하면 어떻게 원인을 확인해요?

실행하면 캔버스 하단에 **Run Results 드로어**가 열려요. 실패한 노드는 빨간 테두리로 표시되고, 해당 노드를 클릭하면 `Error` 탭이 우선 선택돼 에러 메시지·스택·직전 입력을 보여줘요. 더 긴 이력을 보려면 [실행 결과 보기](/docs/05-run-and-debug/run-results)와 [에러 다루기](/docs/05-run-and-debug/error-handling)를 참고해요.

## Q3. 표현식에서 `undefined is not an object` 같은 에러가 떠요.

이전 노드의 결과 구조가 예상과 달라서 나오는 에러예요. `{{ $input.data.title }}`처럼 **경로 전체가 항상 존재**하는지 확인해요. 안전하게 쓰려면 삼항 조건 `{{ $input.data ? $input.data.title : "기본값" }}`을 써요. 문법과 함수 목록은 [표현식 치트시트](/docs/04-expression-language/cheatsheet)에 있어요.

## Q4. 노드를 잠시 꺼 두고 싶어요.

노드를 선택한 뒤 우클릭 메뉴에서 **비활성화**를 고르거나, 설정 패널의 토글을 꺼요. 비활성화된 노드는 실행 시 `skipped` 상태로 건너뛰고, 출력은 입력을 그대로 전달하지 않아요. 연결선은 유지되니 다시 켜면 곧바로 흐름에 복귀해요.

## Q6. LLM을 쓰는 AI 노드는 비용이 얼마나 나와요?

비용은 사용한 프로바이더(OpenAI, Anthropic 등)의 **토큰 과금**을 그대로 따라요. 제품은 요청을 대신 라우팅만 해요. 얼마나 썼는지는 통계 메뉴에서 **LLM 토큰 사용량**으로 확인할 수 있어요. 비용을 줄이려면 프롬프트를 짧게 유지하고, 가능한 곳에서는 Text Classifier처럼 구조화된 AI 노드를 써요.

## Q7. 한 워크플로우가 동시에 여러 번 실행돼도 괜찮아요?

괜찮아요. 실행 엔진은 **큐 기반 워커 풀** 구조라 여러 실행이 병렬로 진행돼요. 각 실행은 고유한 `executionId`를 가지고 독립된 컨텍스트에서 돌아가요. 다만 같은 행을 동시에 갱신(Database Query)하거나 같은 외부 리소스에 동시에 쓰는(HTTP Request) 경우처럼 **외부 리소스에서의 경합**은 제품이 자동으로 직렬화해 주지 않아요. 충돌 가능성이 있는 흐름이라면 외부 시스템의 트랜잭션·고유 키·낙관적 잠금(예: `version` 필드 비교)을 워크플로우 안에서 명시적으로 다뤄 주세요.

## Q8. 동시 실행 수에 한계가 있어요?

SaaS에서는 요금제에 따라 **동시 실행 한도**와 분당 실행 수가 정해져 있어요. 셀프 호스팅에서는 워커 인스턴스 수를 늘려 수평 확장해요. 상한을 넘은 요청은 큐에서 `pending`으로 대기하다가 차례가 오면 `running`으로 넘어가요.

## Q9. 실행 중간에 멈추고 싶어요.

실행 중에 헤더의 Run 버튼이 `Stop`으로 바뀌어요. 누르면 현재 실행을 `cancelled` 상태로 표시해요. 이미 호출된 외부 API는 되돌릴 수 없다는 점을 기억해요. 복구가 필요한 경우 `error` 포트를 이용한 보상 로직을 미리 넣어 둬요.

## Q10. 이전 버전으로 되돌릴 수 있어요?

네. 에디터 헤더의 더보기(⋮) 메뉴에서 **버전 히스토리**를 열면 이전 스냅샷 목록이 나와요. 원하는 시점으로 복원하거나 차이를 비교할 수 있어요. 자세한 흐름은 [버전 히스토리](/docs/05-run-and-debug/version-history)에 있어요.

## Q11. 외부에서 호출할 API Key는 어디서 만들어요?

사이드바의 `Authentication` 메뉴에서 만들어요. API Key, Bearer Token, Basic Auth 같은 방식을 발급할 수 있고, 특정 워크플로우의 Webhook 트리거에 연결해 접근을 제한할 수 있어요. 자세한 내용은 [통합 관리](/docs/06-integrations-and-config/integration-management)를 참고해요.

## Q12. 워크플로우를 다른 계정으로 옮기고 싶어요.

워크플로우 목록의 더보기 메뉴에서 **내보내기(JSON)**를 써요. 받은 JSON 파일을 다른 계정에서 **가져오기**로 복원하면 노드 구조와 설정이 그대로 옮겨져요. 단, Integration의 실제 인증 정보(API Key 등)는 보안상 포함되지 않으니, 옮긴 쪽에서 다시 연결해 줘요.

## Q13. 노드 실행 시간이 30초를 넘어가요.

기본 노드 타임아웃은 **30초**예요. HTTP Request 노드처럼 `timeout` 필드를 지원하는 노드는 설정으로 늘릴 수 있어요. 오래 걸리는 작업은 외부 작업 큐로 위임하거나, Webhook으로 결과만 받는 비동기 패턴으로 분리하는 것을 권해요.

## Q14. 표현식에서 반복문을 쓸 수 있어요?

표현식 언어는 **읽기 전용·무한 루프 금지**로 설계돼 있어요. 반복이 필요하면 `Loop`, `ForEach`, `Map` 같은 Logic 컨테이너 노드를 써요. 복잡한 변환은 Transform이나 Code 노드로 옮기는 것이 안전해요.

## Q15. 개인 데이터와 팀 데이터가 섞이지 않나요?

각 워크플로우는 **워크스페이스** 단위로 격리돼요. 가입 시 자동 생성되는 개인 워크스페이스가 기본이고, 팀 워크스페이스로 확장해 역할 기반 권한(RBAC)으로 공유할 수 있어요. 어떤 워크스페이스에서 작업 중인지는 사이드바 하단 사용자 영역에서 확인할 수 있어요.

## Q16. Authentication 메뉴와 LLM Config 메뉴는 뭐가 달라요?

방향이 정반대예요. **Authentication**은 외부 시스템이 내 워크플로우의 트리거나 API를 **호출할 때 쓰는 인증 방식**(API Key·Bearer Token·Basic Auth)을 발급·관리해요. **LLM Config**는 내 AI 노드가 **호출할 LLM 프로바이더**(OpenAI·Anthropic·Local 등)와 기본 모델·파라미터를 보관해요. "들어오는 요청"과 "나가는 호출"이라는 다른 흐름을 다루기 때문에 메뉴가 분리돼 있어요. 자세한 흐름은 각각 [통합 관리](/docs/06-integrations-and-config/integration-management)와 [LLM 설정](/docs/06-integrations-and-config/llm-config)에 있어요.

## Q17. Knowledge Base의 Vector 모드와 Graph 모드는 어떻게 골라요?

기본은 **Vector**예요. 청크 임베딩 + 코사인 유사도 검색이라 빠르고 비용이 낮아 일반 FAQ·문서 검색에 충분해요. **Graph 모드**는 임베딩에 더해 LLM이 문서에서 **entity와 relation**까지 추출해 그래프 + 벡터를 같이 검색해요. 인물·조직·사건처럼 관계가 답변 품질을 좌우하는 문서에 강하지만, 임베딩 단계에 LLM 비용·시간이 더 들어요. 모드는 컬렉션 생성 시 결정되고 변경하려면 새 컬렉션을 만들어 옮겨야 해요. 자세한 비교는 [지식 저장소 — Vector 모드 vs Graph 모드](/docs/06-integrations-and-config/knowledge-base)에 있어요.

## Q18. AI Agent가 외부 도구(예: 사내 검색)를 호출하게 하려면 어떻게 해요?

**워크플로우 내부 노드**를 도구로 노출하고 싶다면 캔버스 **Tool Area**에 노드를 끌어다 놓아요. **외부 시스템의 도구**(사내 검색 엔진, Slack, GitHub 등)는 **MCP 서버**로 연결해요. `/integrations`에서 MCP 서버를 등록하고, AI Agent의 `MCP Servers` 필드에 추가하면 LLM이 자동으로 그 서버의 도구를 호출할 수 있어요. 자세한 흐름은 [MCP 서버 통합](/docs/06-integrations-and-config/mcp-servers).

## Q19. AI 어시스턴트와 AI Agent 노드는 뭐가 달라요?

이름이 비슷하지만 위치와 역할이 달라요.

- **AI 어시스턴트**는 **에디터 안의 채팅 도우미**예요. `Cmd/Ctrl + /` 로 패널을 열고 자연어로 요청하면, 캔버스에 노드와 엣지를 직접 추가·수정해 줘요. 워크플로우 자체를 *만들기 위한* 기능이에요. 자세한 사용법은 [AI 어시스턴트 개요](/docs/03-workflow-editor/overview).
- **AI Agent 노드**는 **워크플로우 안에서 실행되는 LLM 호출 단위**예요. 캔버스에 직접 올려 두고 시스템 프롬프트·사용자 프롬프트·RAG·도구를 설정해 두면, 실행 시 매번 LLM 을 호출해 답을 내요. 워크플로우의 *부품*이에요. 필드 설명은 [AI 노드](/docs/02-nodes/ai).

어시스턴트가 만들어 주는 워크플로우 안에 AI Agent 노드가 포함될 수 있다는 점에서 둘은 보완 관계예요.
