## Stop Workflow

기본 정책이에요. 에러 발생 즉시 실행을 멈춰요.

- `Execution.status = "failed"`로 기록돼요.
- 실패 노드가 캔버스에 빨간 하이라이트로 표시돼요.
- 실패 노드의 하류는 실행되지 않아요.

**언제 쓰나요**: 잘못된 데이터가 뒤로 흘러 나가면 절대 안 되는 결정적 경로에 어울려요. 예를 들어 결제 승인 흐름에서 인증 노드가 실패했을 때처럼요.

## Skip Node

실패한 노드를 무시하고 다음 노드로 진행해요.

- `NodeExecution.status = "skipped"`가 되고, 에러 정보는 `NodeExecution.error` 필드에 보존돼요.
- 출력은 `null`이에요. 다음 노드는 `null`을 입력으로 받아요.

<Callout type="warn">다음 노드가 `null` 입력을 견디지 못하면 연쇄 실패가 날 수 있어요. `null` 체크나 `||` 기본값과 함께 쓰는 걸 권장해요.</Callout>

**언제 쓰나요**: 부가적인 알림 전송(Slack, 이메일) 같은 비필수 단계에서 써요. 실패해도 본 흐름은 계속되어야 할 때요.

## Use Default Output

에러 발생 시 **미리 설정한 기본 출력값**을 대신 내보내요.

설정 패널에서 `Error Handling`을 `Use Default Output`으로 바꾸면 JSON 에디터가 나타나요.

<Example title="Default Output 값 예시">
```json
{
  "result": null,
  "status": "fallback"
}
```
</Example>

- 기본값을 직접 설정하지 않으면 노드 출력 타입에 맞는 자동 기본값이 쓰여요.

- `NodeExecution.status`는 `completed`로 기록되지만 경고가 함께 남아요.

**언제 쓰나요**: 외부 API가 간헐적으로 실패할 수 있지만 **안전한 기본값**으로 흐름을 이어가도 문제없는 경우에요. 예: 캐시 조회 실패 시 빈 객체를 다음 단계에 전달.

## Retry

설정한 횟수만큼 자동으로 재시도해요.

<FieldTable rows={[
  { name: "maxRetries", type: "Integer", description: "최대 재시도 횟수예요. 기본값 3." },
  { name: "retryInterval", type: "Integer (ms)", description: "최초 재시도 간격이에요. 기본값 1000ms." },
  { name: "backoffMultiplier", type: "Float", description: "지수 백오프 배수예요. 기본값 2.0." },
  { name: "maxInterval", type: "Integer (ms)", description: "재시도 간격 상한이에요. 기본값 30000ms." }
]} />

재시도 간격은 아래 공식으로 계산돼요.

<Example title="재시도 간격 계산">
```ts
interval = min(retryInterval * backoffMultiplier ^ attempt, maxInterval)
```
</Example>

모든 재시도가 실패하면 **Stop Workflow로 폴백**돼요.

**언제 쓰나요**: 네트워크 흔들림이나 LLM Rate Limit처럼 **일시적**으로 실패할 가능성이 있는 외부 호출에 적합해요.

## Route to Error Port

에러 발생 시 에러 데이터를 노드의 **error 포트**로 보내고, 연결된 다음 노드가 에러 데이터를 입력으로 받아 복구 흐름을 이어가게 해요.

- 정책을 선택하면 노드 우하단에 빨간 원(●) **error 포트**가 동적으로 생겨요.
- error 포트에 연결선이 연결되어 있지 않으면 `ERROR_PORT_FALLBACK` 로그와 함께 Stop Workflow로 폴백돼요.
- `NodeExecution.status`는 `failed`로 기록되지만, **Execution 전체는 계속 진행**돼요.

에러 포트로 전달되는 데이터 구조예요.

<Example title="Error 포트 전달 페이로드">
```json
{
  "error": {
    "code": "NODE_EXECUTION_FAILED",
    "message": "LLM connection timeout",
    "nodeId": "uuid-of-failed-node",
    "nodeType": "ai_agent",
    "timestamp": "2026-04-16T12:00:00.000Z",
    "details": { },
    "originalInput": { }
  }
}
```
</Example>

**언제 쓰나요**: 실패했을 때 **별도 복구 경로**(예: 관리자 알림, 대체 LLM 호출, 원본 입력 저장)를 태우고 싶을 때 써요.

## 팁 & 참고

- 워크플로우 단위 자동 재시도(Trigger/Schedule 기반) 설정은 워크플로우 설정에서 따로 지정해요. 최대 5회까지 가능해요.
- 에러 코드 목록은 [실행 결과](/docs/05-run-and-debug/run-results)의 "에러 메시지 해석" 섹션을 참고해요.
- 노드의 공통 설정과 에러 포트 동작 상세는 [Overview](/docs/02-nodes/overview)에 있어요.
- 노드가 실패하지 않게 **입력을 미리 검증**하는 패턴은 [표현식 치트시트](/docs/04-expression-language/cheatsheet)의 `isEmpty`, `||` 기본값 예시를 참고해요.
