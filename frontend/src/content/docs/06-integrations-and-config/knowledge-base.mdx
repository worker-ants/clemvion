이 페이지에서는 지식 저장소(Knowledge Base)를 만들고 문서를 업로드해서 AI Agent의 RAG(Retrieval-Augmented Generation)에 활용하는 방법을 설명해요.

## 지식 저장소 구조

지식 저장소는 컬렉션(Collection) 단위로 관리해요. 컬렉션마다 임베딩 모델과 청크 전략을 고정하고, 그 아래에 문서를 올려요. 업로드된 문서는 자동으로 청크 분할 → 임베딩 → 벡터 저장소 저장을 거쳐요.

<Callout type="note">임베딩 모델은 LLM 설정의 프로바이더 중 임베딩을 지원하는 항목에서만 고를 수 있어요. 프로바이더를 먼저 등록하려면 [LLM 설정](/docs/06-integrations-and-config/llm-config) 문서를 참고해 주세요.</Callout>

## 컬렉션 만들고 문서 업로드하기

<Steps>
<Step>`/knowledge-bases` 에서 `[+ New Collection]` 을 눌러 이름 · 설명 · 임베딩 모델 · 청크 파라미터를 입력해요.</Step>
<Step>목록에서 새 컬렉션을 클릭해 상세 화면으로 들어가, `[Upload Documents]` 버튼을 누르거나 파일을 드래그 앤 드롭해요.</Step>
<Step>지원 형식은 `.txt`, `.md`, `.pdf`, `.csv` 에요. 여러 파일을 한 번에 올릴 수 있고, 업로드 즉시 임베딩이 시작돼요.</Step>
<Step>각 문서의 상태 배지가 `🔄 Processing` 에서 `✅ Ready` 로 바뀌면 검색에 사용할 수 있어요. 상태는 WebSocket으로 실시간 갱신돼요.</Step>
<Step>실패한 문서는 `❌ Error` 로 표시돼요. 항목의 ⋮ 메뉴에서 `재임베딩` 을 눌러 파이프라인을 다시 실행할 수 있어요.</Step>
</Steps>

<Callout type="warn">CSV는 행 단위로 독립 청크가 생성돼요. 헤더가 없으면 각 셀의 의미가 흐려지므로, 첫 줄에 컬럼명을 두는 편이 검색 품질에 유리해요.</Callout>

## Vector 모드 (현재) · Graph 모드 (로드맵)

컬렉션의 `RAG 모드`는 문서 처리 파이프라인과 검색 방식을 정해요. 모드는 컬렉션 생성 후에 바꿀 수 없으므로, 모드를 변경하려면 새 컬렉션을 만들어 문서를 다시 업로드해요.

<Callout type="warn">
**현재 활성화된 모드는 `vector` 뿐이에요.** `graph` 모드는 entity·relation 추출과 그래프 검색을 결합한 차세대 RAG 로 로드맵 단계예요. 컬렉션 생성 화면에서 graph 옵션은 비활성 상태로 노출되거나 숨겨질 수 있어요. 일반 출시 일정은 PRD 9 (Graph RAG) 와 `spec/5-system/10-graph-rag.md` 를 참고해 주세요.
</Callout>

<FieldTable rows={[

]} />

## AI Agent에서 참조하기

AI Agent 노드의 설정 패널에 `Knowledge Bases` 멀티 선택 필드가 있어요. 선택한 컬렉션은 실행 시 사용자 프롬프트를 기준으로 유사도 검색되고, 결과가 시스템 프롬프트 하단에 주입돼요.

<Example title="AI Agent 노드의 KB 참조 설정">
```ts
{
  "type": "ai_agent",
  "config": {
    "llmConfigId": "llm_01H...",
    "systemPrompt": "You are a customer support assistant.",
    "userPrompt": "{{ $input.parameters.question }}",
    "knowledgeBases": [
      "kb_01H...FAQ",
      "kb_01H...POLICY"
    ],
    "ragTopK": 5,           // 최대 결과 수 (기본 5)
    "ragThreshold": 0.7     // cosine 유사도 하한 (기본 0.7)
  }
}
```
</Example>

실행이 끝나면 AI Agent의 노드 결과에 RAG 출처 메타가 함께 들어와요. 어떤 문서의 어떤 청크가 참조됐는지, 점수는 얼마였는지 확인할 수 있어요. 표현식으로도 꺼내 쓸 수 있어요. 예를 들어 `{{ $node["AI Agent"].meta.ragSources[0].documentName }}` 처럼요.

<Callout type="tip">검색 결과가 `threshold` 를 넘지 못하면 RAG 없이 LLM이 호출돼요. 답변 품질이 떨어지면 임계값을 0.6 수준으로 낮추거나, 청크 크기를 줄여 재임베딩해 보세요.</Callout>

## 팁 & 참고

- 여러 컬렉션을 동시에 선택하면, 동일한 임베딩 모델을 쓰는 KB들은 한 번의 쿼리로, 서로 다른 모델을 쓰는 KB들은 각각 별도 임베딩 후 결과를 병합해요.
- 주입되는 컨텍스트는 모델 컨텍스트 윈도우의 50%를 넘지 않도록 자동으로 잘려요. 점수가 낮은 청크부터 제거되므로 상위 결과는 항상 포함돼요.
- 문서 미리보기에서 생성된 청크 목록을 하나씩 확인할 수 있어요. 특정 청크가 이상하면 원문을 다듬고 `재임베딩` 을 돌려 주세요.
- 임베딩 모델을 바꾸고 싶다면 새 컬렉션을 만들어 문서를 옮기는 편이 안전해요. 기존 컬렉션의 모델을 바꾸면 모든 청크가 재생성돼요.
- 표현식과 노드 결과에 대한 기본은 [변수와 컨텍스트](/docs/04-expression-language/variables-and-context)에서 설명해요.
- RAG 검색 쿼리·거리 함수·에러 처리 규칙은 `spec/5-system/9-rag-search.md` 에서 확인할 수 있어요.
