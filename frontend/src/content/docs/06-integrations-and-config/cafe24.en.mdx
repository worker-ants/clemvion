This page walks you through registering a Korean e-commerce SaaS **Cafe24** mall as an integration and using it from both workflow nodes and the AI Agent.

## Why use the Cafe24 integration?

- **One credential, two surfaces** — a single Cafe24 integration is reused by (a) the `Cafe24` workflow node and (b) the AI Agent's MCP tool list. Token refresh, leaky-bucket rate limits, and activity logs are unified across both call paths.
- **All 18 Admin API categories** — Product, Order, Customer, Category, Promotion, Mileage, ... every endpoint of Cafe24 Admin API is reachable through a single metadata-driven node. Most operations are two dropdowns away (Resource → Operation).
- **Natural-language AI calls** — through an in-process Internal MCP Bridge, the LLM can translate commands like *"give me last week's unshipped orders"* into the right endpoint call automatically.

## Create the integration (`/integrations`)

### 1) Pick Cafe24 in the Add Integration modal

`/integrations` → **+ Add Integration** → click the **🛒 Cafe24** card → routed to `/integrations/new?service=cafe24&step=auth`.

### 2) (Private apps only) Create the app on Cafe24 Developers first

To finish Step 2 with App Type = **Private** you need a `client_id` / `client_secret` from Cafe24 itself. Cafe24 has no explicit "public/private" toggle — **any app that isn't published to the App Store is automatically private** (only your own mall can install it).

1. Sign in to https://developers.cafe24.com with the Cafe24 mall owner account.
2. Top menu **`Apps`** → **`My Apps`** (a.k.a. "앱 관리").
3. Click **`ADD PRODUCT` / `Create app`** → choose **`Web`**.
4. Fill in:

| Field | Value |
|---|---|
| **App Name (관리 상품명)** | Anything, e.g. `Clemvion Integration` |
| **App URL** | `https://<your-deployment>` — this product's base URL |
| **Redirect URI** | **`https://<your-deployment>/api/integrations/oauth/callback/cafe24`** — this backend's OAuth callback endpoint. **Cafe24 verifies this character-by-character** — trailing slash, `http` vs `https`, port, typos all matter. If you have multiple environments (local, staging, production), register every URL here so any environment's OAuth flow goes through the same app |
| **API Permissions (Scope)** | Tick the read/write categories your workflows will hit. Mirrors the Scope checkboxes on the Step 2 form |
| Webhook settings | Optional, leave blank |

5. The same page shows the freshly generated **`Client ID`** / **`Client Secret`** — copy both.

> No App Store review is needed for self-distributed apps. When the OAuth popup our service opens hits your mall's authorize URL, Cafe24 shows an "Install this app?" consent screen and proceeds to token issuance the moment you accept.

### 3) Step 2 — pre-flight inputs

Unlike Google/GitHub, Cafe24's URL itself depends on `mall_id`, so you supply it before the OAuth popup opens.

| Field | Description |
|---|---|
| **Name** | Integration alias. Must be unique within the workspace |
| **Scope** | Personal / Organization. Only admins can create organisation-scoped integrations |
| **Mall ID** | Cafe24 mall identifier; the `{mall_id}` part of `https://{mall_id}.cafe24api.com`. Must match `/^[a-z0-9-]{3,50}$/` |
| **App Type** | **Public** (Cafe24 App Store app — server env supplies client_id/secret) / **Private** (your own self-issued app — paste client_id/secret in the form) |
| **Client ID / Client Secret** | Shown only when App Type = Private. The OAuth credentials you got from your shop's admin |
| **Scopes** | Per-category read/write checkboxes (e.g. `mall.read_product`, `mall.write_order`). At least one is required |

### 4) [Connect with Cafe24]

A popup opens with Cafe24's consent screen. After consent the popup tells the parent window the result, the backend exchanges the token, and the integration is created.

> Tokens auto-refresh. When `access_token` is within 60 seconds of expiring, the backend issues a refresh call to the mall-specific token URL just before the next node call. Repeated refresh failures flip the integration to `expired` and fire a notification.

## Use it in a workflow — the `Cafe24` node

Drop **Integration → Cafe24** from the node palette. In the settings panel:

| Field | What it does |
|---|---|
| **Integration** | Select the Cafe24 integration you created |
| **Resource** | One of 18 categories (Product / Order / Customer / Category / Promotion ...) |
| **Operation** | The endpoint id (`product_list`, `order_get`, `customer_update`, ...). See `spec/conventions/cafe24-api-metadata.md` or the [official Cafe24 docs](https://developers.cafe24.com/docs/ko/api/admin/) |
| **Fields** | Per-operation inputs. Expressions like `{{ $input.x }}` are supported |
| **Pagination** | Limit / Offset (only when the operation is paginated) |

A successful response is preserved verbatim on `output.response` — downstream nodes can read `$node["Cafe24"].output.response.products[0].product_no`.

### On failure

- 4xx/5xx → `port='error'`, `output.error.code` = `CAFE24_404` / `CAFE24_422` / `CAFE24_4XX` / `CAFE24_5XX`.
- 401/403 → `CAFE24_AUTH_FAILED` and the integration auto-transitions to `error(auth_failed)`. The user must re-authenticate.
- 429 (rate limit) → the wrapper waits per the response header and retries up to 2 times. If it still fails, `CAFE24_RATE_LIMITED`.
- Network errors → `CAFE24_TRANSPORT_FAILED`.

## Use it from an AI Agent — MCP tools

In the AI Agent settings panel → **MCP Servers → + Add MCP Server**. The picker groups available integrations:

```
🌐 Generic MCP (HTTP) servers
  - My Notion MCP
🛒 Cafe24 stores (Internal Bridge)
  - My Shop
```

Once a Cafe24 integration is added, the LLM sees tools named `mcp_<sid>__<resource>_<operation>` (e.g. `mcp_abcdef12__product_list`).

- **allowlist** — set `enabledTools` to the operation ids you actually want exposed. Cuts down LLM context size and improves answer quality. Recommended: limit to the categories needed for that workflow's job.
- **shared rate limit** — when the same integration is used by both the cafe24 node and the AI Agent simultaneously, the leaky bucket is shared. If one path is backing off on a 429, the other waits too (within a single backend instance).

## Environment variables

| Variable | Use |
|---|---|
| `CAFE24_CLIENT_ID` | Public-app flow — the Cafe24 App Store app's client_id used by every Public-type integration on this deployment |
| `CAFE24_CLIENT_SECRET` | Secret for the above. Never leaves the backend |
| `OAUTH_STUB_MODE` | When `true`, OAuth exchange is stubbed (fake tokens) — local dev / unit tests only. Ignored in `NODE_ENV=production` |

## Recommended OAuth scope presets

| Category | scope value |
|---|---|
| Product | `mall.read_product` / `mall.write_product` |
| Order | `mall.read_order` / `mall.write_order` |
| Customer | `mall.read_customer` / `mall.write_customer` |
| Category | `mall.read_category` / `mall.write_category` |
| Promotion | `mall.read_promotion` / `mall.write_promotion` |
| Mileage | `mall.read_mileage` / `mall.write_mileage` |
| Shipping | `mall.read_shipping` / `mall.write_shipping` |
| Salesreport | `mall.read_salesreport` (read only) |

See the [official Cafe24 docs](https://developers.cafe24.com/docs/ko/api/admin/#api-index) for the full list across all 18 categories.

## FAQ

**Q. OAuth start fails with `invalid_request: The redirec_uri added by Cafe24 Developers is invalid`.**
The `redirect_uri` parameter this service sent does not exactly match any Redirect URI registered on the Cafe24 Developers app. Cafe24 compares the URL character-by-character — trailing slash (`…/cafe24` vs `…/cafe24/`), `http` vs `https`, host typos, and port specs all count. Fastest fix: copy the `redirect_uri=` query value out of the authorize URL that errored, and paste it verbatim into the app's Redirect URI field on Cafe24 Developers.

**Q. How do I register callback URLs for multiple environments (local / staging / production)?**
The Cafe24 app's Redirect URI field accepts multiple entries. Add the full `https://<env-host>/api/integrations/oauth/callback/cafe24` for every environment that will start an OAuth flow through this app.

**Q. OAuth start fails with `invalid_scope: The scope added by Cafe24 Developers is invalid`.**
Cafe24 requires **comma-delimited** scopes on `/oauth/authorize` (a deviation from RFC 6749 §3.3 which mandates space). This service already sends comma-delimited scopes, but if you still see `invalid_scope`, one of these is the cause:

1. **Missing pre-registration on the app (most common)** — Cafe24 only honors OAuth requests for scopes that are *pre-registered on the app*. If even one scope you ticked in the integration form is missing from the Cafe24 app's permission settings, the *entire* OAuth call is rejected.

   - Sign in to https://developers.cafe24.com → **`Apps` → `My Apps`** → click the app
   - Open **`App Information`** (or **`개발 정보`** in Korean UI) → scroll to the **`Permissions` / `Scope`** section
   - Tick **every** scope you selected in the integration form → save → retry [Connect with Cafe24]

2. **App type mismatch** — a Front-store app requesting Admin scope (`mall.*`) or vice versa.

> Tip: To narrow down which scope is missing, reduce the integration form selection to a single scope (e.g. `mall.read_product`) first. If it succeeds, add the rest one by one to find the failing one.

**Q. Can I register an integration while my service is still in development?**
Yes. You don't need to publish to the Cafe24 App Store — the **Private (self-issued) app** flow works end-to-end. Create an app on Cafe24 Developers from your own mall, tick every scope you plan to use in the app's permission settings, then pick App Type = Private in the integration form and paste your `client_id` / `client_secret`. No App Store review is required; the app installs straight into your own mall.

**Q. I mistyped the mall_id — how do I fix it?**
Only the integration's display name (alias) can be edited after creation. Recreate the integration. An invalid mall_id is rejected at the OAuth begin step (`/^[a-z0-9-]{3,50}$/`).

**Q. I get `OAUTH_CONFIG_MISSING` on the first Public-app authentication.**
The server `CAFE24_CLIENT_ID` / `CAFE24_CLIENT_SECRET` env vars are not set. Ask your operator to configure them, or switch App Type to Private and paste your own app's credentials.

**Q. The AI Agent throws `CAFE24_MCP_NO_SESSION` when calling a tool.**
The AI Agent node's `mcpServers` list must include this Cafe24 integration. Re-add it on that node — adding the integration later than the AI Agent execution start triggers this error.

**Q. Throughput feels slow under heavy use.**
You've hit Cafe24's leaky-bucket quota and the backend is auto-backing off. `meta.callUsage` / `meta.callRemain` headers expose the current bucket state. If many workflows share the same integration, split usage across separate integrations to widen the quota.
