This page walks you through registering a Korean e-commerce SaaS **Cafe24** mall as an integration and using it from both workflow nodes and the AI Agent.

## Why use the Cafe24 integration?

- **One credential, two surfaces** тАФ a single Cafe24 integration is reused by (a) the `Cafe24` workflow node and (b) the AI Agent's MCP tool list. Token refresh, leaky-bucket rate limits, and activity logs are unified across both call paths.
- **All 18 Admin API categories** тАФ Product, Order, Customer, Category, Promotion, Mileage, ... every endpoint of Cafe24 Admin API is reachable through a single metadata-driven node. Most operations are two dropdowns away (Resource тЖТ Operation).
- **Natural-language AI calls** тАФ through an in-process Internal MCP Bridge, the LLM can translate commands like *"give me last week's unshipped orders"* into the right endpoint call automatically.

## Create the integration (`/integrations`)

### 1) Pick Cafe24 in the Add Integration modal

`/integrations` тЖТ **+ Add Integration** тЖТ click the **ЁЯЫТ Cafe24** card тЖТ routed to `/integrations/new?service=cafe24&step=auth`.

### 2) (Private apps only) Create the app on Cafe24 Developers first

To finish Step 2 with App Type = **Private** you need a `client_id` / `client_secret` from Cafe24 itself. Cafe24 has no explicit "public/private" toggle тАФ **any app that isn't published to the App Store is automatically private** (only your own mall can install it).

1. Sign in to https://developers.cafe24.com with the Cafe24 mall owner account.
2. Top menu **`Apps`** тЖТ **`My Apps`** (a.k.a. "ьХ▒ ъ┤Аыжм").
3. Click **`ADD PRODUCT` / `Create app`** тЖТ choose **`Web`**.
4. Fill in:

| Field | Value |
|---|---|
| **App Name (ъ┤Аыжм ьГБэТИыкЕ)** | Anything, e.g. `Clemvion Integration` |
| **App URL** | `https://<your-deployment>` тАФ this product's base URL |
| **Redirect URI** | **`https://<your-deployment>/api/integrations/oauth/callback/cafe24`** тАФ this backend's OAuth callback endpoint. Must match exactly |
| **API Permissions (Scope)** | Tick the read/write categories your workflows will hit. Mirrors the Scope checkboxes on the Step 2 form |
| Webhook settings | Optional, leave blank |

5. The same page shows the freshly generated **`Client ID`** / **`Client Secret`** тАФ copy both.

> No App Store review is needed for self-distributed apps. When the OAuth popup our service opens hits your mall's authorize URL, Cafe24 shows an "Install this app?" consent screen and proceeds to token issuance the moment you accept.

### 3) Step 2 тАФ pre-flight inputs

Unlike Google/GitHub, Cafe24's URL itself depends on `mall_id`, so you supply it before the OAuth popup opens.

| Field | Description |
|---|---|
| **Name** | Integration alias. Must be unique within the workspace |
| **Scope** | Personal / Organization. Only admins can create organisation-scoped integrations |
| **Mall ID** | Cafe24 mall identifier; the `{mall_id}` part of `https://{mall_id}.cafe24api.com`. Must match `/^[a-z0-9-]{3,50}$/` |
| **App Type** | **Public** (Cafe24 App Store app тАФ server env supplies client_id/secret) / **Private** (your own self-issued app тАФ paste client_id/secret in the form) |
| **Client ID / Client Secret** | Shown only when App Type = Private. The OAuth credentials you got from your shop's admin |
| **Scopes** | Per-category read/write checkboxes (e.g. `mall.read_product`, `mall.write_order`). At least one is required |

### 4) [Connect with Cafe24]

A popup opens with Cafe24's consent screen. After consent the popup tells the parent window the result, the backend exchanges the token, and the integration is created.

> Tokens auto-refresh. When `access_token` is within 60 seconds of expiring, the backend issues a refresh call to the mall-specific token URL just before the next node call. Repeated refresh failures flip the integration to `expired` and fire a notification.

## Use it in a workflow тАФ the `Cafe24` node

Drop **Integration тЖТ Cafe24** from the node palette. In the settings panel:

| Field | What it does |
|---|---|
| **Integration** | Select the Cafe24 integration you created |
| **Resource** | One of 18 categories (Product / Order / Customer / Category / Promotion ...) |
| **Operation** | The endpoint id (`product_list`, `order_get`, `customer_update`, ...). See `spec/conventions/cafe24-api-metadata.md` or the [official Cafe24 docs](https://developers.cafe24.com/docs/ko/api/admin/) |
| **Fields** | Per-operation inputs. Expressions like `{{ $input.x }}` are supported |
| **Pagination** | Limit / Offset (only when the operation is paginated) |

A successful response is preserved verbatim on `output.response` тАФ downstream nodes can read `$node["Cafe24"].output.response.products[0].product_no`.

### On failure

- 4xx/5xx тЖТ `port='error'`, `output.error.code` = `CAFE24_404` / `CAFE24_422` / `CAFE24_4XX` / `CAFE24_5XX`.
- 401/403 тЖТ `CAFE24_AUTH_FAILED` and the integration auto-transitions to `error(auth_failed)`. The user must re-authenticate.
- 429 (rate limit) тЖТ the wrapper waits per the response header and retries up to 2 times. If it still fails, `CAFE24_RATE_LIMITED`.
- Network errors тЖТ `CAFE24_TRANSPORT_FAILED`.

## Use it from an AI Agent тАФ MCP tools

In the AI Agent settings panel тЖТ **MCP Servers тЖТ + Add MCP Server**. The picker groups available integrations:

```
ЁЯМР Generic MCP (HTTP) servers
  - My Notion MCP
ЁЯЫТ Cafe24 stores (Internal Bridge)
  - My Shop
```

Once a Cafe24 integration is added, the LLM sees tools named `mcp_<sid>__<resource>_<operation>` (e.g. `mcp_abcdef12__product_list`).

- **allowlist** тАФ set `enabledTools` to the operation ids you actually want exposed. Cuts down LLM context size and improves answer quality. Recommended: limit to the categories needed for that workflow's job.
- **shared rate limit** тАФ when the same integration is used by both the cafe24 node and the AI Agent simultaneously, the leaky bucket is shared. If one path is backing off on a 429, the other waits too (within a single backend instance).

## Environment variables

| Variable | Use |
|---|---|
| `CAFE24_CLIENT_ID` | Public-app flow тАФ the Cafe24 App Store app's client_id used by every Public-type integration on this deployment |
| `CAFE24_CLIENT_SECRET` | Secret for the above. Never leaves the backend |
| `OAUTH_STUB_MODE` | When `true`, OAuth exchange is stubbed (fake tokens) тАФ local dev / unit tests only. Ignored in `NODE_ENV=production` |

## FAQ

**Q. I mistyped the mall_id тАФ how do I fix it?**
Only the integration's display name (alias) can be edited after creation. Recreate the integration. An invalid mall_id is rejected at the OAuth begin step (`/^[a-z0-9-]{3,50}$/`).

**Q. I get `OAUTH_CONFIG_MISSING` on the first Public-app authentication.**
The server `CAFE24_CLIENT_ID` / `CAFE24_CLIENT_SECRET` env vars are not set. Ask your operator to configure them, or switch App Type to Private and paste your own app's credentials.

**Q. The AI Agent throws `CAFE24_MCP_NO_SESSION` when calling a tool.**
The AI Agent node's `mcpServers` list must include this Cafe24 integration. Re-add it on that node тАФ adding the integration later than the AI Agent execution start triggers this error.

**Q. Throughput feels slow under heavy use.**
You've hit Cafe24's leaky-bucket quota and the backend is auto-backing off. `meta.callUsage` / `meta.callRemain` headers expose the current bucket state. If many workflows share the same integration, split usage across separate integrations to widen the quota.
