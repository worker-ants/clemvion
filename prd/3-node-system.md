## 1. 개요

노드는 워크플로우의 기본 구성 단위이다. 각 노드는 특정 기능을 수행하며, 입력 포트로 데이터를 받아 처리한 후 출력 포트로 결과를 전달한다. 노드는 **Trigger**, **Logic**, **Flow**, **AI**, **Integration**, **Data**, **Presentation** 일곱 가지 카테고리로 분류된다.

## 3. Trigger 노드 (1종)

트리거 노드는 워크플로우의 시작점이다. 모든 워크플로우는 반드시 1개의 Trigger 노드를 가져야 하며, 워크플로우 생성 시 자동으로 배치된다. Trigger 노드는 입력 포트가 없으며, 출력 포트를 통해 워크플로우 입력 데이터를 후속 노드에 전달한다.

### 3.1 Manual Trigger

| ID | 요구사항 | 우선순위 | 상태 |
|----|----------|----------|-------|
| ND-MT-01 | 수동 실행(Run 버튼, API 호출)으로 워크플로우를 시작하는 진입점 | 필수 | ✅ |
| ND-MT-02 | 입력 포트 없음, 출력 포트 1개 (Output) | 필수 | ✅ |
| ND-MT-03 | 워크플로우 입력 데이터를 그대로 출력 포트로 전달 (pass-through) | 필수 | ✅ |
| ND-MT-04 | 워크플로우 생성 시 자동 배치. 사용자가 삭제할 수 없음 | 필수 | ✅ |
| ND-MT-05 | 워크플로우당 Manual Trigger 노드는 1개만 존재 가능 | 필수 | ✅ |

> **🚧 P1 구현 완료**: `PARALLEL_ENGINE=v1` 환경변수로 활성화하면 `ParallelExecutor`가 `p-limit` + `Promise.allSettled`로 분기를 진짜 동시 실행한다(기본값 `off`이면 기존 순차 동작). branchCount(2~16), maxConcurrency(0=무제한, 1~16) 지원. 분기 내 블로킹 노드(form/buttons/ai_conversation) 금지, back-edge 금지, 중첩 Parallel 금지(P2 예정). waitAll은 항상 true로 동작하며 false는 P2에서 지원 예정이다.

### 4.11 Background

> **✅ 구현 완료**: 노드 정의·핸들러·BullMQ 큐(`background-execution`)·워커가 모두 `ExecutionEngineService`에 통합되었다. 핸들러 실행 직후 `scheduleBackgroundBody()`가 컨텍스트 스냅샷과 함께 본문 진입점들을 enqueue하고, 워커는 `executeBackgroundSubgraph()`를 통해 진입점에서부터 forward-reachable한 서브그래프를 격리된 컨텍스트로 실행한다. 컨테이너 멤버십(`containerId`) 모델 대신 `background` 포트 엣지로만 본문을 식별한다.

| ID | 요구사항 | 우선순위 | 상태 |
|----|----------|----------|-------|
| ND-BG-01 | 하위 노드 그룹을 백그라운드로 실행 (메인 흐름을 블로킹하지 않음) | 필수 | ✅ |
| ND-BG-02 | 메인 흐름은 즉시 다음 노드로 진행 | 필수 | ✅ |
| ND-BG-03 | 백그라운드 실행 실패 시 알림 설정(`notifyOnFailure`) | 필수 | ✅ |
| ND-BG-04 | 백그라운드 실행 상태 모니터링 (NodeExecution 타임라인, `parentNodeExecutionId` 그룹핑) | 필수 | ✅ |
| ND-BG-05 | Background 노드를 컨테이너로 렌더링 — 별도 결정으로 일반 다중 출력 포트 노드로 처리(아래 참조) | 필수 | ✅ (대안 구현) |

**ND-BG-05 대안 구현 노트**: 본문은 `background` 포트 엣지로 명시 연결되어 시각적으로도 분기가 분명하므로, 캔버스에서 별도 멤버십 박스(컨테이너 그룹 박스)로 감싸지 않는다. Loop/ForEach/Map과 다르게 본문이 메인과 같은 캔버스 그래프 안에 평면적으로 존재한다는 의미를 더 직접적으로 드러낸다.
