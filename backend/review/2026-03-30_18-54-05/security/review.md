### 발견사항

- **[INFO]** `GET /users/me` 엔드포인트에 인가 가드(AuthGuard) 선언이 코드상 보이지 않음
  - 위치: `users.controller.ts:11` — `@Get('me')`
  - 상세: `@CurrentUser()` 데코레이터가 JWT 페이로드를 주입하지만, 컨트롤러/핸들러 레벨에 `@UseGuards(JwtAuthGuard)` 등의 명시적 가드가 없음. 글로벌 가드가 설정되어 있다면 문제없으나, 코드만 봐서는 인증 강제 여부를 확인할 수 없음.
  - 제안: 글로벌 가드가 없다면 `@UseGuards(JwtAuthGuard)`를 명시적으로 추가. 글로벌 설정이 있다면 주석으로 명시하여 의도를 드러낼 것.

- **[INFO]** 사용자 미발견 시 401/404 대신 200 + `{ data: null }` 반환
  - 위치: `users.controller.ts:14-16`
  - 상세: 유효한 JWT를 가진 사용자가 DB에 없는 경우(삭제된 계정 등) 200을 반환. 클라이언트가 에러를 인지하지 못할 수 있으며, 탈취된 토큰으로 계정 존재 여부를 소극적으로 탐색하는 데 악용될 수 있음.
  - 제안: `throw new NotFoundException()` 또는 `throw new UnauthorizedException()`으로 명확한 HTTP 상태코드를 반환.

- **[INFO]** `V003` 마이그레이션은 DDL 변경이므로 트랜잭션 롤백 불가
  - 위치: `V003__add_trigger_category.sql:2`
  - 상세: PostgreSQL에서 `ALTER TYPE ... ADD VALUE`는 트랜잭션 내에서 롤백할 수 없음. 배포 실패 시 수동 정리가 필요하며, 운영 DB에 직접 적용될 경우 되돌리기 어려움.
  - 제안: Flyway/Liquibase 등 마이그레이션 도구의 동작을 이해하고, 해당 제약을 팀 문서에 명시. 롤백 시나리오를 별도로 준비할 것.

- **[INFO]** 테스트 코드에 `mockUser.passwordHash` 평문 유사 값 포함
  - 위치: `users.controller.spec.ts:18`
  - 상세: `passwordHash: 'hashed'`는 테스트 환경이므로 실질적 위험은 없으나, 실제 해시값이나 민감 데이터가 테스트 픽스처에 섞이는 습관을 방지하는 차원에서 언급.
  - 제안: 현재 코드는 문제없음. 다만 실제 bcrypt 해시나 실 계정 정보가 테스트 픽스처에 포함되지 않도록 컨벤션을 유지할 것.

### 요약

세 파일 모두 명백한 인젝션·하드코딩 시크릿·암호화 취약점은 없으며, SQL 마이그레이션도 파라미터화가 불필요한 DDL로 인젝션 위험이 없습니다. 주요 관심사는 `UsersController`의 인가 가드 명시성 부재와 사용자 미발견 시 200 응답 반환으로, 글로벌 가드 설정 여부에 따라 실제 인증 우회 가능성이 달라집니다. 전반적으로 보안 수준은 양호하며, 발견된 사항들은 방어적 코딩 관점의 개선 권고 수준입니다.
