# push 가드 오탐 — 커밋 메시지의 "push" 단어가 `git commit` 을 막는다

> `harness-session-anchor-guards.md` §② 에서 **분리**됐다. 그 plan 에서 한 번 구현했다가
> **되돌렸다**(사용자 결정 2026-07-17). 아래는 그 시도가 실측으로 남긴 것들이다 —
> **같은 길을 다시 걷지 말 것.** `.claude/hooks/guard_review_before_push.py` 는 현재
> 원래 정규식 그대로다.

## Overview

`.claude/hooks/guard_review_before_push.py:55` 의 정규식이 push 아닌 명령을 차단한다.

```python
r"(?:^|&&|;|\|)\s*(?:[A-Za-z_][A-Za-z0-9_]*=\S+\s+)*git\b[^&;|]*\bpush\b"
```

두 결함이 겹쳐 있다:

1. **`git`↔`push` 거리 무제한** — `[^&;|]*` 가 개행도 먹어 **heredoc 본문을 가로지른다**.
   `git commit -F - <<'EOF'` 뒤 메시지에 "push" 가 있으면 매칭.
2. **셸 인용을 모른다** — 따옴표 안 `\|` 를 파이프 구분자로 읽는다.
   `grep -n "…\|git push\|…" f` 가 매칭.

실제 비용: 커밋 메시지에 "push" 라는 단어를 썼다는 이유로 `git commit` 이 막혔고, 우회는 메시지를
파일로 빼는 것이었다. **가드가 틀리면 사람이 우회를 학습한다.**

## ⚠ 기각된 접근 — 서브커맨드 판정 재작성 (2026-07-17 실측)

"`git` 뒤 어딘가에 push" 가 아니라 **"git 의 서브커맨드가 push"** 를 `shlex` 로 판정하는 재작성을
실제로 구현했고, `/ai-review` 3라운드에서 **매 라운드 새로운 거짓 음성 회귀**가 나와 되돌렸다.

| 라운드 | 새로 드러난 회귀 — 구 정규식은 **차단하던** 것 |
| --- | --- |
| 1 | 개행 단독 구분(`git add -A\ngit push`) · `git --attr-source main push` |
| 2 | `git commit -m "$(git push)"` · 백틱 · `bash -c "cd /tmp && git push"` |
| 3 | `git $'push'`(ANSI-C 인용) · `git $"push"` · 백틱 미종료 |

각 라운드마다 고쳤고, 그때마다 **다음 라운드가 새 클래스를 찾았다**.

### 왜 실패했나 (이 plan 의 핵심 — 재도전 전에 반드시 읽을 것)

**구 정규식은 "무지해서 안전"하다.** 원시 문자열을 훑으므로 `git … push` 의 *어떤 표기든* 걸린다.
정밀 파서는 반대다 — **실행 전에 텍스트를 변형하는 셸 기능마다 구멍이 생긴다**:

`$'…'` · `$"…"` · `$(…)` · 백틱 · `-c` · `eval` · 인용 분할(`'pu''sh'`) · hex escape(`$'\x70…'`) ·
`find -exec` · 프로세스 치환 · git alias · NUL 오염 · …

이건 **열거 불가능한 표면**이다. 반면 정규식의 결함(거짓 양성)은 **유한하고 열거 가능**하다.
재작성은 **유한한 문제를 무한한 문제와 맞바꿨다**.

3라운드에서 도입한 `old ⊆ new` 차등 테스트조차 **주어진 코퍼스 위에서만** 수렴한다. 코퍼스가
ANSI-C 인용을 몰랐기에 3라운드가 그걸 찾았다. 다음 코퍼스도 무언가를 모를 것이다.

## 권장 방향 — 설계 반전 (unbounded → bounded)

**막는 쪽을 무지하게 두고, 푸는 쪽만 정밀하게 한다.**

1. **1차 판정 = 현행 정규식 그대로** — blind scan. 거짓 음성 0. 이 성질을 **절대 훼손하지 않는다**.
2. **그 위에 열거된 allowlist** — "이 매칭은 증명 가능하게 오탐이다" 인 형태만 해제한다.

allowlist 후보(각각 반드시 실측으로 안전 논증):

- **heredoc 본문 / `-m`·`-F` 메시지 텍스트 영역 안의 `push`** — 단 그 영역에 `$(`·백틱이
  **없을 때만**. (있으면 셸이 실제로 실행하므로 해제 금지 — 2라운드가 실증했다.)
- **따옴표 안에서만 등장하는 `\|`** 로 인해 세그먼트가 갈린 경우(grep 패턴).

이 방향이 수렴하는 이유: 해제 목록이 **틀리면 거짓 음성**이 되지만, 목록은 **우리가 고르는 유한
집합**이라 각 항목을 개별 실증할 수 있다. 반대로 파서는 **셸이 고르는 무한 집합**을 따라가야 한다.

## 검증 (재도전 시 필수)

- [ ] **차등 테스트 필수** — 구 정규식을 기준선으로 박아넣고, 코퍼스 전체에서
      `legacy(c) and not new(c)` 인 명령은 **열거된 예외**일 때만 허용. 예외마다 "왜 안전한가" 실측 근거.
- [ ] 코퍼스에 아래를 **반드시** 포함(3라운드가 찾아낸 것들 — 재발 방지 기준선):
      `git add -A\ngit push` · `git --attr-source main push` · `git commit -m "$(git push)"` ·
      ``git commit -m "`git push`"`` · `bash -c "cd /tmp && git push"` · `git $'push'` · `git $"push"` ·
      `git 'pu''sh' --force` · `GIT push` · `git push\x00 extra` · `eval "git push"`
- [ ] 오탐 해제 목록 전수 통과: heredoc 메시지 · `-m` 메시지 · 따옴표 grep
- [ ] **해제 목록에 `$(`·백틱이 있으면 해제되지 않는지** (2라운드 회귀의 재발 방지)
- [ ] 신규 테스트가 수정 전 코드에서 실패하는지(비-vacuity)

## 알려진 선재 갭 (이번에도 고칠 대상 아님 — 구 정규식도 못 잡는다)

`eval "git push"` · `bash -c "git push"`(구분자 없음) · `find … -exec git push \;` ·
프로세스 치환 `diff <(git push)` · git alias(`git p`) · hex escape `git $'\x70\x75\x73\x68'`.

전부 정적 텍스트 가드의 구조적 한계다. **설계 반전은 이것들을 고치지 않는다** — 현행과 동일하게
남는다. 오탐만 줄이는 것이 이 plan 의 스코프다.

## Rationale

**왜 별도 plan 인가**: `harness-session-anchor-guards.md` 의 ①(세션 앵커 reap)은 검증 완료·독립적이고
**재발이 확정적인 가용성 사고**였다. ②를 계속 붙들면 ①이 함께 묶여 지연된다. 원 plan 도
"우선순위는 ① > ②" 로 명시했다. 그래서 ①만 머지하고 ②를 분리했다(사용자 결정 2026-07-17).

**왜 되돌렸나 (재작성을 살려두지 않고)**: 3라운드 시점의 코드에는 `git $'push'`·`git $"push"`
**실동작 회귀 2건이 열려 있었다**. 거짓 양성(불편)을 고치려다 거짓 음성(미검토 코드 통과)을 남기는
건 순손실이다 — 원 plan 의 "거짓 음성을 만들지 말 것, 이쪽이 안전 방향이다" 원칙에 정면으로 어긋난다.
부분 개선을 남기는 대신 가드 건전성을 오늘 상태로 되돌리고, 설계를 다시 정한 뒤 재도전한다.

**기각된 대안 — 파서 계속 하드닝**: 3/3 라운드가 매번 새 클래스를 찾았다. 4라운드도 그럴 것이라는
게 합리적 추정이고, hex escape 는 이미 "구 정규식도 못 잡던" 케이스로 확인돼 파서가 **구 정규식보다
넓어지지도 못한다**는 것까지 드러났다. 무한 표면을 추격하는 설계는 기각한다.
