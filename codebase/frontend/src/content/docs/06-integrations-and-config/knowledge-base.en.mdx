This page explains how to create a Knowledge Base, upload documents, and use it as RAG (Retrieval-Augmented Generation) context in an AI Agent.

## Structure of a knowledge base

A Knowledge Base is organized by **collection**. Each collection pins an embedding model and a chunking strategy; documents live inside. Uploads go through: parse → chunk → embed → vector-store persist.

  { name: "Description", type: "string", required: "Optional", description: "Short purpose. Helpful for team collaboration." },

  { name: "RAG mode", type: "`vector` / `graph`", required: "Optional", description: "Defaults to `vector`. `graph` mode extracts entities and relations from documents with an LLM, builds a knowledge graph inside the KB, and combines vector seed + graph expansion at query time. Mode is fixed at creation time." },

  { name: "Chunk overlap", type: "integer (tokens)", required: "Optional", description: "Default 200. Overlap between adjacent chunks." }
]} />

<Callout type="note">Only providers that support embeddings can be selected as the embedding model. Register a provider first — see [LLM configuration](/docs/06-integrations-and-config/llm-config).</Callout>

## Creating a collection and uploading documents

<Steps>
<Step>At `/knowledge-bases`, click `[+ New Collection]` and fill in name, description, embedding model, and chunk parameters.</Step>
<Step>Click the new collection to open its detail view. Use `[Upload Documents]` or drag-and-drop files.</Step>
<Step>Supported formats: `.txt`, `.md`, `.pdf`, `.csv`. Multiple files at once are fine — embedding starts immediately.</Step>
<Step>When each document's status flips from `🔄 Processing` to `✅ Ready`, it's searchable. Status streams in real time via WebSocket.</Step>
<Step>Failures show `❌ Error`. Use the row's ⋮ menu → `Re-embed` to re-run the pipeline.</Step>
</Steps>

<Callout type="warn">CSV rows become independent chunks. Without a header, each cell loses context, so putting column names in the first row improves retrieval quality.</Callout>

## Vector mode · Graph mode

A collection's `RAG mode` shapes the ingestion pipeline and the query path. It can't be changed after creation — to switch modes, create a new collection and re-upload the documents.

<FieldTable rows={[
  { name: "Vector (default)", type: "Embedding + cosine similarity", required: "-", description: "Each chunk is embedded and stored in the vector store; queries return the chunks with the highest cosine similarity. Fast and cheap, ideal for FAQs and general document search." },
  { name: "Graph", type: "Embedding + entity/relation extraction", required: "-", description: "After embedding finishes, an LLM automatically extracts entities (person, organization, concept, location, event) and relations from documents, building a knowledge graph inside the KB. At query time, vector top-K seed chunks are collected, the graph is expanded from entities mentioned in those chunks up to `maxHops`, additional chunks are pulled back and reranked (Hybrid). Surfaces richer context when relationships matter." }
]} />

<Callout type="note">Graph mode KBs incur extra LLM calls. The collection detail page shows real-time extraction progress, entity/relation counts, and cumulative token usage. The LLM used for extraction is selected via `Extraction LLM Config` (defaults to the workspace default).</Callout>

### Graph mode KB search parameters

<FieldTable rows={[
  { name: "maxHops", type: "1 or 2", required: "Optional", description: "Default 1. Max depth to expand the graph from seed-chunk entities." },
  { name: "vectorSeedTopK", type: "integer", required: "Optional", description: "Default 5. Number of top-similarity chunks retrieved in stage 1." },
  { name: "expandedChunkLimit", type: "integer", required: "Optional", description: "Default 15. Cap on total chunks (seed + expansion)." }
]} />

The `AI Agent` node's KB connection UI is identical regardless of mode (`ragTopK` / `ragThreshold` only). Graph parameters are controlled per KB.

### Managing entities / relations

Graph mode KBs add **Entity**, **Relation**, and **Graph** tabs on the collection detail page.

- Entity / Relation tabs: search, sort, and delete extracted items. Cleaning up imprecise LLM extractions takes effect on the next query.
- Graph tab: visualizes the extracted graph in 3D / 2D. Zoom in or hover a node to preview the chunks where that entity appears.
- If extraction results look off or you want to re-run with a new prompt, use the `Re-extract` action on the collection detail page to trigger full or per-document re-extraction.

## Referencing from an AI Agent

The AI Agent node's settings panel has a multi-select `Knowledge Bases` field. Selected collections are searched by similarity against the user prompt at run time, and the results are injected under the system prompt.

<Example title="KB reference in an AI Agent node">
```ts
{
  "type": "ai_agent",
  "config": {
    "llmConfigId": "llm_01H...",
    "systemPrompt": "You are a customer support assistant.",
    "userPrompt": "{{ $input.parameters.question }}",
    "knowledgeBases": [
      "kb_01H...FAQ",
      "kb_01H...POLICY"
    ],
    "ragTopK": 5,           // max results (default 5)
    "ragThreshold": 0.7     // cosine similarity floor (default 0.7)
  }
}
```
</Example>

After execution, the AI Agent result carries RAG-source metadata so you can see which chunks from which documents were referenced, and with what scores. You can also read them via expressions — e.g. `{{ $node["AI Agent"].meta.ragSources[0].documentName }}`.

<Callout type="tip">When the top match scores below `threshold`, the LLM is called without RAG context. If answers look weak, try lowering the threshold to about 0.6, or shrink chunk size and re-embed.</Callout>

## Tips & notes

- When multiple collections are selected, KBs sharing the same embedding model share a single query; others are queried independently and the results merged.
- Injected context is auto-truncated to fit within 50% of the model's context window, dropping the lowest-scoring chunks first — top results are always kept.
- Document preview shows every generated chunk. If a chunk looks off, edit the source and hit `Re-embed`.
- To switch embedding model, create a new collection and move documents over. Swapping the model on an existing collection triggers a full re-embed of every chunk.
- Expressions and node-result basics: [Variables and context](/docs/04-expression-language/variables-and-context).
- Query/distance/error rules for RAG search live in `spec/5-system/9-rag-search.md`.
