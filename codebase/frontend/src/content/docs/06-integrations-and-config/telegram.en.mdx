Connect a Telegram bot as your workflow's conversational interface. Users send messages to the bot to start a workflow or provide input to a running execution.

## 1. Create a Telegram bot

1. Open Telegram and search for [@BotFather](https://t.me/botfather).
2. Send `/newbot` and follow the prompts to choose a name and username.
3. BotFather gives you an **API Token** (format: `123456789:ABCdef…`). Keep it safe.

<Callout type="warn">
The Bot API Token grants full control over your bot account. Never embed it directly in code or public repositories.
</Callout>

## 2. Add a Chat Channel to a Webhook trigger

**GUI flow (recommended)**:

<ImplAnchor
  kind="ui-entry"
  file="codebase/frontend/src/app/(main)/triggers/page.tsx"
  symbol="chatChannelProvider"
  describes="Chat Channel section in the trigger create dialog (Spec Chat Channel §4.1 / 2-trigger-list §2.3.1 R-8)"
/>

1. Sidebar → **Triggers** → top-right **"+ Add Webhook Trigger"**
2. Enter trigger name, target workflow, and auth type
3. Tick the **"Connect a Chat Channel (Telegram bot)"** checkbox at the bottom
4. Paste the BotFather token (`123456789:ABC...`) into the **"New Bot Token"** field that appears
5. Click **Create** — the trigger is created and `setWebhook` is called automatically, linking the bot to the workflow immediately

In the trigger detail drawer, the **"Chat Channel"** card lets you review/edit health, UI mapping, and language hints, plus rotate the bot token via the **"Rotate Bot Token"** button.

**API flow (automation / CI)**:

To register without the GUI, send `chatChannel` in the trigger create body:

```bash
curl -X POST https://<your-host>/api/triggers \
  -H 'Authorization: Bearer <your-token>' \
  -H 'Content-Type: application/json' \
  -d '{
    "type": "webhook",
    "name": "telegram-bot",
    "workflowId": "<workflow-id>",
    "config": {
      "chatChannel": {
        "provider": "telegram",
        "botToken": "<your BotFather token>"
      }
    }
  }'
```

When the trigger activates, `setWebhook` is called automatically — no manual webhook configuration in BotFather is needed.

- `botToken` is never returned in responses — it is stored AES-256-GCM-encrypted in the secret store, and responses only expose `hasBotToken: true`.
- `botToken` format: `^\d{6,}:[A-Za-z0-9_-]{30,}$` (BotFather standard). Violations return 400 `BOT_TOKEN_INVALID`.

## 3. Supported interaction nodes

| Node type | Telegram representation |
|---|---|
| **AI Multi Turn** | Free-form text conversation. When the AI calls a `render_carousel` / `render_table` / `render_chart` / `render_template` tool, the resulting card / table / chart / template is sent as a sequential message right after the AI's text reply (`render_form` is tracked in a separate plan). |
| **Button Presentation** | Inline keyboard buttons |
| **Form** | Sequential field-by-field text input (contact share, file upload included) |
| **Chart** | Monospace mini bar chart text (labels + values aligned, first series shown) |
| **Table** | Monospace MarkdownV2 table (auto column widths, first 20 rows — `"+N more rows"` appended if exceeded) |
| **Carousel** | Sequential per-card messages (title + description + image URL, up to 10 cards) |
| **Template** | `output.rendered` body is MarkdownV2-escaped and sent as a text message automatically. The body is delivered even when the node has no buttons (non-blocking mode). HTML / Markdown modes are flattened to text in v1 (SSR PNG upgrade is tracked in plan `chat-channel-visual-ssr-png`). |

<Callout type="note">
Visual nodes (Chart, Table, Carousel) currently render as **MarkdownV2 text** in Telegram. A future PR will upgrade them to SSR PNG image delivery (tracked in plan `chat-channel-visual-ssr-png`).

`uiMapping.visualNode` options (default `"auto"`):

| Value | Behavior |
|---|---|
| `"auto"` (default) | Per-node smart choice — chart/table render as text; carousel sends `sendPhoto` per card when `imageUrl` is set, otherwise text |
| `"text"` | Force text-only for all visual nodes (carousel `imageUrl` is ignored) |
| `"photo"` | After v2 SSR infrastructure lands, sends PNG images. On v1 it falls back to `text` behavior (this is normal and does not affect health) |

> The legacy enum value `"text_only"` is deprecated. Existing values in the DB are read-time normalized to `"text"` by the adapter, so older triggers keep working.
</Callout>

## 4. Bot commands

| Command | Behavior |
|---|---|
| `/start` | Start a new execution. Cancels any running one first. |
| `/cancel` | Cancel the current execution |
| `/help` | Display help text |

## 5. Health monitoring

Check `chatChannelHealth` on the trigger detail view:

| Value | Meaning |
|---|---|
| `unknown` | `setupChannel` not yet called or pending |
| `healthy` | Bot connected normally |
| `degraded` | `setupChannel` / `rotate-bot-token` failed or recent consecutive `sendMessage` failures. **The trigger is not auto-disabled** (CCH-SE-01) — only an explicit user toggle changes activation |

The last error message is available in `chatChannelLastError`.

## 6. Rotating the bot token (single-path)

Token changes go through the rotate API only. PATCHing `config.chatChannel.botTokenRef` directly returns 400 `VALIDATION_ERROR` (`details.field='botTokenRef'`).

```bash
curl -X POST https://<your-host>/api/triggers/<trigger-id>/chat-channel/rotate-bot-token \
  -H 'Authorization: Bearer <your-token>' \
  -H 'Content-Type: application/json' \
  -d '{ "newBotToken": "<new BotFather token>" }'
```

- The old token is honored for a **24-hour grace window** alongside the new one, so in-flight updates are not lost. After 24h the old token is cleaned up automatically.
- Response: `{ data: { triggerId, rotatedAt, chatChannelHealth, botIdentity } }`
- Errors: 400 `BOT_TOKEN_INVALID`, 404 `TRIGGER_NOT_FOUND`, 502 `CHAT_CHANNEL_SETUP_FAILED`, etc. ([Spec Chat Channel §5.4](https://github.com/worker-ants/clemvion/blob/main/spec/5-system/15-chat-channel.md#54-bot-token-rotation-api-응답-계약))

## 7. Customizing notification messages (`languageHints` / `languageLocale`)

You can override the bot's built-in notification messages via `config.chatChannel.languageHints`.

### 7.1 `languageLocale` — default locale

`languageLocale` selects the **language of built-in default messages** for keys that have no override.

| Value | Description |
|---|---|
| `"ko"` | Korean defaults (**default** when omitted) |
| `"en"` | English defaults |

**Lookup order** (per key):
1. `languageHints[key]` — operator override always wins
2. Built-in default for the chosen `languageLocale`
3. `"ko"` fallback when `languageLocale` is also omitted

### 7.2 Execution failure keys (CCH-ERR-*)

When a workflow execution fails, the bot sends the user a failure-specific message. You can override any of the 6 keys below.

<Callout type="note">
Sensitive-data protection: the raw `error.message`, `nodeId`, `executionId`, and `details.url` are never included in notification messages. The only allowed placeholder in operator overrides is `{statusCode}`. Any other `{...}` placeholder returns 400 `VALIDATION_ERROR` (`details.code='UNKNOWN_PLACEHOLDER'`).
</Callout>

| Key | Sent when | `{statusCode}` allowed |
|---|---|---|
| `executionFailedThirdParty4xx` | External service (LLM / MCP / HTTP node) rejected with 4xx (`HTTP_4XX`) | optional |
| `executionFailedThirdParty5xx` | External service 5xx transient error (`HTTP_5XX`) | optional |
| `executionFailedThirdParty` | External service unreachable (no status code, not a timeout) | no |
| `executionFailedTimeout` | Processing timed out (`HTTP_TIMEOUT` / `LLM_TIMEOUT` / `EXECUTION_TIMEOUT` etc.) | no |
| `executionFailedRateLimit` | Rate limit exceeded (`LLM_RATE_LIMIT`) | no |
| `executionFailedInternal` | Server-side error (DB / CODE / SUB_WORKFLOW failure + unknown fallback) | no |

**EN default messages** (`languageLocale: "en"`):

```
executionFailedThirdParty4xx  → The external service rejected the request ({statusCode}). Please try again later.
executionFailedThirdParty5xx  → The external service is temporarily unavailable ({statusCode}). Please try again later.
executionFailedThirdParty     → Couldn't reach the external service. Please try again later.
executionFailedTimeout        → The request timed out. Please try again later.
executionFailedRateLimit      → Too many requests. Please try again later.
executionFailedInternal       → The service is temporarily unavailable. Please try again later.
```

## 8. Limitations

- **1:1 (private) chats only** — messages in group chats are silently ignored.
- **Bot messages are ignored** — messages with `from.is_bot=true` are dropped to prevent bot loops.
- **60 requests/minute per conversation** — requests above the cap are ignored (CCH-NF-03).
- **Bot token storage** — the bot token and webhook secret_token are encrypted with AES-256-GCM in the secret store (CCH-SE-03). Only ciphertext lives in the DB; plaintext is transiently decrypted inside the backend process that holds the master key.

## Rationale

### Why a Webhook trigger branch, not a separate trigger type

Instead of creating a dedicated trigger type, the Telegram Chat Channel is a branch of the existing Webhook trigger's `config.chatChannel`. This reuses the existing Webhook entry point with minimal new surface area, and other providers (Slack, KakaoTalk) can follow the same pattern.
