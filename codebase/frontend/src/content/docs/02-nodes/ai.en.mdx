This page covers the three AI nodes: calling an LLM to generate answers, classifying text into categories, and pulling structured fields out of unstructured input.

## AI Agent

### When to use it

- To send prompts to an LLM for **answer generation**, **RAG-backed queries**, or **tool calling**.
- Enabling **Multi Turn** mode lets you run a blocking conversation where the user participates.
- Defining **Conditions** lets the LLM detect specific situations and **branch out through a condition port**.

Conditions are exposed to the LLM as special tools (`cond_*`). When the LLM calls only condition tools, the run branches to that port; when it calls them alongside regular tools, regular tools run first and the decision is re-evaluated. Up to 20 conditions allowed.

### Conversation Context

A workflow run keeps a **Conversation Thread** that accumulates user and AI Agent turns during execution. An AI Agent node can inject that thread into the LLM call automatically — you don't need to splice `{{ $thread.text }}` into the system prompt by hand, just set `contextScope`.

- `contextScope: none` — disables auto-injection; the LLM sees only this node's `systemPrompt` / `userPrompt`. (Default.)
- `contextScope: thread` — sends every turn from the workflow thread alongside the LLM call. Token usage can grow quickly in long conversations.
- `contextScope: lastN` — sends only the most recent `contextScopeN` turns (default 20). Use this to balance token cost against context retention.

Injection mode (`contextInjectionMode`) has two flavors.

- `messages` (default) — prepends per-turn `role: user/assistant` messages to the front of the LLM messages array. Plays well with tool calls and feels natural on chat models like OpenAI / Anthropic.
- `system_text` — renders the turns into a single text block appended to the system prompt. Use it when you want to keep the messages slot empty or target a single-system-prompt model.

By default the thread receives **only the final assistant response and user turns**. Turn on `includeToolTurns` to also persist KB / MCP / condition tool-call turns. Conversely, set `excludeFromConversationThread` to keep a node out of the thread entirely (handy for background classifier-style LLM calls).

To reference the thread directly from expressions, use `{{ $thread.length }}`, `{{ $thread.text }}`, or `{{ $thread.turns[0].data.email }}`. See [Variables and context · $thread](/docs/04-expression-language/variables-and-context#thread-conversation-thread) for details.

### Memory

When a conversation grows long, or when you need to remember a user across sessions, pick a **memory strategy** (`memoryStrategy`) to automate context management. This is a separate **management axis** from the Conversation Context (`contextScope`) above.

- `manual` (default) — uses the Conversation Context fields above as-is. No automatic summarization or cross-session recall; the thread is injected only as your `contextScope` dictates.
- `summary_buffer` — keeps recent turns intact but rolls older turns past the **token budget** (`memoryTokenBudget`, default 8000) into a summary. Long conversations keep earlier context without runaway token growth.
- `persistent` — adds **cross-session memory** that survives after the run ends, on top of the summary buffer. The next run recalls memory about the same user.

When you choose `persistent`, you can scope the memory with a key.

- `memoryKey` — the scope key for persistent memory. The same key recalls the same memory across runs. For example, passing a user identifier like `{{ $input.userId }}` carries that user's memory forward. **Leaving it empty isolates memory per execution**, so prior runs are not recalled.
- `memoryTopK` (default 5) — number of memory chunks recalled per turn. Independent of RAG's `ragTopK`.
- `memoryThreshold` (default 0.7) — minimum similarity (0–1) required to recall memory. Independent of RAG's `ragThreshold`.
- `memoryTtlDays` — retention period (in days) for persistent memory. When set, memories older than this expire and are no longer recalled. **Leave it empty to keep memories without expiry.**
- `summaryModel` / `extractionModel` — let you pick a **separate model** used only for the summary/extraction LLM calls. Leave them empty to reuse the node model (behavior unchanged). Choosing a low-cost model for these simpler-than-main tasks is a **cost-saving option**. (`summaryModel` is shown for `summary_buffer`/`persistent`, `extractionModel` for `persistent`.)

<Callout type="note">
When `memoryStrategy` is not `manual`, the automatic strategy manages context, so the Conversation Context fields (`contextScope`, etc.) above are hidden in the config UI. Fields like the token budget and memory key appear only for the strategy that uses them.
</Callout>

### Presentation Tools

Setting `presentationTools` lets the LLM render responses as **tables, charts, carousels, templates, or forms** directly inside the chat UI — without connecting separate Presentation nodes to the workflow graph. The LLM decides, based on context, which rendering to use.

Each registered entry exposes a virtual tool to the LLM: `render_table`, `render_chart`, `render_carousel`, `render_template`, or `render_form`.

<FieldTable rows={[
  {name: "type", required: true, type: "`table` / `chart` / `carousel` / `template` / `form`", description: "Presentation type to render. Duplicate types within the same node are not allowed.", default: "-"},
  {name: "description", required: false, type: "string", description: "LLM-facing tool description override. Defaults to the built-in description for each type when omitted.", default: "-"},
  {name: "defaults", required: false, type: "object", description: "Brand/style fixed values. Deep-merged with the LLM's payload, with `defaults` taking precedence. Use to lock columns, button labels, or layout so the LLM cannot override them.", default: "-"}
]} />

**Rendering behaviour:**

- `render_table` / `render_chart` / `render_carousel` / `render_template` — display-only. Rendered inline in the LLM response; they do not end workflow execution.
- `render_form` — interactive. In Multi Turn mode, when the LLM invokes this tool the workflow pauses in `waiting_for_input` state and waits for the user to submit the form. **The active form is rendered inline inside the AI response card**, just like other presentation tools — not stacked below the chat input. Once the user submits, the form data is delivered to the LLM's next turn as a tool result so the conversation can continue, and the form area inside the same card turns into a submitted-result summary. In Single Turn mode the tool call is silently dropped.

**Sending text while the form is active (form bypass):**

The chat input stays active even while a form is pending. If the user sends a regular message instead of filling the form, the backend fills the corresponding `render_form` tool result with `{type:'cancelled', reason:'user_sent_message_instead'}` so the LLM is told the form was cancelled, and the text is delivered as a normal user message in the next turn. The LLM then decides what to do next autonomously — re-render the form, reply with text, or call another tool.

**Size and error handling:**

- Carousel and Table follow the same 1 MB tail-truncate policy as their standalone nodes.
- Chart and Template are silently dropped when the payload exceeds 1 MB.
- On schema violations, the node retries once, then silently drops. The `error` port does not fire; any text response is surfaced normally.

<Example title="presentationTools configuration">
```ts
presentationTools: [
  { type: 'table', defaults: { columns: [{ field: 'id', label: 'ID' }] } },
  { type: 'chart' },
  { type: 'form', description: 'Request additional input from the user' },
]
```
</Example>

<Callout type="note">
The input schema of each Presentation node (Carousel / Table / Chart / Form / Template) is reused verbatim as the JSON Schema for the corresponding `render_*` virtual tool parameter. This means the payload the LLM generates has the same structure as if you had wired the node directly into the graph. See [Presentation nodes](/docs/02-nodes/presentation) for full per-node schema details.
</Callout>

### Example

<Example title="Customer inquiry — Single Turn with condition branching">
```ts
// config summary
{
  mode: "single_turn",
  llmConfigId: "…",
  systemPrompt: "You are a customer support agent.",
  userPrompt: "{{ $input.message }}",
  knowledgeBases: ["faq-kb-id"],
  mcpServers: [{ integrationId: "int_mcp_internal_search" }],
  conditions: [
    { id: "uuid-1", label: "Refund request detected",   prompt: "Customer asks for a refund or cancellation" },
    { id: "uuid-2", label: "Tech escalation",           prompt: "Complex technical issue requiring a human agent" }
  ]
}

// Output (on a condition port)
{
  response: "…",
  condition: { id: "uuid-1", label: "Refund request detected", reason: "…" },
  // Single Turn output echoes the full raw config so downstream nodes can read every field
  config: { mode: "single_turn", model: "gpt-4o", systemPrompt: "…", userPrompt: "{{ $input.message }}", knowledgeBases: ["faq-kb-id"], conditions: [...] },
  metadata: { model: "gpt-4o", totalTokens: 1600, toolCalls: 2, ragSources: [...] }
}
```
</Example>

<Callout type="warn">
In Multi Turn mode, user responses are **awaited indefinitely**. No timeout fires unless something external cancels or ends the run — design for long-lived waits.
</Callout>

<Callout type="note">
**Single Turn vs. Multi Turn — `config` echo differences.** Single Turn output echoes the full raw config (systemPrompt, userPrompt, maxTurns, conditions, knowledgeBases, …) verbatim. Multi Turn terminal outputs (`user_ended`, `max_turns`, condition ports) currently echo only `config: { mode, model }` and omit the rest. If you mix both modes in one workflow, expressions like `$node["AI Agent"].config.systemPrompt` may behave differently per mode.
</Callout>

- `class_0`, `class_1`, ... (dynamic per-category data ports)
- `fallback` (when no category matches)
- `error` (LLM API errors, timeouts, etc.)

### Example

<Example title="Classify inquiries and route">
```ts
// config
{
  llmConfigId: "…",
  inputField: "{{ $input.text }}",
  categories: [
    { name: "Billing",   description: "Payment, refund, subscription", examples: ["Refund request"] },
    { name: "Technical", description: "Bugs, technical issues",          examples: ["Cannot log in"] },
    { name: "General",   description: "General questions, feature requests", examples: [] }

```

<Callout type="note">
`__none__` is reserved and cannot be used as a category name. When the LLM returns `__none__` or nothing matches, the item leaves via the `fallback` port.

### Cross-session memory (persistent)

Setting `memoryStrategy` to `persistent` lets the extractor leverage **facts extracted in earlier sessions**.

- **Recall** — Just before the extraction LLM call, facts stored under the same memory key are recalled via semantic search and prepended to the system prompt. This happens once for Single Turn and on the first entry of Multi Turn.
- **Extraction** — After the extraction result is finalized (right after the turn is pushed to the thread), it is accumulated into memory asynchronously, without blocking the response path (hot-path non-blocking). Multi Turn pushes and extracts once when the conversation terminates.
- Memory is scoped by `memoryKey`; empty isolates per execution. It uses the **same scope-key rules** as AI Agent memory, so with the same key an AI Agent and an extractor can share each other's memory.
- Unlike AI Agent, `summary_buffer` is not offered — working-memory token compaction is meaningless for a single extraction task.
- Choosing `persistent` hides the Conversation Context (`contextScope`) fields in the config UI and reveals the memory key/threshold fields instead.

### Ports

- `out` (on successful completion)
- `error` (LLM API errors, timeouts, or exhausted schema-validation retries)

### Example

<Example title="Extract fields from a refund email">
```ts
// config (single_turn)
{
  inputField: "{{ $input.emailBody }}",
  outputSchema: [
    { name: "senderName",  type: "string", description: "Sender name",      required: true  },
    { name: "orderNumber", type: "string", description: "Order number",     required: true  },
    { name: "issueType",   type: "string", description: "Issue type",       required: true,
      enumValues: ["refund", "exchange", "question"] },
    { name: "amount",      type: "number", description: "Related amount",   required: false }
  ]
}

// Reference downstream
{{ $node["Info Extractor"].output.result.extracted.orderNumber }}
```
</Example>

<Callout type="tip">
When schema validation fails, the node **retries up to 2 times**, then routes to the `error` port. Add `enumValues` to a field to force the LLM to pick only from the allowed values.
</Callout>

<Callout type="warn">
In Multi Turn mode, leaving `inputField` empty skips the initial LLM call and waits for user input immediately (`turnCount: 0`). The settings UI hides that field to make the UX explicit. User responses are awaited indefinitely.
</Callout>

## Tips & notes

- This page covers the **AI Agent node that runs inside a workflow**. The in-editor [AI Assistant](/docs/03-workflow-editor/overview) is a separate feature — a chat companion that builds workflows for you. Workflows the Assistant generates may contain AI Agent nodes.
- Registering a tool is as simple as dropping the node into the canvas **Tool Area**. The AI Agent picks it up as a callable tool, and condition tools are auto-separated so their names don't clash with regular ones.
- LLM provider connections are managed under [LLM config](/docs/06-integrations-and-config/llm-config); Knowledge Base setup lives in [Knowledge Base](/docs/06-integrations-and-config/knowledge-base); MCP server registration and tool-exposure policy is in [MCP server integration](/docs/06-integrations-and-config/mcp-servers).
- Multi Turn execution history is inspectable per turn (request/response/token usage) via the Conversation Inspector. Details: [Run results](/docs/05-run-and-debug/run-results).
- Power-user tip: when `responseFormat=json`, the LLM returns a structure matching your schema. If you add, remove, or rename fields, also check downstream `{{ $node["AI Agent"].output.result.response.field }}` references.
- Downstream reference paths (Principle 1.1/8):
  - AI Agent response body: `$node["AI Agent"].output.result.response`
  - Text Classifier single result: `$node["TC"].output.result.category` · multi-label: `$node["TC"].output.result.categories`
  - Information Extractor result: `$node["IE"].output.result.extracted.<fieldName>`
  - Model/token metrics: `$node["X"].meta.{model, inputTokens, outputTokens, totalTokens}`
  - Error branching: `$node["X"].output.error?.code` / `$node["X"].port === 'error'`
