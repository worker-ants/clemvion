This page covers the five integration nodes that talk to external services (HTTP Request, Database Query, Send Email, Cafe24, and MakeShop). Most of them share a common pattern: they reference an **Integration entity** for reusable auth/connection info, and they route success and failure through separate output ports.

## Shared patterns

### Integration references

Every integration node exposes an `integrationId` field that points to an Integration entity (sidebar → Integrations) holding the credentials. Credentials are automatically encrypted and decrypted with AES-256-GCM. If the connection is not in the `connected` state, the run fails with `INTEGRATION_NOT_CONNECTED`.

## HTTP Request

### When to use it

- To **call an arbitrary HTTP API**.
- Reference an Integration to auto-inject auth headers, or call without auth.

### Fields

<FieldTable rows={[
  {name: "method", required: true, type: "enum", description: "GET / POST / PUT / PATCH / DELETE / HEAD / OPTIONS.", default: "GET"},
  {name: "url", required: true, type: "expression", description: "Request URL. When the Integration defines `base_url`, you can use a relative path that gets automatically joined.", default: "-"},
  {name: "authentication", required: true, type: "`none` / `integration` / `custom`", description: "Authentication mode.", default: "none"},
  {name: "integrationId", required: false, type: "UUID", description: "Required when `authentication=integration`.", default: "-"},
  {name: "headers", required: false, type: "KeyValue[]", description: "Request headers.", default: "[]"},
  {name: "queryParams", required: false, type: "KeyValue[]", description: "URL query parameters.", default: "[]"},
  {name: "body", required: false, type: "object", description: "Request body.", default: "-"},
  {name: "bodyType", required: false, type: "enum", description: "`json` / `form-data` / `x-www-form-urlencoded` / `raw` / `binary`.", default: "json"},
  {name: "responseType", required: false, type: "enum", description: "`json` / `text` / `binary`.", default: "json"},
  {name: "timeout", required: false, type: "integer (ms)", description: "Request timeout.", default: "30000"},
  {name: "followRedirects", required: false, type: "boolean", description: "Follow redirects.", default: "true"},
  {name: "verifySsl", required: false, type: "boolean", description: "Verify SSL certificates.", default: "true"}
]} />

### Ports

- `success` — HTTP 2xx response
- `error` — 4xx/5xx or network failure

<Callout type="note">
The response body is capped at **256KB**. If it overflows, the body is truncated and `bodyTruncated: true` is added to the output. For large responses, prefer paginating or narrowing fields server-side, or post-process via a Code node.
</Callout>

### Example

<Example title="Call an external API with a Bearer token">
```ts
// config
{
  method: "POST",
  url: "/tickets",                 // joined automatically when base_url is set on the credential
  authentication: "integration",
  integrationId: "…",
  headers:    [{ key: "Accept", value: "application/json" }],
  bodyType:   "json",
  body:       { title: "{{ $input.subject }}", body: "{{ $input.message }}" },
  timeout:    30000
}

// success port
{ data: { id: "TCK-1" }, meta: { statusCode: 201, headers: {...}, duration_ms: 250 } }

// error port
{ error: { statusCode: 500, message: "Internal Server Error", body: {...} }, meta: { duration_ms: 100 } }
```
</Example>

<Callout type="tip">
Based on the Integration's `auth_type`, headers/query params are auto-injected: api_key (header/query), bearer_token, and basic. Headers you set on the node **override** credential-injected headers (last wins).
</Callout>

## Database Query

### When to use it

- To **run SQL directly** against PostgreSQL or MySQL.
- SELECT to read data, or INSERT/UPDATE/DELETE to write.

### Fields

<FieldTable rows={[
  {name: "integrationId", required: true, type: "UUID", description: "Reference to a DB Integration. `driver` must be `postgres` or `mysql`.", default: "-"},
  {name: "queryType", required: true, type: "enum", description: "`select` / `insert` / `update` / `delete` / `raw`.", default: "select"},
  {name: "query", required: true, type: "expression", description: "SQL to execute. Bind values with `$1`, `$2`, ... (MySQL driver converts them to `?` internally).", default: "-"},
  {name: "parameters", required: false, type: "array or JSON string", description: "Values bound to placeholders in order. A JSON string is auto-parsed into an array.", default: "[]"}
]} />

### Ports

- `success` — query completed successfully
- `error` — runtime failures (connection/query errors). Pre-flight errors don't go through this port; they fail the whole workflow.

### Example

<Example title="SELECT with parameter binding">
```ts
// config
{
  integrationId: "…",
  queryType: "select",
  query: "SELECT id, name FROM users WHERE tenant_id = $1 AND status = $2",
  parameters: ["{{ $input.tenantId }}", "active"]
}

// success output
{
  rows: [{ id: 1, name: "Kim" }, ...],
  rowCount: 42,
  fields: [{ name: "id", dataTypeID: 23 }, ...],
  query: "…",
  queryType: "select",
  durationMs: 18,
  status: "ok"
}
```
</Example>

<Callout type="warn">
Pre-flight errors (`INTEGRATION_INCOMPLETE`, `INVALID_PARAMETERS`, `INTEGRATION_SERVICE_UNAVAILABLE`) are **thrown and fail the entire run**. Runtime SQL errors are routed to the `error` port instead. If you want to branch-handle failures, wire something to the `error` port.
</Callout>

## Send Email

### When to use it

- To send emails via **SMTP**.
- Supports To/CC/BCC, subject, body (text or HTML), and attachments.

### Fields

<FieldTable rows={[
  {name: "integrationId", required: true, type: "UUID", description: "Reference to an SMTP Integration. host, port, secure, username, password, default_from are required.", default: "-"},
  {name: "to", required: true, type: "string[]", description: "Recipient emails. Each entry accepts expressions.", default: "[]"},
  {name: "cc", required: false, type: "string[]", description: "CC recipients.", default: "[]"},
  {name: "bcc", required: false, type: "string[]", description: "BCC recipients.", default: "[]"},
  {name: "subject", required: true, type: "expression", description: "Email subject.", default: "-"},
  {name: "body", required: true, type: "expression", description: "Email body.", default: "-"},
  {name: "bodyType", required: false, type: "`text` / `html`", description: "Body format.", default: "text"},
  {name: "attachments", required: false, type: "Attachment[]", description: "Attachments. Each has `filename` and `content` (Base64 or URL).", default: "[]"}
]} />

### Ports

- `out` — delivery result

<Callout type="note">
The echoed `body` field in the output is capped at **256KB**. If it overflows, the body is truncated and `bodyTruncated: true` is added. The actual mail sent over SMTP is unaffected — only the echo body used by downstream nodes carries the cap.
</Callout>

### Example

<Example title="Send a welcome email">
```ts
// config
{
  integrationId: "smtp-integration-id",
  to: ["{{ $input.user.email }}"],
  subject: "Welcome, {{ $input.user.name }}",
  body: "<h1>Hello</h1><p>Thanks for joining.</p>",
  bodyType: "html"
}

// output
{
  data: {
    messageId: "<…@smtp.example.com>",
    accepted: ["user@example.com"],
    rejected: [],
    to: ["user@example.com"],
    cc: [],
    subject: "…",
    bodyType: "html",
    status: "sent",
    durationMs: 412
  }
}
```
</Example>

<Callout type="note">
To/CC/BCC accept both arrays and comma-separated strings. Values are trimmed, blanks dropped, and if no valid recipient remains the run fails with `"No valid recipients"`.
</Callout>

## Cafe24

### When to use it

- To call the Korean e-commerce SaaS **Cafe24** Admin API across its 18 categories (products, orders, customers, promotions, mileage, and so on — about 180 endpoints) from a workflow.
- The same Cafe24 integration is reused by both this node and the AI Agent's MCP tool surface, so OAuth tokens, leaky-bucket rate limits, and call logs stay unified.

### Fields

<FieldTable rows={[
  {name: "integrationId", required: true, type: "UUID", description: "Reference to an Integration with `service_type='cafe24'`. Create one from sidebar → Integrations → Cafe24 card.", default: "-"},
  {name: "resource", required: true, type: "enum", description: "Cafe24 category — one of 18 values: `store`, `product`, `order`, `customer`, `community`, `design`, `promotion`, `application`, `category`, `collection`, `supply`, `shipping`, `salesreport`, `personal`, `privacy`, `mileage`, `notification`, `translation`.", default: "-"},
  {name: "operation", required: true, type: "string", description: "Operation identifier for the selected resource — e.g. `product_list`, `product_get`, `order_list`, `order_update_status`. The metadata table fills the dropdown automatically.", default: "-"},
  {name: "fields", required: false, type: "Record<string, unknown>", description: "Input fields for the selected operation. Expression `{{ }}` allowed. Required vs. optional varies per operation.", default: "{}"},
  {name: "pagination", required: false, type: "object", description: "`{ limit?, offset?, cursor? }`. Meaningful only when the operation supports pagination.", default: "-"}
]} />

### Ports

- `success` — API call succeeded
- `error` — auth expired, leaky-bucket rate limit exceeded, Cafe24-side 5xx, etc.

### Example

<Example title="Fetch unshipped orders from yesterday to today">
```ts
// config
{
  integrationId: "…",
  resource: "order",
  operation: "order_list",
  fields: {
    shop_no: 1,
    order_status: "N00",
    start_date: "{{ formatDate(addTime($now, -1, \"day\"), \"YYYY-MM-DD\") }}",
    end_date:   "{{ formatDate($now, \"YYYY-MM-DD\") }}"
  },
  pagination: { limit: 50, offset: 0 }
}
```
</Example>

<Callout type="tip">
If this is your first time, register the integration from **sidebar → Integrations → Cafe24** first. The full onboarding (app type, OAuth callback, registering a Private app) is covered in the [Cafe24 integration guide](/docs/06-integrations-and-config/cafe24). That page also explains how to expose the same integration as an MCP tool on an AI Agent node.
</Callout>

## MakeShop

### When to use it

- To call the Korean e-commerce SaaS **MakeShop** Shop API across its 7 sections (shop, product, order, member, benefit, board, CPIK — about 161 endpoints) from a workflow.
- The same MakeShop integration is reused by both this node and the AI Agent's MCP tool surface, so OAuth tokens and call logs stay unified.

### Fields

<FieldTable rows={[
  {name: "integrationId", required: true, type: "UUID", description: "Reference to an Integration with `service_type='makeshop'`. Create one from sidebar → Integrations → MakeShop card.", default: "-"},
  {name: "resource", required: true, type: "enum", description: "MakeShop section — one of 7 values: `shop`, `product`, `order`, `member`, `benefit`, `board`, `cpik`.", default: "-"},
  {name: "operation", required: true, type: "string", description: "Operation identifier for the selected resource — e.g. `get-product`, `get-order-1`, `post-cart-create`. The metadata table fills the dropdown automatically.", default: "-"},
  {name: "fields", required: false, type: "Record<string, unknown>", description: "Input fields for the selected operation. Expression `{{ }}` allowed.", default: "{}"},
  {name: "pagination", required: false, type: "object", description: "`{ limit?, offset? }`. Meaningful only when the operation supports pagination.", default: "-"}
]} />

### Ports

- `success` — API call succeeded
- `error` — auth expired, API error, network failure, etc.

### Example

<Example title="Fetch a MakeShop product">
```ts
// config
{
  integrationId: "…",
  resource: "product",
  operation: "get-product",
  fields: {
    product_id: "{{ $input.productId }}"
  }
}

// success port
{ response: { product_id: "1001", product_name: "Sample product", … } }

// error port
{ error: { code: "MAKESHOP_API_ERROR", message: "…" } }
```
</Example>

<Callout type="tip">
If this is your first time, register the integration from **sidebar → Integrations → MakeShop** first. MakeShop integrations use an install-first flow: the integration stays in a pending state (pending_install) until the ShopStore app install completes and OAuth authentication finishes. The [MakeShop integration guide](/docs/06-integrations-and-config/makeshop) covers the full onboarding and how to expose the same integration as an MCP tool on an AI Agent node.
</Callout>

## Canvas summary

- HTTP Request: `{method} {url}` (URL trimmed at 35 chars)
- Database Query: `{queryType} · {first line of the query}` (trimmed)
- Send Email: `to: {recipient}` (with `+N` when there are more than 2)
- Cafe24: `{resource} · {operation}` (operation name trimmed when long)
- MakeShop: `{resource} · {operation}` (operation name trimmed when long)

When the referenced Integration is deleted, a `⚠ Missing integration` badge appears on the canvas summary.

## Tips & notes

- Adding integrations and managing their status is covered in [Integration management](/docs/06-integrations-and-config/integration-management).
- Each call is logged in the Usage log at run time. `status`, `durationMs`, and `error.code` help trace failure causes.
- Power-user tip: HTTP Request headers follow **Integration default headers < node-specified headers** (last wins). When the Integration has a `base_url`, node URLs like `/users/123` are joined automatically.
