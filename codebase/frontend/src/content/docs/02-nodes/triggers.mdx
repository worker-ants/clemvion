이 페이지에서는 트리거 노드의 역할과 설정 방법을 설명해요. 현재 캔버스에 배치 가능한 트리거 노드는 **Manual Trigger** 한 종류이고, 웹훅과 스케줄은 별도의 트리거 시스템에서 같은 파라미터 스키마를 공유하는 방식으로 동작해요.

## Manual Trigger

### 언제 쓰나요?

- 워크플로우에 **입력 파라미터를 정의**하고 싶을 때 써요.
- Run 버튼 또는 `POST /workflows/:id/execute`로 수동 실행하는 모든 워크플로우가 기본으로 갖고 있어요.
- 워크플로우를 만들면 자동으로 한 개 생기고, **삭제할 수 없어요**. 워크플로우당 정확히 한 개만 허용돼요.

### 예시

<Example title="입력 파라미터 스키마 선언과 참조">
```ts

  "parameters": [
    { "name": "email", "type": "string", "required": true, "description": "대상 이메일" },
    { "name": "count", "type": "number", "required": false, "defaultValue": 10 }
```

</Callout>

### 포트

Manual Trigger는 입력 포트가 없고, 출력 포트 `out` 1개를 포함해요. 워크플로우 실행 입력이 검증·기본값 적용·타입 변환을 거쳐 `{ parameters }` 형태로 나가요.

## 웹훅과 스케줄은 어떻게 이어지나요?

웹훅과 스케줄은 **별도의 트리거 시스템**이에요. 캔버스의 Manual Trigger 노드와 1:1로 매핑되며, Manual Trigger의 `parameters` 스키마를 **단일 소스**로 공유해요.

</Callout>

## Webhook 트리거 자세히

웹훅 트리거를 만들면 워크스페이스 도메인 아래에 고유 엔드포인트가 발급돼요. 외부 서비스(GitHub, Stripe, 자체 백엔드 등)가 이 URL로 HTTP POST를 보내면 해당 워크플로우가 실행돼요.

### URL과 메서드

```
POST {base_url}/api/hooks/{endpoint_path}
```

- `endpoint_path`는 트리거 생성 시 UUID로 자동 발급돼요 (브루트포스 방지). 트리거 목록 화면에서 📋 버튼으로 전체 URL을 복사할 수 있어요.
- 메서드는 `POST`만 받아요. `Content-Type`은 `application/json` 또는 `application/x-www-form-urlencoded`.
- 본문 최대 크기는 1MB예요. 초과 시 `413 Payload Too Large`.

### 인증 옵션

트리거 상세에서 인증 방식을 셋 중 하나로 설정해요. 검증 실패는 모두 동일한 `401 Unauthorized` 메시지로만 응답해요 (알고리즘이 무엇이었는지는 응답에 노출되지 않아요 — 정보 유출 방지).

<FieldTable rows={[
  { name: "None (공개)", required: false, type: "검증 없음", description: "누구나 호출할 수 있어요. URL의 UUID 자체가 사실상 비밀이에요. 외부 노출이 우려되는 워크플로우에는 권장하지 않아요.", default: "-" },

<Callout type="warn">

</Callout>

### 응답 코드

| 상태 | 의미 |
|------|------|
| `202 Accepted` | 수신 성공. 응답 본문에 `executionId`가 들어 있어요 — 실제 실행은 비동기로 진행돼요. |
| `400 Bad Request` | 본문 파싱 실패, 또는 `required` 파라미터 누락·타입 변환 실패. 응답 본문에 누락 필드 목록이 들어 있어요. **Execution은 생성되지 않아요**. |
| `401 Unauthorized` | 인증 실패. |
| `404 Not Found` | 해당 `endpoint_path`의 트리거가 존재하지 않아요. |
| `410 Gone` | 트리거가 비활성 상태예요. 트리거 상세에서 다시 활성화해야 받을 수 있어요. |
| `413 Payload Too Large` | 본문 크기 1MB 초과. |
| `429 Too Many Requests` | 분당 60건 rate limit 초과. |

### 워크플로우 안에서 웹훅 페이로드 접근

웹훅으로 들어온 실행은 Manual Trigger의 `output`에 transport 컨텍스트가 함께 채워져요.

<Example title="웹훅 페이로드 참조">
```ts
// $params 는 Manual Trigger 의 parameters 스키마로 추출·검증된 값
{{ $params.orderId }}                           // = body.orderId 가 string 으로 강제됨

// 원본 transport 는 Manual Trigger 노드의 output.request 에서 직접 접근
{{ $node["Manual Trigger"].output.request.method }}                // "POST"
{{ $node["Manual Trigger"].output.request.headers["x-event-type"] }} // 헤더 키는 소문자
{{ $node["Manual Trigger"].output.request.query.source }}
{{ $node["Manual Trigger"].output.request.body.raw }}              // 파싱된 원본 본문

// 실행 출처 분기
{{ $node["Manual Trigger"].meta.source }}        // "manual" | "webhook" | "schedule"
```
</Example>

<Callout type="note">
Manual / Schedule 어댑터로 실행된 경우에는 `output.request`가 아예 존재하지 않아요. 같은 워크플로우를 여러 트리거로 재사용한다면 `meta.source`로 분기하거나, `output.request?.method`처럼 옵셔널 체이닝을 써요.
</Callout>

## 외부 인터랙션 채널 (웹훅 + Form/AI 대화)

웹훅으로 시작된 실행이 도중에 **사람의 응답이 필요한 노드** (Form 노드, Presentation 버튼, AI Agent · Information Extractor의 Multi Turn) 에 도달하면, 외부 호출자가 그 사실을 알고 응답까지 보내야 워크플로우가 다시 진행돼요. 이를 위해 두 가지 채널을 옵션으로 켤 수 있어요. 둘 다 **선택 사항**이고, 인터랙션이 없는 단순 자동화는 따로 설정하지 않아도 돼요.

### 트리거 생성 페이로드에 추가하기

웹훅 트리거를 만들거나 수정할 때 두 옵션 필드를 추가해요.

<Example title="외부 인터랙션 옵션을 켠 트리거 생성">
```jsonc
POST /api/triggers
{
  "type": "webhook",
  "workflowId": "uuid",
  "endpointPath": "uuid-or-slug",
  "authType": "bearer",

  // Outbound: 우리 서버 → 외부 URL 로 이벤트 push
  "notification": {
    "url": "https://customer.example/webhook/wf-callback",
    "events": [
      "execution.waiting_for_input",
      "execution.completed",
      "execution.failed",
      "execution.cancelled"
      // "execution.ai_message"  // 옵션: AI Multi Turn 의 어시스턴트 메시지마다 발송 (노이즈 큼)
    ],
    "signing": { "algorithm": "hmac-sha256" },
    "retry":   { "maxAttempts": 5, "backoff": "exponential" }
  },

  // Inbound: 외부 → 우리 서버 로 인터랙션 명령
  "interaction": {
    "enabled": true,
    "tokenStrategy": "per_execution"   // "per_execution" (기본) | "per_trigger"
  }
}
```
</Example>

응답에서 `notification.secret`(`wsk_*`)과, `tokenStrategy="per_trigger"`인 경우의 `interaction.triggerToken`(`itk_*`)은 **생성 시점에 단 한 번만 평문으로 노출**돼요. 이후 모든 조회 응답에서는 마스킹되니, 그 자리에서 안전하게 저장하세요.

### Outbound Notification 동작

- 워크플로우가 트리거의 `events` 화이트리스트에 포함된 상태로 진입할 때마다 등록 URL로 POST를 보내요.
- 서명은 Stripe-style이에요 — `X-Clemvion-Signature: t=<unix>,v1=<hex>` 형태로 헤더에 동봉돼요. 검증 측은 `timestamp ± 5분` 이내, timing-safe 비교를 권장해요.
- 같은 이벤트의 재시도는 동일한 `X-Clemvion-Delivery: <uuid>` 헤더를 유지해요. 클라이언트는 이 UUID로 dedup하면 돼요 (at-least-once 보장).
- 페이로드의 `seq`는 execution 안에서 단조 증가하는 카운터예요. SSE 스트림의 이벤트 `id`와 같은 값을 공유해요 — 두 채널을 동시에 구독해도 정렬·dedup 가능.
- 전송은 2xx만 성공으로 봐요. 그 외 / 타임아웃 (10초) → 지수 백오프 (기본 5회: 1s · 4s · 16s · 64s · 256s) 재시도.
- 최종 실패 시 트리거의 `notificationHealth` 가 `degraded`로 바뀌고 트리거 상세 화면에 표시돼요. **트리거 본체는 자동으로 비활성화되지 않아요** — 워크플로우 실행은 계속 받아요. 사용자가 직접 비활성화 토글을 눌러야 멈춰요.
- URL은 `https://`만 허용해요 (개발 환경에서 `ALLOW_HTTP_HOOKS=1`일 때만 예외). 사설 IP·메타데이터 IP·loopback은 SSRF 방지로 차단돼요.

<Callout type="note">
시크릿이 유출됐다고 의심되면 `POST /api/triggers/:id/notification/rotate-secret`으로 회전할 수 있어요. 새 시크릿이 만들어진 뒤 24시간 동안은 old/new 둘 다 유효 — 외부 시스템이 새 시크릿으로 전환할 여유 시간이에요.
</Callout>

### Webhook 호출 응답이 바뀌어요 (interaction 활성화 시)

`interaction.enabled = true` + `tokenStrategy = "per_execution"`인 트리거는 웹훅 호출에 즉시 응답하는 본문이 확장돼요.

<Example title="확장된 202 응답">
```jsonc
{
  "executionId": "uuid",
  "status":      "pending",
  "message":     "Webhook received, workflow execution started",

  "interaction": {
    "token":     "iext_<short-lived-jwt>",
    "expiresAt": "ISO8601",
    "endpoints": {
      "stream":  "/api/external/executions/{id}/stream",
      "submit":  "/api/external/executions/{id}/interact",
      "status":  "/api/external/executions/{id}",
      "cancel":  "/api/external/executions/{id}/cancel",
      "refresh": "/api/external/executions/{id}/refresh-token"
    }
  }
}
```
</Example>

외부 시스템은 이 token으로 곧장 SSE 스트림을 열거나 인터랙션 명령을 보낼 수 있어요. `per_trigger` 전략을 골랐다면 token이 응답에 동봉되지 않아요 — 트리거 생성 시 받은 `itk_*`를 호출자가 이미 갖고 있다는 전제예요.

### Inbound 명령

```
POST /api/external/executions/{executionId}/interact
Authorization: Bearer <iext_jwt | itk_token>
Content-Type:  application/json
Idempotency-Key: <client-uuid>   // 권장
```

<Callout type="warn">
`Idempotency-Key`를 항상 동봉하세요. 동일 키 + 동일 body면 24시간 캐시된 응답을 그대로 받아요. 동일 키 + 다른 body면 `409 Conflict`. 같은 execution의 같은 노드에 두 명령이 거의 동시에 들어오면 뒤늦은 쪽은 `409 STATE_MISMATCH`로 거절돼요 (의도적 직렬화).
</Callout>

주요 에러 응답:

| 상태 | 코드 | 의미 |
|------|------|------|
| `400 Bad Request` | `VALIDATION_FAILED` | Form 필드 검증 실패. 실행 상태 유지 → 재제출 가능 |
| `400 Bad Request` | `INVALID_COMMAND` | 미지원 command 또는 필수 필드 누락 |
| `401 Unauthorized` | `TOKEN_INVALID` / `TOKEN_EXPIRED` | 응답 헤더 `X-Refresh-Token-Url` 로 갱신 경로 안내 |
| `409 Conflict` | `STATE_MISMATCH` | 현재 노드/실행 상태와 명령 불일치 |
| `410 Gone` | `EXECUTION_TERMINATED` | 이미 종료된 execution |
| `429 Too Many Requests` | `RATE_LIMITED` | execution 당 분당 60건 초과 |

### Inbound 이벤트 스트림 (SSE)

```
GET /api/external/executions/{executionId}/stream
Authorization: Bearer <iext_jwt | itk_token>
Accept:        text/event-stream
Last-Event-Id: 42         # 재연결 시
```

- `event:` 이름은 내부 WebSocket 이벤트와 정합해요 — `execution.started`, `execution.node.started/completed/failed/skipped`, `execution.waiting_for_input`, `execution.ai_message`, `execution.tool_call_started/completed`, `execution.completed/failed/cancelled` 등.
- `id:` 필드는 execution 안의 monotonic `seq` 예요. 끊겼다 재연결하면 `Last-Event-Id` 헤더로 5분 버퍼 안의 누락 이벤트를 받을 수 있어요.
- 15초마다 `: heartbeat` comment 라인을 보내요 — 중간 프록시의 idle timeout을 우회하려는 목적이에요.
- terminal 이벤트(`execution.completed` / `execution.failed` / `execution.cancelled`) 발송 후 서버가 자동으로 연결을 끊어요.
- 5분 버퍼가 만료된 경우 `execution.replay_unavailable` 이 한 번 발송돼요 — 그때는 `GET /api/external/executions/:id` 로 현재 상태를 다시 조회하세요.
- execution 당 동시 SSE 연결은 기본 3개로 제한돼요.

브라우저 EventSource API는 `Authorization` 헤더를 보낼 수 없으므로, SSE에 한해 `?token=<jwt>` 쿼리 파라미터도 받아요 (다른 endpoint는 항상 Bearer 헤더).

### 인터랙션 없는 자동화는 그대로

`notification`도 `interaction`도 켜지 않은 트리거는 기존 동작 그대로예요 — 웹훅이 들어오면 즉시 실행이 시작되고, 결과는 트리거 화면의 호출 이력 또는 워크플로우의 실행 내역에서 확인하면 돼요.

## 캔버스 요약

Manual Trigger는 파라미터 개수를 요약에 표시해요. 없으면 `(none)`으로 표시돼요.

## 팁 & 참고

- 파라미터 값 접근 문법은 [표현식 변수와 컨텍스트](/docs/04-expression-language/variables-and-context)에서 자세히 다뤄요.
- 실행 요청과 응답이 궁금하면 [워크플로우 실행](/docs/05-run-and-debug/running-a-workflow)을 참고해요.
- 파워유저 팁: Manual·Webhook·Schedule 어느 경로로 트리거되든 입력값은 동일하게 기본값이 채워지고 타입이 보정돼요. 같은 워크플로우를 여러 트리거로 재사용해도 파라미터가 일관되게 전달돼요.
