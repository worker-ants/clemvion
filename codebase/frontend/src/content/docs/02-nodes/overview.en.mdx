This page covers what a node is made of, which port rules it follows, and how data and errors flow. The full specs for individual nodes live on the category-specific pages that follow.

## What is a node

A node is a **unit of work** in a workflow. It takes input on the left, does its job based on configuration, and emits output on the right. Every node follows the same shared structure (type · category · input/output ports · config schema), so placing and connecting any node works the same way.

- **Type and category** describe what the node does (e.g. `if_else` → logic, `ai_agent` → ai).
- **Ports** define how data flows in and out.
- **Config schema** drives the form fields and their validation. The palette and settings-panel UI are generated directly from this schema.

## Ports

### Inputs and outputs

- **Input ports** sit on the left, **output ports** on the right.
- Default identifiers are `in` and `out`. Branching nodes use role-based IDs like `true`/`false` or `success`/`error`.
- Dynamically-added ports (Switch cases, Text Classifier categories, Presentation button ports) get a **UUID v4** as their ID. Renaming or reordering leaves the ID untouched, so **existing connections keep working**.

<Callout type="note">
Container nodes (Loop, ForEach, Map) start the inner subgraph from `body` and collect results back via the `emit` port. You must wire **exactly one** node to `emit`.
</Callout>

## Data flow

Data passed between nodes is JSON. The previous node's output becomes the next node's input.

Expression references look like this:

<Example title="Cross-node references">
```ts
// Output of the directly connected previous node
{{ $input.text }}

// Reference a specific node by name
{{ $node["Info Extractor"].output.result.extracted.amount }}

// Variables, execution id, current time
{{ $var.counter }}
{{ $execution.id }}
{{ $now }}
```
</Example>

## Error policies

Each node decides how to react to failures.

<FieldTable rows={[
  {name: "Stop Workflow", required: false, type: "Default", description: "Halts the entire run immediately and leaves it in the `failed` state.", default: "Selected"},
  {name: "Skip Node", required: false, type: "Option", description: "Skips this node and continues. The output is `null`.", default: "-"},
  {name: "Use Default Output", required: false, type: "Option", description: "Treats the failure as success using a default output you specify. A badge on the canvas indicates the default was used.", default: "-"},
  {name: "Retry", required: false, type: "Option", description: "Retries with a configurable max count and delay.", default: "-"},
  {name: "Route to Error Port", required: false, type: "Option", description: "Sends the error payload out through the `error` port. Selecting it dynamically adds the error port to the node.", default: "-"}
]} />

<Callout type="tip">
With `Route to Error Port`, if nothing is connected to the error port, the node automatically falls back to `Stop Workflow`. Always wire a follow-up node onto the error path.
</Callout>

## Canvas summary

The third line of the node body shows a one-line config summary. If required fields are empty, an amber `⚠ Not configured` badge appears. The summary hides when zoom drops below 50%.
