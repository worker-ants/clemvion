이 페이지에서는 LLM을 호출해서 답변을 만들거나, 텍스트를 분류하거나, 비정형 데이터에서 필드를 뽑아내는 AI 노드 세 가지를 설명해요.

## AI Agent

### 언제 쓰나요?

- LLM에 프롬프트를 보내 **답변 생성**, **RAG 기반 질의**, **도구 호출**을 수행하고 싶을 때 써요.
- **Multi Turn** 모드를 켜면 사용자와 대화형으로 주고받는 블로킹 실행을 할 수 있어요.
- 조건(Conditions)을 정의하면 LLM이 특정 상황을 감지해 **조건 포트로 분기**할 수 있어요.

<Callout type="note">
조건(Conditions)은 LLM에게 특별한 도구(`cond_*`)로 제공돼요. LLM이 조건 도구만 호출하면 해당 포트로 분기하고, 일반 도구와 함께 호출하면 일반 도구를 먼저 실행한 뒤 재평가해요. 최대 20개 조건까지 허용해요.
</Callout>

### Conversation Context

워크플로우는 실행 동안 사용자·AI Agent 의 대화 turn 을 누적한 **Conversation Thread** 를 유지해요. AI Agent 노드는 이 thread 를 LLM 호출 직전에 자동으로 끼워 넣을 수 있어요. 직접 `{{ $thread.text }}` 같은 표현식으로 system 프롬프트에 박아 넣을 필요 없이, `contextScope` 만 설정하면 끝이에요.

- `contextScope: none` — 자동 주입을 끄고 본 노드의 `systemPrompt`/`userPrompt` 만 LLM 에 보내요. (기본값)
- `contextScope: thread` — 워크플로우 thread 의 모든 turn 을 LLM 에 함께 보내요. 긴 대화에서는 토큰 사용량이 빠르게 늘 수 있어요.
- `contextScope: lastN` — 가장 최근 `contextScopeN` 개(기본 20) turn 만 보내요. 토큰 비용과 맥락 유지의 균형을 잡고 싶을 때 써요.

주입 방식(`contextInjectionMode`)은 두 가지가 있어요.

- `messages` (기본) — LLM messages 배열의 앞쪽에 turn 별 `role: user/assistant` 메시지로 prepend 해요. 도구 호출과 매끄럽게 섞이고, OpenAI/Anthropic 등 대부분의 채팅 모델에서 자연스러워요.
- `system_text` — turn 들을 하나의 텍스트 블록으로 렌더해서 system 프롬프트 뒤에 append 해요. messages 슬롯을 비워 두고 단일 system 프롬프트 모델에 적합해요.

기본적으로 thread 에는 **최종 assistant 응답과 사용자 turn 만** push 돼요. KB/MCP/조건 도구 호출까지 보존하고 싶으면 `includeToolTurns` 를 켜요. 반대로 특정 AI Agent 노드를 thread 에서 제외하고 싶으면 `excludeFromConversationThread` 를 켜요(예: 백그라운드 분류용 LLM 호출).

표현식에서 thread 를 직접 참조하려면 `{{ $thread.length }}`, `{{ $thread.text }}`, `{{ $thread.turns[0].data.email }}` 처럼 쓸 수 있어요. 자세한 내용은 [변수와 컨텍스트 §$thread](/docs/04-expression-language/variables-and-context#thread-conversation-thread) 를 참고해요.

### 예시

<Example title="고객 문의 처리 — Single Turn + 조건 분기">
```ts
// config 요약
{
  mode: "single_turn",
  llmConfigId: "…",
  systemPrompt: "당신은 고객 지원 에이전트예요.",
  userPrompt: "{{ $input.message }}",
  knowledgeBases: ["faq-kb-id"],
  mcpServers: [{ integrationId: "int_mcp_internal_search" }],
  conditions: [
    { id: "uuid-1", label: "환불 요청 감지",     prompt: "고객이 환불/취소를 원할 때" },
    { id: "uuid-2", label: "기술 에스컬레이션",   prompt: "복잡한 기술 문제로 상담사 연결이 필요할 때" }
  ]
}

// 출력(조건 포트)
{
  response: "…",
  condition: { id: "uuid-1", label: "환불 요청 감지", reason: "…" },
  // Single Turn 출력은 입력 raw config 전체를 echo 해요 — 다운스트림에서 그대로 참조 가능
  config: { mode: "single_turn", model: "gpt-4o", systemPrompt: "…", userPrompt: "{{ $input.message }}", knowledgeBases: ["faq-kb-id"], conditions: [...] },
  metadata: { model: "gpt-4o", totalTokens: 1600, toolCalls: 2, ragSources: [...] }
}
```
</Example>

<Callout type="warn">
Multi Turn 모드에서 사용자 응답은 **무제한 대기**해요. 외부 cancel/종료가 아니면 타임아웃이 발생하지 않으니, 장시간 대기를 전제로 설계해요.
</Callout>

<Callout type="note">
**Single Turn vs Multi Turn — `config` echo 차이.** Single Turn 출력은 입력한 raw config 전체(systemPrompt · userPrompt · maxTurns · conditions · knowledgeBases 등)를 그대로 echo 해요. Multi Turn 의 종료 출력(`user_ended` / `max_turns` / 조건 포트)은 현재 `config: { mode, model }` 까지만 echo 하고 나머지 필드는 생략돼요. 두 모드를 한 워크플로우에서 섞어 쓸 때 `$node["AI Agent"].config.systemPrompt` 같은 참조가 모드별로 다르게 동작할 수 있다는 점에 주의해요. 전체 echo 통일은 후속 작업으로 예정돼 있어요.
</Callout>

## Text Classifier

### 언제 쓰나요?

- 입력 텍스트를 **미리 정의한 카테고리**로 분류하고 싶을 때 써요.
- 카테고리별로 서로 다른 후속 노드로 라우팅하고 싶을 때 유용해요.

### 필드

  {name: "instructions", required: false, type: "string", description: "추가 분류 지시사항이에요.", default: "-"},

### 포트

- `class_0`, `class_1`, … (카테고리별 동적 포트, 데이터 타입)
- `fallback` (어떤 카테고리에도 매칭되지 않을 때)
- `error` (LLM API 오류, 타임아웃 등)

### 예시

<Example title="문의 분류 후 라우팅">
```ts
// config
{
  llmConfigId: "…",
  inputField: "{{ $input.text }}",
  categories: [
    { name: "Billing",   description: "결제·환불·구독 관련",   examples: ["환불 요청"] },
    { name: "Technical", description: "기술 문제, 버그 리포트", examples: ["로그인 안돼요"] },
    { name: "General",   description: "일반 문의·기능 요청",   examples: [] }
  ],
  includeConfidence: true
}

// 출력 (Single-label)
{ category: "Billing", confidence: 0.95, originalInput: "환불해주세요" }
```
</Example>

<Callout type="note">
카테고리 이름으로 `__none__`은 예약어라 사용할 수 없어요. LLM이 `__none__`을 반환하거나 매칭 실패 시 `fallback` 포트로 빠져요.
</Callout>

## Information Extractor

### 언제 쓰나요?

- 비정형 텍스트(이메일, 메시지 등)에서 **정해진 스키마**의 값을 뽑아 구조화된 JSON으로 만들고 싶을 때 써요.
- Multi Turn 모드를 켜면 필요한 필드가 채워질 때까지 **사용자와 대화하며** 값을 수집할 수 있어요.

### 필드

  {name: "examples", required: false, type: "ExampleDef[]", description: "Few-shot 예시(입력 텍스트와 기대 출력)예요.", default: "[]"},

### 포트

- `out` (정상 완료 시)
- `error` (LLM API 오류, 타임아웃, 스키마 검증 재시도 소진 시)

### 예시

<Example title="환불 이메일에서 정보 추출">
```ts
// config (single_turn)
{
  inputField: "{{ $input.emailBody }}",
  outputSchema: [
    { name: "senderName",  type: "string", description: "발신자 이름",       required: true  },
    { name: "orderNumber", type: "string", description: "주문 번호",          required: true  },
    { name: "issueType",   type: "string", description: "문제 유형",          required: true,
      enumValues: ["refund", "exchange", "question"] },
    { name: "amount",      type: "number", description: "관련 금액",          required: false }
  ]
}

// 다운스트림에서 참조
{{ $node["Info Extractor"].output.result.extracted.orderNumber }}
```
</Example>

<Callout type="tip">
스키마 검증에 실패하면 **최대 2회 재시도**하고, 이후에는 `error` 포트로 라우팅해요. 필드에 `enumValues`를 넣어 두면 LLM이 허용된 값 범위 안에서 답하도록 강제할 수 있어요.
</Callout>

<Callout type="warn">
Multi Turn 모드에서 `inputField`를 비워 두면 LLM을 호출하지 않고 바로 사용자 입력을 기다려요(`turnCount: 0`). 설정 UI도 해당 필드를 숨겨서 이 UX를 명확히 해요. 사용자 응답은 무제한 대기예요.
</Callout>

## 캔버스 요약

- AI Agent: `{mode} · {model}` + `· {N} tools / {N} KB / {N} cond` (해당 시 추가).
- Text Classifier: `{model} · {N} categories`.
- Info Extractor: `{model} · {N} fields` (Multi Turn이면 `Multi Turn` 접두어).

## 팁 & 참고

- 본 페이지는 **워크플로우 안에서 실행되는 AI Agent 노드**를 다뤄요. 캔버스 안에서 채팅으로 워크플로우 자체를 만들어 주는 [AI 어시스턴트](/docs/03-workflow-editor/overview)와는 별개의 기능이에요. 어시스턴트가 만드는 워크플로우 안에 AI Agent 노드가 포함될 수도 있어요.
- 도구 등록은 캔버스의 **Tool Area**에 노드를 끌어다 놓으면 끝이에요. AI Agent가 해당 노드를 도구로 호출할 수 있게 되고, 조건 도구는 일반 도구와 이름이 겹치지 않도록 자동으로 분리돼요.
- LLM 프로바이더 연동은 [LLM 설정](/docs/06-integrations-and-config/llm-config)에서, Knowledge Base 구성은 [지식 저장소](/docs/06-integrations-and-config/knowledge-base)에서 관리해요. MCP 서버 등록과 도구 노출 정책은 [MCP 서버 통합](/docs/06-integrations-and-config/mcp-servers)에서 관리해요.
- Multi Turn 실행 이력은 Conversation Inspector로 턴별 요청/응답/토큰 사용량을 확인할 수 있어요. 자세한 동작은 [실행 결과](/docs/05-run-and-debug/run-results)를 참고해요.
- 파워유저 팁: 응답 형식을 `json`으로 지정하면 LLM이 정의한 출력 스키마에 맞춘 구조로 돌려줘요. 필드를 추가·제거·이름 변경할 때는 다운스트림 노드의 `{{ $node["AI Agent"].output.result.response.field }}` 참조 경로도 같이 점검해요.
- 다운스트림 참조 경로 (신규, Principle 1.1/8):
  - AI Agent 응답 본문: `$node["AI Agent"].output.result.response`
  - Text Classifier 단일 결과: `$node["TC"].output.result.category` · multi-label: `$node["TC"].output.result.categories`
  - Information Extractor 추출 결과: `$node["IE"].output.result.extracted.<fieldName>`
  - 모델/토큰 메트릭: `$node["X"].meta.{model, inputTokens, outputTokens, totalTokens}`
  - 에러 분기: `$node["X"].output.error?.code` / `$node["X"].port === 'error'`
