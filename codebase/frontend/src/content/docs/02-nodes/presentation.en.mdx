This page covers the five presentation nodes that either **visualize** a workflow's results or **collect user input**. Carousel, Table, Chart, and Template can also operate in Human-in-the-loop mode via buttons; Form is a blocking input node by default.

## Shared pattern — buttons and blocking mode

Carousel, Table, Chart, and Template share a `buttons` field. As soon as any button is configured, the run enters the **waiting_for_input** state and waits indefinitely for a user click.

Up to **10 buttons per node**. If a port button's dynamic port has no downstream connection, a warning is shown (not an error). The run waits indefinitely until a button is clicked — design for long waits.

## Carousel

### When to use it

- To present data as a **slideshow**.
- Define slides by hand in **static** mode, or map an array automatically in **dynamic** mode.

## Table

### When to use it

- To display data in a **table** with sorting and pagination.
- Define rows by hand in static mode, or map from an array source in dynamic mode.

### Fields

<FieldTable rows={[
  {name: "mode", required: false, type: "`static` / `dynamic`", description: "Static or dynamic mode.", default: "dynamic"},
  {name: "dataSource", required: false, type: "expression", description: "Array data source in dynamic mode. Falls back to the previous node's input when empty.", default: "-"},
  {name: "columns", required: true, type: "ColumnDef[]", description: "Column definitions. Each has `field`, `label`, `width`, `sortable`, `format`. `field` accepts a dot-path or per-item `{{ }}` expression.", default: "[]"},
  {name: "rows", required: false, type: "RowDef[]", description: "Static-mode row data. Each row is an object keyed by column field.", default: "[]"},
  {name: "pagination", required: false, type: "boolean", description: "Toggle pagination.", default: "true"},
  {name: "pageSize", required: false, type: "integer (1–200)", description: "Rows per page.", default: "20"},
  {name: "sortBy", required: false, type: "string", description: "Default sort column field.", default: "-"},
  {name: "sortOrder", required: false, type: "`asc` / `desc`", description: "Default sort direction.", default: "asc"},
  {name: "buttons", required: false, type: "ButtonDef[]", description: "Blocking buttons.", default: "[]"}
]} />

In dynamic-mode column expressions, `$dataSource`, `$sourceItem`, and `$sourceItemIndex` are available as per-item variables.

### Example

<Example title="Dynamic mode + per-item expression columns">
```ts
// config
{
  mode: "dynamic",
  dataSource: "{{ $node[\"API\"].output.users }}",
  columns: [
    { field: "name", label: "Name", width: "200px" },
    { field: "{{ $sourceItem.first + ' ' + $sourceItem.last }}", label: "Full name" },
    { field: "score", label: "Score", sortable: true, format: "0,0.00" }
  ],
  pagination: true,
  pageSize: 20,
  sortBy: "score",
  sortOrder: "desc"
}
```
</Example>

<Callout type="note">
Table has pagination on by default. With a hard cap of 200 rows/page, large datasets are surfaced safely.
</Callout>

## Form

### When to use it

- When you need **user input** in the middle of a workflow (Human-in-the-loop).
- Execution pauses, then resumes (after validation) when the form is submitted.

### Fields

  {name: "description", required: false, type: "markdown", description: "Form description.", default: "-"},

  {name: "fields", required: true, type: "FormField[]", description: "Form fields. Each has `name`, `type`, `label`, `required`, `options`, `defaultValue`, `validation`.", default: "[]"}

File-field-specific settings:

  {name: "allowedMimeTypes", required: false, type: "string[]", description: "Allowed MIME types. Defaults to images, PDFs, office docs, and text.", default: "built-in default list"},
  {name: "maxFileSize", required: false, type: "number (MB)", description: "Max size per file.", default: "10"},
  {name: "maxTotalSize", required: false, type: "number (MB)", description: "Max total size across all files in the field.", default: "50"},
  {name: "maxFiles", required: false, type: "number", description: "Max files per field.", default: "5"}
]} />

### Example

<Example title="Approval request form">
```ts
// config
{
  title: "Approval Request",
  description: "Review the details and choose whether to **approve**.",
  submitLabel: "Submit",
  fields: [
    {
      name: "approval", type: "select", label: "Decision", required: true,
      options: [{ label: "Approve", value: "approved" }, { label: "Reject", value: "rejected" }]
    },
    { name: "comment", type: "textarea", label: "Comment", required: false,
      validation: { maxLength: 500, message: "Please keep the comment under 500 characters" } }
  ]
}

// Output via `out` port after submission
{
  type: "form",
  submittedData: { approval: "approved", comment: "Looks good" },
  submittedAt: "…",
  submittedBy: "user-uuid"
}
```
</Example>

<Callout type="warn">
Form runs wait **indefinitely until submission**. The run state flips to `waiting_for_input`; no timeout fires unless cancelled or ended externally. Executables (.exe, .sh), scripts (.js, .py), and archives (.zip) are not in the default allowed MIME list — extend it explicitly when needed.
</Callout>

## AI tool mode (Presentation Tools)

The input schema of all five nodes in this category (Carousel / Table / Chart / Form / Template) is reused verbatim as the JSON Schema for the corresponding `render_*` virtual tool parameters in the AI Agent's `presentationTools` setting.

In other words, you do not need to wire Presentation nodes into the workflow graph at all. When `presentationTools` is configured on an AI Agent node, the LLM can produce the same rendering directly inside the chat UI, based on its own judgment.

**Behaviour summary:**

- `render_table` / `render_chart` / `render_carousel` / `render_template` — display-only. Rendered inline in the LLM response without ending workflow execution.
- `render_form` — interactive. In Multi Turn mode, presents a form to the user and waits for submission. Silently dropped in Single Turn mode.
- `defaults` overlay: the `defaults` field in the AI Agent `presentationTools` config pins brand/style values (columns, buttons, layout, etc.). On deep-merge, `defaults` takes precedence over what the LLM generates.
- 1 MB cap: Carousel and Table use tail truncation; Chart and Template are silently dropped when the cap is exceeded.
- Schema violations: retried once, then silently dropped. The `error` port does not fire.

For full details (defaults overlay, 1 MB cap, schema violation fallback), see [AI nodes § Presentation Tools](/docs/02-nodes/ai#presentation-tools).

## Canvas summary

- Carousel: `{layout} · {titleField}` or `{layout} · {N} buttons`
- Table: `{N} columns · pagination` or `{N} columns · {N} buttons`
- Chart: `{chartType} · {xAxis.field} / {yAxis.field}` or `{chartType} · {N} buttons`
- Form: `{N} fields · "{title}"`
- Template: `{outputFormat} · {N} lines` or `{outputFormat} · {N} buttons`

## Tips & notes

- How each presentation node renders in the run-results drawer is covered in [Run results](/docs/05-run-and-debug/run-results).
- When button-blocking, form waits, or AI conversations combine in one workflow, the run flips to `waiting_for_input` and stays paused until the user clicks a button or submits the form — it resumes automatically on user action.
- Power-user tip: Presentation nodes with buttons auto-create a dedicated output port per button. Only the clicked button's port carries the flow onward, so you can design per-button follow-ups independently.
