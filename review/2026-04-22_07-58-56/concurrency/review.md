### 발견사항

해당 변경사항은 주로 동기 로직(사이클 탐지 알고리즘 수정 + 테스트 추가)이며, 실질적인 동시성 위험은 없다. 다만 설계 수준에서 아래 한 가지를 언급할 만하다.

- **[INFO]** `ShadowWorkflow` 인스턴스 공유 시 잠재적 경쟁 조건
  - 위치: `shadow-workflow.ts` — `this.nodes`, `this.edges` (인스턴스 필드)
  - 상세: `nodes`/`edges`는 변경 가능한 `Map`이다. 현재 테스트 패턴과 코드 구조상 요청마다 `new ShadowWorkflow(snapshot, ...)` 으로 인스턴스를 생성하는 것으로 보여 문제없다. 그러나 NestJS 서비스 레이어에서 이 인스턴스를 캐시하거나 싱글턴으로 재사용할 경우, Node.js 이벤트 루프의 단일 스레드 보장에도 불구하고 `async` 함수 사이의 인터리빙으로 경쟁 조건이 생길 수 있다.
  - 제안: 호출 지점(서비스/핸들러)에서 인스턴스가 요청 스코프(`REQUEST` scope) 또는 함수 내 지역 변수로 생성되는지 명시적으로 확인하고, 주석 또는 타입으로 의도를 고정.

- **[INFO]** `wouldCreateCycle` 내 `edges.values()` 순회 중 외부 수정 가능성
  - 위치: `shadow-workflow.ts:331–343`
  - 상세: `wouldCreateCycle` 호출 이전에 유효성 검사를 완료하고 이후에만 `this.edges.set()`을 호출하므로, 단일 동기 호출 흐름에서는 안전하다. Node.js 단일 스레드 모델 덕분에 실질적 위험은 없다.
  - 제안: 현 구조 유지. 향후 Worker Thread 도입 시 재검토 필요.

### 요약

변경된 코드는 전부 동기 로직(DFS 사이클 탐지 + `containerId` 체인 순회)으로 구성되어 있으며, `async/await`·I/O·공유 상태 접근이 없다. Node.js의 단일 스레드 이벤트 루프 하에서 `apply()` 한 번의 호출은 중단 없이 완료되므로 경쟁 조건·데드락·원자성 문제가 발생하지 않는다. `isContainerAncestor`의 순환 방어(`visited` Set)도 적절히 구현되어 있다. 유일한 주의점은 인스턴스가 요청 범위를 벗어나 재사용될 경우인데, 현재 코드 패턴상 그런 설계는 보이지 않는다.
