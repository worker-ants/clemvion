### 발견사항

- **[INFO]** 테스트 코드에 하드코딩된 API 키 리터럴 사용
  - 위치: `makeClientWithStreamResult` 및 개별 테스트 케이스 (line 33, 204, 224, 243, 263, 285)
  - 상세: `new GoogleClient('test-key', ...)` 형태로 플레이스홀더 키를 직접 사용. 테스트 환경이므로 실제 위협은 없으나, 동일 패턴이 프로덕션 코드로 복사될 경우 실제 키 하드코딩으로 이어질 수 있음.
  - 제안: 관행적으로 `process.env.TEST_API_KEY ?? 'test-key'` 또는 상수로 분리하여 패턴을 명확히 구분.

- **[INFO]** `@ts-expect-error`를 통한 내부 필드 직접 접근
  - 위치: `client.genAI = { ... }` (line 38~43, 210~216 등)
  - 상세: 타입 시스템을 우회하여 내부 SDK 인스턴스를 교체하는 방식. 테스트 목적으로 의도된 패턴이지만, 실제 클라이언트에서 `genAI` 필드가 `public`이거나 보호 수준이 낮다면 외부에서도 동일하게 교체 가능하여 의존성 주입 공격 벡터가 될 수 있음.
  - 제안: 프로덕션 코드(`google.client.ts`)에서 `genAI` 필드를 `private readonly`로 선언되어 있는지 확인. 테스트용으로는 `jest.spyOn` 또는 DI 기반 목킹으로 교체 고려.

- **[INFO]** 에러 메시지에 HTTP 상태 코드 문자열 포함 (테스트 입력값)
  - 위치: line 226 `new Error('401 Unauthorized')`, line 245 `new Error('429 Too Many Requests')`
  - 상세: 테스트가 에러 메시지 문자열 파싱으로 에러 분류를 검증하고 있음. 이는 프로덕션 코드가 에러 메시지 문자열에서 상태 코드를 파싱한다는 의미. 외부 SDK가 에러 메시지 포맷을 변경하면 분류 로직이 깨질 수 있고, 정교하게 조작된 에러 메시지로 분류를 우회할 가능성이 있음.
  - 제안: `google.client.ts`에서 에러 분류를 메시지 문자열 파싱이 아닌 SDK가 제공하는 구조화된 에러 타입 또는 HTTP 상태 코드 속성 기반으로 처리하는지 확인.

### 요약

이 파일은 테스트 코드이므로 직접적인 보안 취약점은 없다. 다만 세 가지 간접적 신호가 주목할 만하다: (1) 프로덕션 코드의 `genAI` 필드 접근 수준이 지나치게 느슨할 경우 외부 교체 위험, (2) 에러 분류 로직이 메시지 문자열 파싱에 의존한다면 분류 우회 또는 SDK 업데이트에 취약, (3) API 키 리터럴 하드코딩 패턴이 프로덕션으로 전파될 수 있는 관습적 위험. 모두 테스트 파일 자체보다는 테스트가 암시하는 프로덕션 코드 설계의 보안 시사점이다.
