# 보안(Security) Review

## 발견사항

### 파일 1: workspace.decorator.spec.ts

- **[INFO]** 테스트 커버리지 강화 — 보안상 긍정적 변경
  - 위치: 전체 diff (lines 39-54)
  - 상세: `WORKSPACE_ID_REQUIRED` 에러 코드 단언 테스트 추가, 빈 문자열 헤더(falsy) 케이스 커버리지 추가. 이로써 `X-Workspace-Id: ""` 와 같은 falsy 입력이 실제로 BadRequestException 을 발생시킴을 검증한다. 데코레이터 수준의 입력 검증이 올바르게 동작하는지 확인하는 회귀 테스트로 보안 관점에서 바람직하다.
  - 제안: 현 수준 유지. 추가로 헤더 값에 UUID 형식이 아닌 값(예: 공백 문자열만, 매우 긴 문자열)을 넣었을 때 동작을 확인하는 테스트도 장기적으로 고려할 수 있다.

### 파일 5: spec/5-system/1-auth.md

- **[INFO]** 인증 토큰 24h 유효 기간 명시 추가 — 보안 문서 정확도 향상
  - 위치: §1.1 표 `인증 메일 재발송` 행 (line 1085)
  - 상세: `POST /api/auth/resend-verification` 에서 발급되는 인증 토큰이 24h 유효함을 §5 와 동기화해 명시한다. 이 변경은 문서 정확성 개선이며, 실제 보안 로직에 영향을 주지 않는다. 24h 유효 기간은 이메일 인증 토큰으로서 적절한 수준이다(비밀번호 재설정 토큰 30분 대비 더 길지만, 이메일 인증은 계정 탈취 위협 수준이 상대적으로 낮고 사용자 편의가 중요). 스펙 §5 에 이미 정의된 계약을 §1.1 에도 반영한 일관성 개선이다.
  - 제안: 이상 없음. 다만 향후 이 토큰 유효 기간 변경 시 §1.1 과 §5 양쪽을 동시 갱신하는 규약을 별도로 명문화하는 것을 고려할 수 있다.

### 파일 6: spec/5-system/11-mcp-client.md

- **[INFO]** Internal Bridge 표에 `makeshop`/`MakeshopMcpToolProvider` 행 추가 — 보안 중립
  - 위치: §3.1 Internal Bridge 표 (신규 행)
  - 상세: 기존 `cafe24` 패턴과 동일하게 `makeshop` service_type 을 Internal Bridge 구현체로 등록하는 스펙 문서 갱신이다. Internal Bridge (§2.3) 는 외부 fetch 가 없고 SSRF 검증이 불필요한 in-process 호출임을 이미 스펙이 명시하고 있다. 새로 추가된 `MakeshopMcpToolProvider` 도 동일 패턴으로, SSRF 표면이 생기지 않는다.
  - 제안: 이상 없음. `MakeshopMcpToolProvider` 구현 시 Cafe24 패턴처럼 rate limit, 인증 실패 자동 격하 정책(§8.4), mutex 동시성 제어가 동일하게 적용되는지 구현 리뷰 시 확인을 권장한다.

## 요약

이번 변경은 `WORKSPACE_ID_REQUIRED` 에러 코드 통일을 중심으로 한 테스트 강화, i18n 라벨 추가, 스펙 문서 동기화로 구성된다. 보안 관점에서 실질적인 취약점이 도입되지 않았으며, 오히려 `workspace.decorator.spec.ts` 의 falsy 헤더 케이스 및 에러 코드 단언 테스트 추가는 입력 검증 회귀를 방지하는 긍정적 변화다. 에러 메시지 내용이 내부 구현을 노출하지 않고 사용자 친화적으로 작성되었으며, `1-auth.md` 의 인증 토큰 유효 기간 명시는 문서 정확성을 높여 운영 혼선을 줄이는 보안 위생 개선이다. 하드코딩된 시크릿, 인젝션 취약점, 인증·인가 우회, 안전하지 않은 암호화 알고리즘 등 주요 보안 위협에 해당하는 변경은 없다.
