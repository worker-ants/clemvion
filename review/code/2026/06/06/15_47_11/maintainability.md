# 유지보수성(Maintainability) 리뷰

**대상 파일**: `spec/5-system/9-rag-search.md`
**변경 유형**: Spec 문서 갱신 (D1 동적 점수 컷 + conditional escalate 도입)

---

## 발견사항

### [INFO] `§3.4` 흐름 블록과 `§3.3.2` 흐름 블록 간 단계 번호 체계 중복
- **위치**: `spec/5-system/9-rag-search.md` §3.4 흐름 블록 (추가된 라인 116-124)
- **상세**: `§3.3.2` 흐름은 `1)~5)` 단계로 번호를 매긴다. `§3.4` 흐름도 동일하게 `1) wide 회수`, `2) 동적 점수 컷 a/b/c` 구조를 쓴다. 두 흐름 블록이 별도 섹션에 독립되어 있으나 단계 번호가 겹쳐, 독자가 "§3.4 step 2"를 참조할 때 어느 흐름인지 명확성이 낮아진다. 다른 spec 문서(`§2` 흐름도)는 단계 번호 없이 ASCII 화살표로 흐름을 표현하는 패턴을 사용한다.
- **제안**: §3.4 흐름 블록도 `§2` 스타일(화살표 다이어그램)로 통일하거나, 단계 레이블을 `R1/R2` vs `C1a/C1b/C1c` 처럼 접두어로 구분해 참조 명확성을 높인다. 또는 두 흐름 블록을 하나로 합쳐 `§3.3~§3.4 통합 흐름`으로 일원화한다.

---

### [INFO] `cutoffApplied` 설명의 "의미 확장(semantics expansion)" 인라인 경고 처리
- **위치**: §4.2 `ragDiagnostics` 주석 (추가된 라인 154, `cutoffApplied` 항목)
- **상세**: `cutoffApplied`는 기존에 "rerank 점수 컷이 하나라도 적용됐을 때 `true`" 였던 의미에서, D1 이후 token-budget 컷 / inject-cap 컷까지 포함하는 "어느 컷이든 적용 시 `true`"로 의미가 넓어졌다. 이 변경이 필드 설명 주석 한 줄 안에 `(의미 확장)` 으로만 표기되고 있어, 향후 클라이언트 코드나 진단 UI 코드를 유지보수하는 사람이 이전 동작과의 차이를 인지하기 어렵다.
- **제안**: 필드 설명에 "D1 이전: rerank 점수 컷만 포함. D1 이후: token-budget·inject-cap 컷도 포함" 처럼 변경 전/후를 명시하거나, `§Rationale` 또는 `§6` 에러 테이블에 컷 의미 변경 히스토리를 한 줄로 추가한다.

---

### [INFO] 동일 숫자 `50`의 두 개의 독립 상수 구분이 주석에만 의존
- **위치**: §3.4 상수 목록 (추가된 라인 126-129) 및 §3.1 파라미터 테이블 (수정된 라인 64)
- **상세**: `RAG_RECALL_K`(50)과 `rerank_candidate_k` 기본값(50)은 수치가 동일하지만 독립 코드패스임을 spec이 명시한다. 동일 값이라는 우연의 일치가 구현 시 "하나를 변경하면 다른 쪽도 바꿔야 하나?" 라는 혼란을 초래할 가능성이 있다. spec 문서는 이미 이 점을 각주와 상수 설명에서 짚고 있지만, 두 곳(§3.1 파라미터 테이블과 §3.4 상수 목록)에 분산 서술되어 있다.
- **제안**: 두 상수의 독립성 설명을 §3.4 상수 목록 한 곳으로 모으고, §3.1 파라미터 테이블에서는 `§3.4 참조`로 단방향 링크를 두는 방식으로 중복 설명을 줄인다.

---

### [INFO] `[^recall]` 각주와 본문 설명의 내용 중복
- **위치**: §3.1 파라미터 테이블 각주 `[^recall]` (추가된 라인 68) 및 바로 아래 `> 회수·컷 분기` 블록 (추가된 라인 70)
- **상세**: `[^recall]` 각주는 "$4 는 회수 폭이지 주입 청크 수가 아니다"와 "D1 이전의 고정 LIMIT topK 선차단 대체"를 서술한다. 이어지는 블록인 `> 회수·컷 분기` 는 각주 내용을 거의 그대로 재서술하면서 추가 분기 상세를 붙인다. 내용이 이중 서술되어 독자가 어느 것이 canonical인지 판단하기 어렵다.
- **제안**: `[^recall]`을 "회수 폭 정의 + §3.4 참조 링크" 한 문장으로 축소하고, 분기 상세는 `> 회수·컷 분기` 블록에 일원화한다.

---

### [INFO] §3.3.2 흐름과 §3.4 흐름의 단계 4)/5) 내용 재진술
- **위치**: §3.3.2 흐름 블록 단계 `4)` `5)` (수정된 라인 96-97) 와 §3.4 흐름 블록 `2)b`, `2)c` (추가된 라인 122-123)
- **상세**: §3.3.2 의 단계 4 (`§3.4 동적 점수 컷: rerank_score_threshold(θ) 미달 drop + token-budget 상한`) 와 단계 5 (`inject-cap`)는 §3.4 에서 동일 내용이 재서술된다. `≠off` 경로의 흐름을 §3.3.2 와 §3.4 양쪽에서 서술하는 구조라 수정 시 두 곳을 동시에 갱신해야 한다.
- **제안**: §3.3.2 의 단계 4/5를 "§3.4 동적 점수 컷 적용 → 상세는 §3.4 참조" 한 줄로 대체해 단일 진실을 §3.4에 둔다.

---

### [WARNING] §4.2 `rerank` 서브객체에 `gradingNoGrounding` 필드 추가는 있으나, §4.1 `ragSources` 와의 연결 설명 부재
- **위치**: §4.2 `gradingNoGrounding` 설명 (추가된 라인 153) 및 §3.3.2 '근거 없음' 전달 항목 (추가된 라인 105)
- **상세**: `gradingNoGrounding = true` 일 때 "KB tool 호출은 결과 메타에 '관련 근거 없음'을 명시해 agent 환각을 억제한다"고 서술하는데, 이 '관련 근거 없음' 명시가 실제로 어느 출력 필드(tool_result content? ragSources의 특수 항목? 별도 메타?)로 나타나는지 spec이 정의하지 않는다. 구현자가 이 신호를 어떻게 처리해야 하는지 불명확하다.
- **제안**: `gradingNoGrounding = true` 시의 tool_result 출력 포맷을 §2.2 KB tool 결과 포맷 섹션에 추가 예시로 명시한다 (예: `"noGrounding": true` 필드 또는 `"results": []`와의 차이 등).

## 요약

이번 변경은 §3.4 동적 점수 컷 신설, conditional escalate 도입, `gradingNoGrounding` 진단 필드 추가를 포함하는 실질적 spec 확장이다. 전반적으로 근거(Rationale) 서술이 충분하고 byte-identical 조항 폐기와 같은 기존 결정의 명시적 철회도 문서화되어 있어 변경 추적성은 양호하다. 다만 §3.3.2 흐름과 §3.4 흐름 간의 내용 중복, `[^recall]` 각주와 `> 회수·컷 분기` 블록의 이중 서술, `cutoffApplied` 의미 확장의 최소화 표기 등 몇 가지 정보 분산 패턴이 있다. 이들은 단일 진실 원칙을 spec 문서 내부에서도 적용해 참조 방향을 단방향으로 정리하면 해소된다. `gradingNoGrounding` 신호가 실제 tool_result 포맷에 어떻게 반영되는지에 대한 구현 명세 빈칸(WARNING)은 구현 단계에서 해석 오류로 이어질 수 있어 보완이 권장된다.
