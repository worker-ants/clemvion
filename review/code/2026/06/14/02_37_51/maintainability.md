### 발견사항

- **[INFO]** `validateTheme` 헬퍼가 `describe` 블록 최상위 스코프에 `const`로 선언되어 있음
  - 위치: `update-me.dto.spec.ts` 라인 42–45
  - 상세: `it` 블록 밖 스코프에 `async const` 헬퍼를 두는 패턴은 기존 `user.entity.spec.ts`의 `makeUser` 함수와 동일한 방식이라 내부 일관성은 있음. 다만 `it.each`와 일반 `it` 사이에서 헬퍼가 시각적으로 혼재되어 테스트 순서 파악 시 잠깐 흐름이 끊김. `beforeEach`로 분리하거나 파일 최상단(`describe` 외부)에 두는 관행과 차이가 있음. 실용적 문제는 없으나 큰 스펙 파일에서는 흐름을 끊는 요인.
  - 제안: 현 규모(30줄)에서는 허용 범주. 파일이 커질 경우 `describe` 블록 외부 또는 전용 `helpers` 유틸로 분리 고려.

- **[INFO]** `it.each` 인자 변수명이 `t`로 단축 축약됨
  - 위치: `update-me.dto.spec.ts` 라인 47
  - 상세: `it.each(['light', 'dark', 'system'])('theme=%s 는 검증 통과', async (t) => {` — 파라미터 `t`는 "theme" 의미를 전달하지 않음. 기존 코드베이스 spec 파일들은 이 패턴에 대한 선례가 없음(다른 spec 파일은 `it.each` 미사용).
  - 제안: `t` → `theme` 으로 rename.

- **[INFO]** 스펙 참조 번호가 `describe` 제목에만 표기되고 `it` 설명에는 없음
  - 위치: `update-me.dto.spec.ts` 전체
  - 상세: `describe('UpdateMeDto — theme (§2.0/§2.1 System 옵션)', ...)` 로 spec 근거를 제목에만 기재. 개별 `it` 케이스는 spec 근거 없이 동작 설명만 있음. 현재 파일 규모에서는 충분하나, 향후 케이스 추가 시 어느 spec 항목에서 비롯된 케이스인지 추적이 어려워짐.
  - 제안: 수용 가능한 현 상태. 케이스가 증가하면 `it` 설명에도 `(§2.1)` 등 간단 태그 추가 검토.

- **[INFO]** `update-me.dto.ts` 주석이 한 줄 이상의 인라인 설명을 포함하나 포맷이 `//` 단순 주석
  - 위치: `update-me.dto.ts` 라인 118–119 (새로 추가된 2줄)
  - 상세: 기존 `USER_LOCALES` 상수는 주석 없이 선언되어 있고, 새로 추가된 `USER_THEMES`만 2줄 주석을 달고 있음. 두 상수 간 문서화 깊이 불일치.
  - 제안: 불일치 자체는 경미. `USER_LOCALES`에도 짧은 inline 주석을 추가하거나, 반대로 `USER_THEMES` 주석을 JSDoc `/** */` 형식으로 통일하면 IDE hover 지원도 얻을 수 있음.

- **[INFO]** plan 문서 progress note가 블록 인용(`>`) 단락으로 삽입되어 이후 체크리스트와 시각적 분리가 일정하지 않음
  - 위치: `plan/in-progress/spec-sync-user-profile-gaps.md` 라인 213–215
  - 상세: 진척 note를 `> **구현 진척 ...**: ...` 인용 블록으로 앞에 두는 패턴은 가독성면에서 명확하고 기술적 문제는 없음. 다만 동일 plan 파일 내 `## 비고` 섹션과 역할이 일부 중복(진척 내용이 비고에도 반영될 수 있음).
  - 제안: 현 상태로 유지 가능. 진척 note는 시간 기반 스냅샷이므로 별도 섹션(`## 진척`)으로 분리하면 추후 업데이트 시 위치가 명확해짐.

### 요약

변경 범위가 작고(DTO 상수 1개 추가 + 30줄 테스트 신설 + 문서 갱신), 코드 구조는 기존 코드베이스 패턴과 대체로 일관적이다. `it.each` 파라미터 변수명 `t`가 의미 전달 측면에서 아쉬운 유일한 네이밍 문제이며, `USER_THEMES` 주석과 `USER_LOCALES` 무주석 간 문서화 깊이 불일치는 경미한 스타일 비일관성이다. 함수 길이, 중첩 깊이, 매직 넘버, 중복 코드, 순환 복잡도 측면에서 지적할 사항은 없다. 전반적으로 유지보수성이 양호하며 리팩터링 없이 진행 가능하다.
