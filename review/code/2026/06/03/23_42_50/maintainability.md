# 유지보수성(Maintainability) 리뷰 결과

## 발견사항

### 발견사항 1
- **[INFO]** 변경 대상이 전부 Markdown spec 문서의 인트라-문서 링크(앵커 `#fragment`) 수정이다.
  - 위치: 파일 1~24 전체 (spec/4-nodes, spec/5-system, spec/conventions, spec/7-channel-web-chat)
  - 상세: 모든 diff 는 코드가 아닌 Markdown 문서 내 `[텍스트](경로#앵커)` 형식의 앵커 참조를 실제 헤딩 슬러그와 일치하도록 수정한 패치다. 유지보수성의 전통적 지표(순환 복잡도·함수 길이·중첩 깊이 등)는 적용 대상이 아니다.
  - 제안: 해당 없음.

### 발견사항 2
- **[WARNING]** 앵커 링크가 수십 개의 문서에 걸쳐 수동으로 관리되어 있어, 헤딩 변경 시 링크 깨짐이 반복적으로 발생하는 구조적 취약점이 있다.
  - 위치: 파일 3~24 전체에 분포한 `#fragment` 링크들
  - 상세: 이번 패치 자체는 기존에 깨진 링크를 수정하는 올바른 작업이다. 그러나 이번 변경이 24개 파일 전체에 퍼져 있다는 사실은, 헤딩 텍스트를 변경하면 수십 개의 문서에서 수동으로 앵커 참조를 찾아 수정해야 하는 유지보수 부담을 보여준다. 이는 spec 문서 집합 전체의 구조적 문제이며, 이번 패치로 해소되지 않는다.
  - 제안: 중·장기적으로 `markdown-link-check` 또는 `remark-validate-links` 같은 링크 검증 도구를 CI에 추가해 깨진 앵커를 자동 감지하도록 한다. 현재 `.claude/` 레포 구조에서 이 가드가 없으면 동일 패턴의 수동 패치가 반복된다.

### 발견사항 3
- **[INFO]** 일부 앵커 슬러그가 매우 길고 특수문자(`--`, `.`, `(`, `)`)를 포함해 가독성이 낮다.
  - 위치: 예시 — `#44-사용자-입력-대기-이벤트-상세-executionwaiting_for_input`, `#421-보존-예외--_resumecheckpoint--_retrystate`, `#83-allowlist-mcpserversenabledtools`
  - 상세: 이번 패치가 수정한 앵커들은 헤딩 제목 자체가 길고 특수문자를 포함하기 때문에 슬러그가 복잡해진 것이다. 잘못된 슬러그는 링크 깨짐의 직접 원인이 되고, 긴 슬러그는 참조 시 오타를 유발한다. 이번 패치의 목적(깨진 링크 수정)에는 부합하지만, 원인인 헤딩 명명 방식의 구조적 개선은 이루어지지 않았다.
  - 제안: 섹션 헤딩에 짧고 안정적인 slug를 앵커 태그(`<a id="...">`)로 명시하거나, 헤딩 텍스트 자체를 간결하게 유지하는 컨벤션을 spec 작성 가이드에 추가하는 것을 검토한다.

### 발견사항 4
- **[INFO]** 파일 1(`10-parallel.md`)의 변경은 앵커 수정이 아니라 plan 파일 경로 참조 변경(`parallel-p2.md` → `parallel-p2-followups.md`)이다. 이는 plan 파일 rename 에 따른 정합 수정으로, 단일 진실 원칙을 지키는 올바른 패치이다.
  - 위치: `spec/4-nodes/1-logic/10-parallel.md` 35번째 라인
  - 상세: plan 파일 이름이 변경됐을 때 spec 참조가 갱신되지 않으면 dead link가 발생한다. 이번 패치가 이를 수정했다.
  - 제안: plan 파일 이름 변경 시 spec 역참조를 함께 갱신하는 체크리스트를 plan-lifecycle 문서에 명시하면 유사 누락을 예방할 수 있다.

### 발견사항 5
- **[INFO]** 파일 19(`5-system/_product-overview.md`)는 링크 수정이 아니라 "시스템 영역 spec 맵" 링크 목록을 새로 추가한 변경이다. 이는 문서 탐색성을 높이는 긍정적 유지보수성 개선이다.
  - 위치: `spec/5-system/_product-overview.md` 4~5번째 라인
  - 상세: 기존에는 `[제품 개요]` 하나뿐이던 관련 문서 섹션에 16개 시스템 spec 파일 링크가 추가됐다. 진입 문서로서의 역할이 강화되어 향후 작성자가 관련 문서를 찾는 비용이 줄어든다.
  - 제안: 해당 없음.

## 요약

이번 변경 세트(24개 파일)는 전부 Markdown spec 문서의 인트라-문서 앵커 링크 수정과 plan 파일 경로 참조 수정, 그리고 탐색 링크 목록 보강이다. 코드 변경이 없으므로 순환 복잡도·함수 길이·매직 넘버 같은 전통적 유지보수성 지표는 해당되지 않는다. 개별 패치는 깨진 링크를 올바른 슬러그로 교정한 정합 수정으로, 변경 자체의 방향성은 바르다. 다만 헤딩 텍스트 변경 한 번에 수십 개 문서의 앵커를 수동으로 쫓아가야 하는 구조는 이번 24-파일 패치가 반복될 수 있다는 유지보수 위험을 드러낸다. 중기적으로 CI 링크 검증 자동화를 추가하면 이 유형의 수동 패치 부담을 대폭 줄일 수 있다.
