## 발견사항

- **[WARNING]** `redactThreadForPublic` 이 마스킹하지 않는 두 개의 text-bearing 필드(`ConversationTurn.data`, `ConversationTurn.presentations[].payload`)가 동일한 공개 EIA 표면(SSE `waiting_for_input`/REST `getStatus`)으로 그대로 흘러간다 — 특히 `presentations`는 `toolCalls[].arguments`와 완전히 동일한 위협 모델(AI Agent `render_*` tool call 이 LLM 이 채운 구조화 페이로드)인데 masking 대상에서 빠졌다.
  - 위치: `codebase/backend/src/shared/conversation-thread/thread-renderer.ts:42-66` (`redactThreadForPublic`/`redactTurnForPublic`), 필드 정의는 `codebase/backend/src/shared/conversation-thread/conversation-thread.types.ts:138`(`data?`) · `:154`(`presentations?`) · `:79-97`(`PresentationPayload.payload: Record<string, unknown>`). 공개 wire 계약에 두 필드가 실제로 노출됨은 `codebase/channel-web-chat/src/lib/eia-types.ts:51,55`(`ConversationTurn.data`/`.presentations`) 로 확인.
  - 상세: `redactTurnForPublic`은 `turn.text`와 `turn.toolCalls[].arguments`만 마스킹한다. `toolCalls[].arguments`에 대해서는 코드 주석이 "Raw JSON-string tool arguments ride the same public wire; mask them too" 라고 명시적으로 근거를 대며 마스킹 범위에 포함시켰는데, `turn.presentations`(AI Agent의 `render_*` tool 호출이 채우는, 마찬가지로 LLM 이 만든 구조화 페이로드 — `conversation-thread.types.ts:146-153` 주석 참고)와 `turn.data`("핸들러가 보존한 payload")는 같은 논리가 적용돼야 함에도 빠졌다. `redactThreadForPublic`의 JSDoc 은 "masks secret-shaped tokens ... in every text-bearing field"라고 서술하는데(thread-renderer.ts:21-23), 실제로는 이 두 필드에 대해 그 주장이 거짓이다 — 함수 주석과 구현의 괴리(리뷰 관점 4). 만약 노드 핸들러/LLM 이 `render_table`/`render_carousel` 등의 payload 텍스트(예: 항목 설명)에 시크릿을 echo 하면, 이번 PR 이 닫으려 한 바로 그 위협(API 키·Bearer·Authorization 유출)이 이 경로로는 여전히 무방비로 통과한다.
  - 제안: (a) `redactTurnForPublic` 에 `turn.presentations[].payload`(재귀적으로 string leaf 마스킹 또는 `JSON.stringify`+`redactSecrets`+재파싱)와 필요 시 `turn.data`도 포함하도록 확장하거나, (b) 범위를 의도적으로 좁힌 것이라면 그 이유를 §R17 본문에 `outputData`/`nodeOutput` allowlist 항목과 나란히 "잔여 항목"으로 명시하고 JSDoc 의 "every text-bearing field" 문구를 정정한다. 현재처럼 침묵한 채 두면 후속 리뷰/감사에서 "닫힌 불변식"으로 오인되기 쉽다.

- **[WARNING]** spec 내 신규 참조 링크가 실제로 가리키는 문서와 불일치(dead/misdirected reference).
  - 위치: `spec/5-system/14-external-interaction-api.md` §R17, 새로 추가된 문장 — `` [`SECRET_LEAK_PATTERNS`](../conventions/conversation-thread.md) ``.
  - 상세: `SECRET_LEAK_PATTERNS`는 `codebase/backend/src/shared/utils/sanitize-error-message.ts`에 정의돼 있고, 링크 텍스트 직후 괄호에도 "에러 메시지 sanitizer 와 동일 SoT `shared/utils/sanitize-error-message.ts`"라고 정확히 명시한다. 그런데 마크다운 링크 자체는 `../conventions/conversation-thread.md`를 가리킨다 — 확인 결과 `spec/conventions/conversation-thread.md`에는 `SECRET_LEAK_PATTERNS`나 `sanitize-error-message` 언급이 전혀 없다(`grep` 0 hit). 동일 리포의 다른 spec 문서(예: `spec/5-system/15-chat-channel.md:700`)는 소스 파일을 가리킬 때 `../../codebase/backend/src/...` 상대경로로 직접 링크하는 관례가 있으므로, 이 링크도 그 패턴을 따랐어야 한다.
  - 제안: 링크 대상을 `../../codebase/backend/src/shared/utils/sanitize-error-message.ts`로 교정하거나(코드 심볼 직링크 관례 재사용), 단순 code span(백틱)으로 두고 링크를 제거한다.

## 요약

이번 PR은 EIA §R17 backlog 가 문자 그대로 지목한 "conversationThread turn 텍스트에 민감정보를 남기지 않는다"는 불변식을, REST(`getStatus`)·SSE(4개 waiting emit) 양 경로가 공유하는 단일 helper(`redactThreadForPublic`, SoT `SECRET_LEAK_PATTERNS`)로 정확히 런타임 강제하도록 닫았다. spec 갱신(§R17 "표면 제약(보안)", conversation-thread §8.4)은 실제 구현 범위(`turns[].text`·`toolCalls[].arguments`·`runningSummary`)와 정확히 일치하고, `outputData`/`nodeOutput` 키 allowlist 가 별개의 미구현 잔여 항목이라는 서술도 코드 사실과 부합해 overclaim 은 없다. 다만 (1) 같은 공개 wire 를 타면서 `toolCalls[].arguments`와 동일한 LLM-tool-call 출처를 가진 `ConversationTurn.presentations[].payload`(및 `data`)가 마스킹 대상에서 빠져 있어 "every text-bearing field" 라는 함수 주석의 주장과 실제 구현 사이에 괴리가 있고 — 같은 논리 확장이 자연스러운 인접 필드라 완전 종결로 보기엔 이르다 — (2) 신설된 §R17 텍스트의 `SECRET_LEAK_PATTERNS` 링크가 실제 정의 위치가 아닌 `conversation-thread.md`를 가리키는 오류가 있다. 핵심 청구(turn-text 마스킹 REST·SSE 일관 강제)는 코드·테스트로 확실히 검증됐고 회귀도 없다.
