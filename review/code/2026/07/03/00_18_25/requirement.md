### 발견사항

- **[CRITICAL]** `claimResumeEntry` 는 `NodeExecution.status` 만 원자 갱신 — spec 이 명시한 "짝 상태(Execution·NodeExecution) 단일 트랜잭션 갱신"과 불일치
  - 위치: `codebase/backend/src/modules/execution-engine/execution-engine.service.ts` L854-869 (`claimResumeEntry`), L1826-1830 (`driveResumeAwaited` 의 `updateExecutionStatus(savedExecution, ExecutionStatus.RUNNING)` 호출, `linkedNodeExec` 인자 없음)
  - 관련 spec: `spec/5-system/4-execution-engine.md` L81 "**재개 진입의 `waiting_for_input → running` claim(§7.5)도 이 원자성에 포함** — 조건부 UPDATE 가 짝 상태(Execution·NodeExecution)를 단일 트랜잭션으로 갱신" 및 L975 "claim 의 `waiting_for_input → running` 전이는 §1.1 원자성 보장에 따라 짝 상태(Execution ↔ NodeExecution)를 **단일 트랜잭션**으로 갱신한다"
  - 상세: `claimResumeEntry`(L859-868)는 `nodeExecutionRepository.createQueryBuilder().update(NodeExecution)...` 로 **NodeExecution 단독**만 조건부 UPDATE 한다. `Execution.status` 는 이 메서드 안에서 전혀 건드리지 않는다. `Execution.status → RUNNING` 전이는 이후 `rehydrateAndResume` → `driveResumeAwaited` 안에서 `await this.updateExecutionStatus(savedExecution, ExecutionStatus.RUNNING)` (L1830) 로 **별도로, 나중에** 수행되며, 이 호출에는 `linkedNodeExec` 인자가 없어 `updateExecutionStatus` 의 "NodeExecution 을 동일 트랜잭션에 묶는" 경로(L6660-6662 JSDoc 참조)를 타지 않는다. 즉 claim(NodeExecution WFI→running)과 Execution WFI→running 전이는 서로 다른 시점의 서로 다른 트랜잭션이다 — spec 이 명시한 "단일 트랜잭션으로 짝 상태를 갱신" 은 실제로는 성립하지 않는다. 이는 앞선 consistency-check(23_23_49 WARNING #2 "§1.1 원자성 보장 노트의 '전이 쌍' 서술과 claim UPDATE 도입의 정합성 미검증")가 정확히 예견한 갭이며, rev2 draft·최종 spec 텍스트는 "claim 이 두 상태를 단일 트랜잭션으로 갱신한다"고 **확정적으로 서술**해 이 WARNING 을 해소했다고 간주했지만 실제 구현은 그렇지 않다. 실무 영향: claim 성공(NodeExecution→RUNNING) 후 `driveResumeAwaited` 의 `updateExecutionStatus` 호출 전에 프로세스가 죽으면, NodeExecution 은 RUNNING 인데 Execution 은 여전히 WAITING_FOR_INPUT 인 상태 불일치 창이 실재한다 — `rehydrateAndResume` 상단의 `execution.status !== ExecutionStatus.WAITING_FOR_INPUT` 가드(L911-913)는 이 케이스에서 오히려 **정상 통과**하므로 회귀 자체를 막지는 못하지만(Execution 이 WFI 인 채로 재시도 가능), Execution↔NodeExecution 상태 쌍이 spec 이 선언한 불변식(§1.1 "원자성 보장")과 다르게 잠시 어긋나는 실제 창이 존재한다.
  - 제안: (a) 코드를 spec 대로 고치려면 `claimResumeEntry` 에서 `Execution.status` 도 같은 쿼리빌더 트랜잭션(`manager.transaction` 등)으로 함께 WFI→RUNNING 전이시키도록 구현을 확장하거나, (b) 코드가 의도된 설계(NodeExecution 만 원자 claim, Execution 전이는 이후 rehydration 단계에서 별도 처리 — claim 은 "재개 진입 gate" 역할만 하고 Execution 상태 전이는 기존 로직 그대로 유지)라면 spec L81/L975 문구에서 "짝 상태를 단일 트랜잭션으로 갱신" 서술을 정정해 "claim 은 NodeExecution 만 갱신하며, Execution 전이는 이후 `driveResumeAwaited`/`driveResumeFrame` 진입 시 기존 `updateExecutionStatus` 경로로 별도 수행된다"로 line-level 수정할 것. 이는 코드가 명백히 옳고 spec 만 낡은 SPEC-DRIFT 로 판단하기엔 실제 crash-window 불변식 손상 가능성이 있어(원자성 완화가 사용자가 승인한 "Option A" 설계 의도인지 불명확) 일반 WARNING/CRITICAL 로 사람 판단을 요청한다.

- **[WARNING]** `claimResumeEntry` 는 legacy 우회 시(`__no_node_exec__`/빈 id) `true` 반환하지만, 이 값이 `nodeExecutionId` 파라미터 자체의 유효성(빈 문자열)까지 관대하게 허용 — 실 운영에서 빈 id 가 발생 가능한지 spec 미명시
  - 위치: `execution-engine.service.ts` L855-858
  - 상세: `if (!nodeExecutionId || nodeExecutionId === '__no_node_exec__') return true;` 는 falsy 문자열(빈 문자열 `''`, `undefined` 캐스팅 등) 전부를 "claim 없이 통과"로 처리한다. 테스트(`execution-engine.service.spec.ts` L551-558)는 이를 "legacy 우회" 로 명시적으로 검증하나, `''` 가 실제 프로덕션 코드 경로에서 발생 가능한 값인지, 아니면 오직 `'__no_node_exec__'` sentinel 만 legacy 값이고 빈 문자열은 방어적 fallback 인지 spec/JSDoc 에 구분이 없다. cancel 류를 제외한 모든 continuation job 이 `nodeExecutionId` 를 필수로 지님을 고려하면 빈 문자열 케이스는 사실상 도달 불가능한 방어 코드일 가능성이 높지만, 그렇다면 "우회" 라는 표현보다 defensive-guard 임을 주석에 명시하는 편이 함수명(`claimResumeEntry`)의 "claim 을 시도한다" 라는 기대와의 괴리를 줄인다.
  - 제안: 주석에 `''` 케이스가 실제 도달 가능한지(어떤 job 이 빈 nodeExecutionId 를 갖는지) 1줄 근거 추가, 혹은 `__no_node_exec__` 하나로 sentinel 을 통일하고 빈 문자열은 별도 invariant 위반으로 처리할지 검토.

### 요약

핵심 로직(`isNodeExecutionWaiting` → `claimResumeEntry` 로의 전환, 비원자 SELECT 재검증을 조건부 UPDATE 로 대체, `markNodeExecutionFailed` 의 WAITING/RUNNING 이중 롤백 대상, processor 의 ack-and-discard 분기, `retry_last_turn` 제외)는 spec draft(`plan/in-progress/spec-draft-c2-atomic-claim.md` rev2)가 사용자 승인·consistency-check(BLOCK:NO) 를 거쳐 확정한 설계를 정확히 구현하고 있으며, 테스트도 affected=0/1 분기·markNodeExecutionFailed 회귀 가드를 적절히 커버한다. 다만 spec 이 명시적으로 확정한 "claim UPDATE 가 Execution·NodeExecution 짝 상태를 단일 트랜잭션으로 갱신한다"(L81, L975)는 서술은 실제 구현과 다르다 — `claimResumeEntry` 는 NodeExecution 단독만 갱신하고, `Execution.status → RUNNING` 전이는 이후 `driveResumeAwaited` 에서 `linkedNodeExec` 없이 별도 트랜잭션으로 수행된다. 이는 이전 consistency-check 세션이 WARNING 으로 예견했던 "전이 쌍 원자성 미검증" 갭이 spec 문구 갱신만으로 봉합되고 코드에는 반영되지 않은 채 남은 것으로 보이며, spec 이 선언한 원자성 불변식과 실제 crash-window 안전성 사이에 실질적 차이가 있어 CRITICAL 로 표시한다. 코드/spec 어느 쪽이 맞는지(설계 의도가 애초에 NodeExecution-only claim 이었는지)가 diff 만으로 확정되지 않아 SPEC-DRIFT 단정 대신 사람 판단을 요청하는 일반 CRITICAL 로 분류했다.
