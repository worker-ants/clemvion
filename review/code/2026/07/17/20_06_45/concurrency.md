# 동시성(Concurrency) 코드 리뷰

리뷰 대상: `.claude/tools/bootstrap-session.sh` (mermaid-lint 설치 가드) +
`.claude/tests/test_bootstrap_mermaid_install.py`. (`.gitignore`, `plan/in-progress/harness-guard-followups.md`

## 발견사항

### [WARNING] 락 탈취(steal) 판정이 "생존 여부"가 아니라 "경과 시간"만 검사 — 살아있지만 느린 설치의 락도 도둑맞아 동일 트리에 npm install 이 중복 실행됨 (재현으로 실증)

- 위치: `.claude/tools/bootstrap-session.sh:58-72` (60-61행 탈취 판정, 63행 `mkdir` 획득, 70행 무조건 `rmdir` 해제). 주석: 48-54행.
- 상세:
  - `[ -d "$lock" ] && [ -z "$(find "$lock" -maxdepth 0 -mmin -10 2>/dev/null)" ]` 은 락 디렉터리의 **mtime** 만 본다. 락은 `mkdir` 이후 다시 갱신(하트비트)되지 않으므로 이 조건은 "홀더가 죽었는가" 가 아니라 "마지막으로 락을 만든 지 10분이 지났는가" 만 검사한다. 실제로 살아서 `npm install` 을 계속 수행 중인 프로세스도 10분을 넘기면 다른 세션이 "stale" 로 오판해 `rmdir` + `mkdir` 로 재획득할 수 있다.
  - **재현으로 확인**: 5초 sleep 하는 "느리지만 살아있는" npm stub 으로 프로세스 A 를 띄워 락을 획득시킨 직후, 락의 mtime 을 인위적으로 11분 전으로 되돌렸다(A 는 여전히 설치 진행 중). 곧바로 프로세스 B 를 시작하자 B 는 stale 로 판정 → 탈취 → 재획득에 성공했고, **A(PID 21988)와 B(PID 22004) 의 `npm install` 이 완전히 겹치는 5초 구간 동안 동시에 실행**되어 둘 다 같은 `tool_dir/node_modules` 에 기록했다(양쪽 stdout 모두 "installing mermaid-lint deps"·"ready" 출력, npm 호출 로그의 start/end 타임스탬프가 두 PID 간 동일 구간으로 겹침). 이는 이 PR 이 없애려는 정확히 그 증상(동일 트리에 두 `npm install` 동시 실행)을 "동시 콜드스타트" 가 아닌 "느린 설치 + 시간 임계값" 경로로 재도입한다.
  - **연쇄 가능성**: 락 해제(70행)도 소유권 토큰 없이 경로 존재 여부만으로 무조건 수행된다. 위 재현에서 A·B 는 설치가 끝나면 자신이 만든 락인지와 무관하게 `$lock` 을 지운다 — A 종료 시의 `rmdir` 은 실제로는 B 가 재획득한 락을 지우는 것일 수 있다(역도 가능). B 가 여전히 설치 중인데 락이 사라지면 세 번째 세션이 그 틈을 또 파고들 수 있어, 탈취가 한 번으로 그치지 않고 연쇄될 가능성이 코드 로직상 존재한다(직접 재현은 A/B 2단계까지만 수행).
  - **주석과의 괴리**: 스크립트 주석(48-54행)은 "A `mkdir` LOCK ... so two sessions never npm-install into the same tree concurrently" 라고 **무조건적으로** 서술하지만, 실제 보장은 "홀더가 죽었을 때" 로 한정되고 "홀더가 살아있지만 느릴 때" 는 이 불변식이 깨진다 — 서술이 구현이 실제로 보장하는 범위보다 강하다.
  - 완화 요인: `mermaid-lint` 는 소규모 단일 devDependency 라 정상 조건에서 설치는 수 초 내 끝나 10분 문턱에 도달할 일이 드물고, 저자도 "크래시한 홀더" 케이스에 한해 이 트레이드오프를 의도적으로 선택했다고 주석에 명시했다. 다만 이 하네스가 네트워크 제약이 있는 샌드박스에서도 도는 점을 고려하면 "느리지만 살아있는 설치" 가 순전히 이론상의 걱정만은 아니며, 무엇보다 위에서 실제로 재현됐다.
- 제안: 락 디렉터리 안에 소유자 식별자(예: `echo $$ > "$lock/owner"`)를 기록하고, ① 탈취 전에는 mtime 대신(또는 더해) 그 PID 가 실제로 살아있는지(`kill -0 "$pid" 2>/dev/null` — 로컬 프로세스이므로 유효) 확인하고, ② 해제 시에는 기록된 소유자가 자신과 일치할 때만 지우도록 해 "내가 만들지 않은 락은 절대 건드리지 않는다" 를 보장할 것. 최소한 이 잔여 위험(느리지만-생존-홀더 케이스)을 스크립트 주석과 `plan/in-progress/harness-guard-followups.md` A 항목에 알려진 한계로 명시할 것 — 현재는 "크래시 홀더" 케이스만 언급하고 이 케이스는 언급이 없다.

### [INFO] 콜드스타트 동시성(주 실패 모드)은 `mkdir` 원자성만으로 올바르게 직렬화됨 — 신·구 코드 대조 실행으로 검증

- 위치: 같은 블록. 테스트: `.claude/tests/test_bootstrap_mermaid_install.py`
- 상세: 여러 세션이 **같은 순간** stale 탈취를 시도하는 경우(홀더가 진짜 죽어 다수가 동시에 stale 로 판정하는 경우 포함)에도, 실제 상호배제는 오직 `mkdir` 의 원자적 EEXIST 실패에만 의존하므로 안전하다 — 여러 프로세스가 동시에 `rmdir`(탈취)을 시도해도 그 다음 `mkdir` 은 정확히 하나만 성공한다. 직접 검증:
  - 현재 스크립트에 대해 9개 테스트 전부 통과 (`python3 .claude/tests/test_bootstrap_mermaid_install.py -v` → `OK`, 9 tests).
  - 같은 테스트를 커밋 `cdad5a1ec`(이 수정 직전) 의 구 스크립트에 대입해 실행하면 **6/9 실패**, 특히 `test_concurrent_sessions_install_at_most_once`(5개 동시 프로세스) 는 `5 not less than or equal to 1` 로 실패 — 구 코드에서는 5개 세션이 **직렬화 전혀 없이** 전부 `npm install` 을 호출했다. plan 문서가 주장하는 "구 코드 6건 실패" 가 그대로 재현되어 테스트 스위트가 비-vacuous 함을 확인했다.
- 데드락 가능성: 단일 락만 사용하고(락 순서 문제 없음) 패자는 대기하지 않고 즉시 skip 하므로 데드락 여지가 없다.
- 원자성: 마커 파일 생성(`: > "$marker"`, 단일 open(O_CREAT|O_TRUNC))과 락 생성(`mkdir`)은 각각 그 자체로 원자적이며, 마커는 오직 락을 획득한 단일 승자만 쓰므로(위 WARNING 의 탈취-경합 상황 제외) 별도 동기화가 불필요하다.

## 요약

이번 변경은 여러 worktree 세션이 `SessionStart` 시 공유 `main_root` 의 `mermaid-lint` `node_modules` 에 동시 `npm install` 하던 실제 경쟁 조건을, 완료 마커(원자적 파일 생성) + `mkdir` 기반 상호배제 락으로 고친다. "동시 콜드스타트" 라는 주된(이 저장소의 정석 워크플로 특성상 가장 흔한) 실패 모드는 `mkdir` 원자성에 정확히 의존해 올바르게 해소됨을 신·구 코드 대조 실행(구 코드 6/9 실패, 특히 동시성 테스트는 5개 세션 전원이 직렬화 없이 npm 호출 → 신 코드 9/9 통과)으로 직접 확인했다. 다만 크래시한 홀더의 락을 회수하기 위한 "10분 경과" 탈취 조건은 생존 여부가 아니라 순수 경과 시간만 검사하므로, 실제로 살아서 10분을 넘기는(예: 네트워크 지연) 설치의 락도 탈취될 수 있고, 그 결과 두 `npm install` 이 같은 트리에 동시 실행되는 — 이 PR 이 없애려던 바로 그 — 증상이 더 좁은 창(≥10분 지속 + 세션 시작 겹침)을 통해 재도입될 수 있음을 재현으로 확인했다. 락 해제도 소유권 검증 없이 무조건 수행되어 탈취 후 연쇄 가능성도 있다. 이 잔여 위험은 소규모 devDependency 설치라는 낮은 blast radius 와 "정상적으로는 수 초 내 종료" 라는 낮은 발생 확률로 완화되고 저자도 크래시-홀더 케이스에 한해 트레이드오프를 의도적으로 받아들였다고 명시했지만, 스크립트 주석의 "두 세션은 결코 동시에 설치하지 않는다" 는 무조건적 서술은 실제 보장 범위보다 강하며 PID/토큰 기반 소유권 확인으로 강화할 여지가 있다.
