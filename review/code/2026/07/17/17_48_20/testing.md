## 검증 2 — 회귀 테스트 2건의 기대값 변경: **둘 다 정당함, 직접 실측**

- **"겹친 부팅의 결과가 갈릴 때…" (`:2275`)**: 검증1의 mutation-1(boot 축 무력화) 실행 결과, 실패 지점은
  스택트레이스상 정확히 `:2336` — 바뀐 전제 단언 `expect(phase).not.toBe("blocked")` 이다. 최종 단언
  (`hookPosts`)까지 도달하지 못하고 그 앞에서 실패한다. "최종 단언이 아니라 전제가 깨졌다"는 서술과
  정확히 일치.
- **"겹친 부팅에서 나중 진입이 차단으로 먼저…" (`:2359`)**: 이 테스트만을 대상으로 **"소실" mutation** 을
  직접 설계해 실측했다 — 1차의 첫 재검증 지점에서 stale 판정 시 `pendingResetRef.current = false` 를
  같이 실행하도록 프로덕션 코드를 바꿔(리셋을 이월이 아니라 실제로 폐기하는 버그) 두 버전을 대조:
  - **현재 테스트(후속 boot + `hookPosts===1` 단언 포함)**: 정확히 `hookPosts===1` 단언(파일의 새 줄
    번호로 `:2440` 상당)에서 `expected +0 to be 1` 로 실패 — **소실을 잡아낸다.**
  - **후속 부분만 제거한 실험판(= `expect(hookPosts).toBe(0);` 으로 테스트 종료)**: 동일한 소실
    mutation 을 적용해도 **PASS** — `hookPosts===0` 는 "이월(아직 미소비)"과 "소실"을 구분하지 못한다.
  결론: "이 단언이 없으면 `toBe(0)` 은 소실과 구분되지 않는다"는 주장은 **정확하고, 실제로
  load-bearing 이다.** 사용자가 요청한 정확한 실험("후속 단언을 빼면 소실 mutation 이 통과하는지")을
  그대로 수행해 확인했다.

## 검증 5 — 남은 사각지대

### 5-1. (WARNING) `!cfg.apiBase` 조기 return 이 세대를 올리지 않는다는 불변식 — 무테스트

`beginBootAttempt` JSDoc(`use-widget.ts:261` 부근)이 "조기 return 은 세대를 올리지 않는다 — 올리면
죽은 대체자가 살아있는 시도를 밀어낸다"를 명시적으로 문서화하지만, 파일 전체에서 `apiBase`/
`triggerEndpointPath` 가 비거나 undefined 인 `wc:boot` 페이로드를 보내는 테스트가 **하나도 없다**
(`grep` 확인 — `boot()` 헬퍼가 항상 유효한 payload 만 보냄). `bootGenRef` 자체가 이번 PR 신설이므로
"기형 payload 가 산 시도를 밀어낸다"는 회귀 클래스는 이 PR 이전엔 존재조차 안 했던 새 위험이다.
- 제안: 유효한 1차 boot 이 in-flight 인 동안 `apiBase` 누락 payload 를 보내고, 그 뒤 1차가 정상
  resolve 되면 config 가 적용되는지(밀려나지 않았는지)를 확인하는 회귀 테스트 1건 추가.

### 5-2. (CRITICAL, 단 리뷰 시점에 이미 remediation 진행 중) 리듀서 `ended` 가드는 화면만 지킬 뿐 부작용(재조회·SSE 재오픈)은 못 막는다 — 독립 재현으로 확인

이 리뷰를 진행하는 도중 `MAIN` 워크트리(`webchat-boot-single-flight-8c92b4`)에 **prompt_file 의 diff
에는 없는 미커밋 변경**이 존재함을 발견했다(`git status`) — 즉 **리뷰 대상 스냅샷이 만들어진 뒤에도
같은 세션에서 코드가 계속 바뀌고 있다.** 그 미커밋 diff 를 읽기 전용으로 확인한 결과:

- `seedWaitingFromStatus` 가 이제 boot-시도 토큰을 받아, **대체된 시도가 종료를 확정하면 살아있는
  마지막 시도까지 world 무효화로 덮어쓰는** 별개의 CRITICAL(`ai-review 2026-07-17 17_36_57 concurrency`
  라고 주석에 명시)을 고치고 있었다 — 이는 이번 라운드 이전에 존재했던 **다른** 리뷰 라운드의 산출물이다.

  대응하는 A-6 통합 테스트의 **전제 자체가 뒤집혀 있었다**(`toContain("e1")` → `toBeNull()`, 새 주석:

이것이 실제 결함인지 **내가 리뷰한 커밋(`215cd1c3f`, prompt_file 의 diff 그 자체)에서 직접, 독립적으로
재현**했다 — 별도 격리 worktree 에서 기존 A-6 통합 테스트에 부작용 계측(`statusCalls` 카운터,

statusCallsBefore: 1   statusCallsAfter: 2       (재부팅 후 GET status 재조회가 실제로 1회 더 일어남)

즉 **커밋 `215cd1c3f`(이번 라운드가 검증 대상으로 받은 바로 그 코드) 는 `phase` 는 "ended" 로 정확히

### 5-3. (INFO) 검증했지만 문제 없음으로 판명 — "world 축이 둘째 체크포인트에서 무방비 아닌가" 가설

자신이 내부적으로 `gen` 을 캡처해 독립적으로 world-staleness 를 검사하므로(`outcome !== "continue"`
로 먼저 걸러짐), 둘째 체크포인트의 world 성분은 **논리적으로 항상 그 이전 게이트에 가려져** 독자적
탐지 지점이 될 수 없다 — boot 성분만 둘째 체크포인트의 고유한 가치이고, 신규 테스트는 정확히 그것만
겨냥한다. 사각지대 아님, 오탐 기각.

### 5-4. (WARNING) 리뷰 payload 완결성 — 테스트 파일 diff 가 프롬프트에서 생략됨

`_prompts/testing.md` 파일 15(`use-widget-eager-start.test.ts`, 이번 diff 의 핵심 테스트 파일)와
파일 22(plan 문서)의 diff 가 "prompt size limit" 으로 생략돼 있었다. 이 리뷰는 `git diff
14bc86a53..HEAD` 로 직접 재취득해 검증을 완료했지만, **테스트 리뷰어가 정작 테스트 diff 를 못 보는
상황은 이 리뷰 라운드의 다른 관점 리뷰어(들)에게도 동일하게 적용됐을 가능성**이 있다 — 특히 테스트
커버리지 판단은 diff 없이는 원천적으로 불가능하다. orchestrator 쪽에서 큰 diff 발생 시 (a) 분할
전달하거나 (b) 최소한 "생략됨" 을 명확히 표시해 리뷰어가 직접 fetch 하도록 유도하는 처리가 이미
있었던 것으로 보이나(실제로 생략 표시는 있었음), 리뷰어별로 그 후속 조치(직접 diff 재취득)를 하지
않으면 판단이 비어버린다는 점은 프로세스 리스크로 남는다.

## 요약

검증 요청 5건 중 1~4번은 보고된 수치·주장이 전부 독립 재현으로 확인됐다 — mutation 매트릭스 6개
전원 정확히 일치, 회귀 테스트 2건의 기대값 변경은 각각 다른 이유(전제 파손 vs 의미 변경)로 정당했고
특히 "후속 단언이 없으면 소실 mutation 이 통과한다"는 주장은 직접 설계한 대조 실험으로 실증했다.
A-6 가드의 실패 카운트(2/1)와 "BOOTED 오늘 도달 불가" 주장도 코드 경로 추적으로 확인했다. 신규 6건
에서 이 파일의 3연속 "전제 미고정" 패턴 재발은 발견하지 못했다 — 이번 라운드의 테스트 작성 규율은
견고하다. 다만 이 과정에서 두 가지를 추가로 발견했다: (1) `!cfg.apiBase` 조기 return 이 세대를 안
올린다는 이번 PR 신설 불변식이 무테스트 상태이고(WARNING), (2) 더 중요하게, **리뷰 대상 커밋 자체에서
직접 재현**한 것으로 — A-6 의 리듀서 `ended` 가드는 화면(`phase`)만 지킬 뿐 그 이면의 부작용(종료된
대화에 대한 상태 재조회 재발생·새 SSE 연결 개설)은 막지 못하는 것을 실측 확인했다(CRITICAL급 결함).
다행히 이 문제는 리뷰 진행 중 발견한 `MAIN` 워크트리의 **미커밋 작업**에서 이미 근본 수정 중이었다
(`sendCommand` ERROR 분기에 `teardownSession()` 추가 + 새 회귀 테스트의 `statusCalls`/`getEs()` 동결
단언) — 그러나 이는 이번 라운드가 검증 대상으로 받은 diff·plan"A-6 완료" 선언 시점에는 반영되지
않았던 사실이므로, 이번 라운드의 "A-6 완료"는 그 자체로는 최종 상태가 아니다. 전체적으로 developer 의
자체 mutation 검증 방법론과 정직성은 매우 신뢰할 만하지만(모든 수치가 재현 가능했고 과장이 없었다),
검증 스코프가 "phase 단언"에 머물러 부작용 레벨의 회귀를 그 스코프 안에서는 잡지 못했다는 구조적
한계가 실측으로 드러났다.
