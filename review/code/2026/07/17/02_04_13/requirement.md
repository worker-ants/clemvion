### 발견사항

- **[WARNING]** `applyConfig` 세션 복원 경로가 `seedWaitingFromStatus` 의 신규 terminal 단락(short-circuit)에 대해 무방비 — `openStream`/`scheduleRefresh` 가 teardown 이후에도 무조건 실행됨 (실측으로 재현 확인)
  - 위치: `codebase/channel-web-chat/src/widget/use-widget.ts:563-573` (`applyConfig` 의 저장 세션 복원 분기: `if (clientRef.current) await seedWaitingFromStatus(clientRef.current, saved); openStream(saved, "0"); scheduleRefresh();`). 대비 보호가 있는 `start()` 호출부: `use-widget.ts:309-346` 의 `startGenRef` gen-check(`if (startGenRef.current !== gen) return;`).
  - 상세: 이번 fix(`seedWaitingFromStatus` 내부 terminal 분기, `use-widget.ts:249-255`)는 `status.status` 가 terminal 이면 `teardownSession()`(→ `startGenRef.current++`, `closeStream()`, `clearRefreshTimer()`, `clearSession(...)`) + `dispatch({type:"ENDED"})` + `bridgeRef.current?.sendEvent("conversationEnded", ...)` 를 수행하고 `return` 한다. `start()` 호출부는 이 반환 직후 `if (startGenRef.current !== gen) return;` 로 `teardownSession` 이 증가시킨 gen 을 감지해 `openStream`/`scheduleRefresh` 를 스킵한다(정상 보호). 그러나 **세션 복원 호출부(`applyConfig`, 새로고침 후 저장 세션 로드)는 이런 gen-check 가 전혀 없다** — `await seedWaitingFromStatus(...)` 가 내부적으로 이미 teardown+ENDED 를 마쳤어도 바로 다음 줄에서 **무조건** `openStream(saved, "0")` 와 `scheduleRefresh()` 를 실행한다. `sessionRef.current` 는 `teardownSession()` 이 null 화하지 않으므로 `openStream` 은 방금 `clearSession` 으로 스토리지에서 지운 `saved`(만료·invalidate 된 per_execution 토큰, EIA-AU-04: "execution 종료 시 즉시 invalidate")로 실제 `EventSource` 를 새로 연다.
    - **재현 확인**(임시 테스트, 리뷰 종료 후 삭제): 저장 세션을 미리 `sessionStorage` 에 두고 `getStatus` 가 `status:"completed"` 를 반환하도록 mock 한 뒤 `boot()` 만 호출(사용자 조작 없음) → `result.current.state.phase` 는 올바르게 `"ended"` 로 전이되고 `sessionStorage` 항목도 `null` 로 정리됐지만, **`getEs() !== null` (즉 `EventSource` 생성 = `openStream` 실행) 이 `true`** 로 확인됨. 즉 위젯은 화면상 `[ended]` 를 정확히 보여주면서 동시에 이미 무효화된 토큰으로 SSE 재연결을 계속 시도하는 상태로 남는다 — native `EventSource` 는 실패 시 자동 재연결하므로 이 연결은 사용자가 "새 대화"를 누르거나(→ `newChat()` 이 `closeStream` 재호출) 위젯이 unmount 될 때까지 백그라운드에서 계속 실패-재시도를 반복한다. `scheduleRefresh()` 도 이미 끝난 세션의 `expiresAt` 기준으로 갱신 타이머를 예약해 나중에 실패하는 `refreshToken` 호출을 유발한다(soft-fail 이라 크래시는 없음).
    - 이는 이번 fix 의 JSDoc("**종료 상태 처리**: status 가 terminal 이면 표면 시드 대신 세션 정리 + ENDED 전이 + host 통지를 수행한다", `use-widget.ts:234-235`)과 커밋 메시지/`review/code/2026/07/17/01_42_44/RESOLUTION.md` 의 명시적 주장("세 호출부(start·restore·replay 폴백) 모두가 이 보호를 받는다")과 실제로 어긋난다 — terminal *감지*(teardown+ENDED)는 세 호출부 모두에서 실행되지만, 그 감지 이후 **호출부 자신의 후속 코드**를 가드하는 것은 `start()` 뿐이고 `applyConfig` 복원 경로는 가드가 없다. `replay_unavailable` 호출부는 그 자체가 유일한 문장이라 우연히 안전하다.
    - 신규 회귀 테스트(`use-widget-eager-start.test.ts` 의 두 신규 `it`)는 모두 `replay_unavailable` 트리거 경로만 검증하고, 세션 복원(`applyConfig`) 경로에서 terminal 상태를 만나는 케이스는 커버하지 않아 이 갭이 테스트로 드러나지 않았다.
  - 제안: `start()` 와 대칭으로 복원 경로도 가드할 것. 가장 간단한 방법은 `seedWaitingFromStatus` 가 "terminal 로 이미 처리했음"을 boolean 으로 반환하게 하고, 세 호출부(`start`/`applyConfig`/필요 시 향후 추가 호출부) 모두 그 반환값을 확인해 `true` 면 `openStream`/`scheduleRefresh` 를 스킵하도록 통일. 또는 `teardownSession()` 이 `sessionRef.current = null` 을 함께 수행하고 각 호출부가 `if (!sessionRef.current) return;` 로 방어. 회귀 테스트로 "저장 세션 복원 시 execution 이 이미 terminal → `openStream` 미호출(EventSource 미생성) + phase ended" 케이스 추가 권장.

- **[SPEC-DRIFT]** `spec/7-channel-web-chat/1-widget-app.md §3.1` 이 이번 fix 로 도입된 terminal 단락 의미를 반영하지 않음 — "스트림·세션은 유지" 서술이 이제 예외가 있다
  - 위치: `spec/7-channel-web-chat/1-widget-app.md:104-107` (`EIA 의 버퍼 만료 신호(execution.replay_unavailable)는 ... 위젯은 이 이벤트를 받으면 getStatus snapshot(§5.3)으로 폴백해 현재 표면을 재동기화한다 ... **종료 신호가 아니므로 스트림·세션은 유지**되며 이후 도착하는 이벤트는 정상 처리된다.`). 동일 서술이 `plan/in-progress/spec-sync-external-interaction-api-gaps.md:20` ("종료 신호가 아니므로 스트림·세션은 유지한다")에도 있음.
  - 상세: 이번 fix(`use-widget.ts:244-255`) 는 `seedWaitingFromStatus` 가 반환한 `getStatus` 스냅샷 자체가 이미 terminal(`completed`/`failed`/`cancelled`)이면 — 즉 5분 버퍼 gap 안에 execution 이 종료돼 그 terminal SSE 이벤트가 함께 유실된 경우 — **세션을 정리하고 `[ended]` 로 전이**하도록 의도적으로 동작을 확장했다(정확한 버그 fix, `review/code/2026/07/17/01_42_44/RESOLUTION.md` W-req 근거로 뒷받침됨). 그러나 spec §3.1 은 여전히 "종료 신호가 아니므로 스트림·세션은 유지된다"고 **무조건** 서술한다 — 이 fix commit(`e99f46145`)은 `codebase/**` 와 테스트만 바꿨고 `spec/**`/`plan/**` 는 건드리지 않았다(`git show e99f46145 --stat` 확인). 코드가 옳고(실제 버그를 고쳤고 회귀 테스트로 고정됨) spec 문구가 뒤따라가지 못한 전형적 spec drift.

### 요약
`seedWaitingFromStatus` 에 terminal 분기를 추가해 "5분 버퍼 gap 안에 execution 이 이미 종료됐는데 위젯이 streaming 스피너에 무기한 멈추는" 이전 라운드의 실질 버그(W-req)는 `replay_unavailable` 트리거 경로와 `start()` 경로에서는 올바르게 고쳐졌고 회귀 테스트·mutation 검증도 확인된다. 그러나 **동일한 terminal 단락을 쓰는 세 번째 호출부인 새로고침 세션 복원 경로(`applyConfig`)는 `start()` 와 달리 gen-check 가드가 없어**, 복원된 세션이 이미 terminal 이면 위젯이 화면상으로는 정확히 `[ended]` 를 보여주면서도 이미 무효화된 토큰으로 `openStream`/`scheduleRefresh` 를 실행해 백그라운드에서 실패-재연결을 반복하는 잔존 SSE·타이머를 남긴다(실측 재현 확인). 이는 이 fix 자체의 JSDoc·커밋 메시지·`RESOLUTION.md` 가 명시한 "세 호출부 모두 보호받는다"는 주장과 다르며, 신규 회귀 테스트도 이 경로를 커버하지 않는다. 별도로 `spec/7-channel-web-chat/1-widget-app.md §3.1`(과 대응 plan 항목)은 "종료 신호가 아니므로 스트림·세션은 유지"라는 무조건 서술을 그대로 두고 있어, 이번에 추가된 terminal 예외를 spec 이 아직 반영하지 못한 spec-drift 가 존재한다(코드는 정당한 의도적 확장). webauthn 관련 변경(테스트 신설, DTO 주석)은 실제 컨트롤러 구현·spec 과 line-level 로 정확히 일치하며 문제 없다.
