# 부작용(Side Effect) 리뷰

대상: AI 노드 실패 시 대화 이력 렌더 도달성 복구 (plan/in-progress/ai-node-failed-conversation-preview.md) — frontend 4개 소스 파일 변경 + 테스트/fixture.

전역 변수·환경 변수·파일시스템·네트워크 호출 관점에서는 이슈가 없다 (순수 프론트 렌더/스토어 로직 변경). 아래는 "선언된 스코프(AI 대화형 노드의 실패 종결 미리보기 복구)"보다 실제 파급이 넓은 지점과, 상태(status) 게이트 제거가 여는 새 경로를 중심으로 정리한다.

## 발견사항

- **[WARNING]** `use-execution-events.ts` `handleNodeFailed` 의 `outputData: payload.output ?? null` 은 AI 대화형 노드에 국한되지 않고, error-port 로 라우팅되는 **비AI 노드 8종(Integration·LLM·Code·Workflow)** 의 실패 결과에도 동일하게 적용된다
  - 위치: `codebase/frontend/src/lib/websocket/use-execution-events.ts:871` (변경 지점), `codebase/backend/src/modules/execution-engine/execution-engine.service.ts:5628-5642`(`finalizeErrorPortNode` 호출부 주석 — "Integration·LLM·Code·Workflow 노드 8종은 throw 대신 이 경로로 실패를 surface"), `execution-engine.service.ts:5874-5920`(`finalizeErrorPortNode` — `nodeExecution.outputData = rawOutput` 후 `NODE_FAILED` payload 에 `output: nodeExecution.outputData` 동봉. AI 전용이 아니라 error-port 라우팅 전체에 공통 적용되는 finalizer)
  - 상세: `node.completed`(`use-execution-events.ts:778`)와의 "비대칭 해소"라는 의도 자체는 타당하지만, 실제 효과는 AI 대화형 노드보다 훨씬 넓다. `result-detail.tsx` 의 여러 소비처가 지금까지 "실패 노드는 `outputData === null`" 을 암묵 전제로 동작해 왔다:
    - `showPreview = hasPreview ?? (isPresentation && !!result.outputData)` (`result-detail.tsx:212`) — presentation 카테고리의 실패 노드(예: template/carousel)가 `output` 을 동봉하고 error-port 로 실패하면 이제 Preview 탭이 새로 나타날 수 있다.
    - `hasMeta`/`hasPort`/`hasStatus` (`result-detail.tsx:232-234`, `unwrapNodeOutput(result.outputData)` 기반) — 실패한 Integration/LLM/Code/Workflow 노드의 Meta/Port/Status 탭이 이전엔 항상 숨겨졌으나 이제 조건부로 노출된다.
    - `extractBackgroundRunId(result.outputData)` (`result-detail.tsx:1237`), `isDryRunOutput(result.outputData)` (`result-detail.tsx:1194`) 등도 동일하게 영향받는다.
    - 데이터 자체는 이미 WS payload 로 전송되던 값이라 **네트워크 노출은 신규가 아니다**(기존 backend 동작). 다만 클라이언트가 지금까지 조용히 버리던 값을 이제 store 에 영속·렌더링한다는 점에서 **가시성 확장**은 신규다.
  - 제안: 이 확장이 의도된 것이라면(개연성 있음 — `node.completed` 와의 대칭이 원래 목표에 가까움) plan/spec 에 "AI 대화형 노드 한정" 이 아니라 "error-port 라우팅 노드 전반" 으로 스코프를 명시하고, 비AI 노드(예: http_request 또는 code 노드)가 실제 `output` 을 동반하고 실패하는 케이스에 대해 Preview/Meta/Port/Status 탭 노출을 검증하는 회귀 테스트를 추가로 확보할 것. 현재 `result-detail.test.tsx` 의 "비대화형 노드는 기존대로 오류 탭 기본" 테스트는 `outputData: null` 을 명시적으로 넣어 확인하므로, 실제 `output` 이 채워진 비AI 실패 케이스는 무테스트 상태다.

- **[WARNING]** `isCompletedConversation → isConversationHistory` 의 `status === 'completed'` 게이트 제거가, "cancelled" 상태의 노드에 대해 오래된(stale) 대화 스냅샷을 미리보기로 노출할 수 있는 경로를 새로 연다
  - 위치: `codebase/frontend/src/components/editor/run-results/result-detail.tsx:1046-1053`(`isConversationHistory = isConversationOutput(result.outputData)`, status 무관), `codebase/frontend/src/components/editor/run-results/output-shape.ts:838-839`(`isCanonicalWaiting = unwrapped.status === "waiting_for_input" && hasLegacyMessages` — `outputData` 내부 중첩 `status` 필드만 보고 `result.status` 는 보지 않음), `codebase/frontend/src/lib/websocket/apply-execution-snapshot.ts:91-104`(모든 NodeExecution row 에 대해 `outputData: ne.outputData` 를 상태 무관 반영), `apply-execution-snapshot.ts:393-402`(`isNodeWaitingForInput` — terminal 상태(`cancelled` 포함)는 의도적으로 stale envelope 신호를 채택하지 않음, 즉 `incomingStatus` 는 그대로 `cancelled` 로 남음), `codebase/backend/src/modules/execution-engine/execution-engine.service.ts:910-935`(`cancelParkedExecution` — WAITING_FOR_INPUT → CANCELLED 전이 시 `UPDATE ... SET status = CANCELLED` 만 수행하고 `outputData` 컬럼은 건드리지 않음)
  - 상세: 재현 체인 — ① 노드가 `waiting_for_input` 상태로 대기하며 `outputData` 에 `{config, output:{messages,...}, status:'waiting_for_input', meta:{interactionType:'ai_conversation'}}` 형태의 구조화 envelope 를 영속. ② 사용자가 실행을 취소하면 (해당 취소가 park-release 경로를 타는 경우) backend 는 `status` 컬럼만 `CANCELLED` 로 바꾸고 `outputData` 는 그대로 남긴다. ③ 새로고침 후 REST 스냅샷을 로드하면 `NodeResult.status = 'cancelled'` 이지만 `outputData` 는 여전히 대기 당시의 conversation-shape 값이다. ④ `isConversationOutput` 은 `outputData` 내부 중첩 `status` 만 보고 `isCanonicalWaiting = true` 를 반환하므로, `isConversationHistory` 가 `true` 가 되어 (구 코드였다면 `status === 'completed'` 요구로 인해 반드시 `false` 였을 자리에서) `isConversationNode`/`showTabs` 가 `true` 로 바뀌고, `parseHistoryMessages(result.outputData)` 가 취소 이전의 부분 대화를 그대로 미리보기에 렌더링한다. 이때 `wrapper.error` 가 없으므로 `system_error`/취소 마커가 합성되지 않아, 사용자는 실제로는 중단된 대화를 마치 정상 대화처럼 보게 될 수 있다.
  - 참고(불확실성 고지): 위 `cancelParkedExecution` 은 docblock 상 "form/button top-level park 전용" 으로 명시돼 있고, 멀티턴 AI in-memory 코루틴 취소는 `applyCancellation` 의 별도 `rejectPending` 경로로 처리된다고 주석에 적혀 있다(`execution-engine.service.ts:904-906`). 그 경로가 `outputData` 를 별도로 정리/최종화하는지는 본 diff 범위(frontend 4파일) 밖이라 확인하지 못했다. 다만 (a) `apply-execution-snapshot.ts` 는 상태와 무관하게 `outputData` 를 항상 그대로 반영하고, (b) `isConversationOutput` 은 top-level `result.status` 를 전혀 참조하지 않으므로, **어떤 백엔드 취소 경로든 `outputData` 를 명시적으로 비우거나 재작성하지 않는 한** 동일한 노출이 재현된다. 이전 코드는 `status === 'completed'` 게이트가 이 위험을 구조적으로 차단하고 있었는데, 본 diff 로 그 방어선이 제거됐다.
  - 제안: (1) 백엔드가 모든 취소 경로에서 WAITING 당시의 conversation `outputData` 를 CANCELLED 전이 시 정리(예: `status` 필드를 `cancelled` 로 갱신하거나 별도 마커 추가)하는지 확인. (2) 프론트에서 방어적으로 `isConversationHistory` 계산 시 `result.status === 'cancelled'` 를 별도로 배제하거나, `cancelled` 노드 전용 취급을 plan 의 "known follow-up" 목록에 이 구체 경로로 명시. 현재 plan 은 `showTabs` 의 `cancelled` 누락만 별도 후속으로 미뤘을 뿐, 본 경로(`isConversationHistory` 를 통한 간접 노출)는 언급하지 않는다.

(2) 구형 `role:'system'` RAG 마커 포맷이 실제로 여전히 조회 가능한 이력 데이터인지(마이그레이션 스크립트/구 실행 보존 정책) 확인하고, 조회 가능하다면 `messagesToConversationItems` 에도 동등한 폴백을 추가하거나 최소한 known-limitation 으로 plan/spec 에 기록.

## 요약

네 변경 모두 전역 상태·환경변수·파일시스템·네트워크 호출과 무관한 프론트엔드 렌더/스토어 로직이며, 각 변경의 1차 의도(대화형 노드 실패 종결 시 미리보기 도달성 복구)는 코드상 정확히 구현돼 있다. 다만 부작용 관점에서 세 갈래의 "선언된 스코프보다 넓은 파급"이 확인된다: (1) `handleNodeFailed` 의 `outputData` population 은 AI 대화형 노드에 국한되지 않고 error-port 로 실패하는 비AI 노드 8종의 Preview/Meta/Port/Status 탭 가시성까지 바꾸는데 이 범위에 대한 전용 테스트가 없고, (2) `isConversationHistory` 의 status 게이트 제거는 `outputData` 가 결과 자체의 top-level `status` 와 무관하게 신뢰되게 만들어 — 특히 WAITING 도중 취소된 노드가 stale 한 대화 스냅샷을 새로고침 후 미리보기에 그대로 노출할 수 있는, 이전엔 `status==='completed'` 게이트로 원천 차단돼 있던 경로를 새로 연다(백엔드가 취소 시 `outputData` 를 정리하는지는 이번 diff 범위 밖이라 미확인), (3) `conversation-inspector.tsx` 의 items 우선순위 반전은 CT-S16/17 뿐 아니라 사실상 모든 기존 완료 대화 이력 렌더링의 데이터 소스를 SummaryView 인라인 재파싱에서 `parseHistoryMessages` 로 전면 교체하며, 그 과정에서 구형 RAG 마커 표시 경로 하나가 대응점 없이 소실되고, 이 넓은 변경 범위를 커버하는 tool-call 포함 정상 이력 회귀 테스트가 없다. endReason 화이트리스트 확장(3번)은 스코프가 안전하게 제한돼 있어 문제 없음을 확인했다.
