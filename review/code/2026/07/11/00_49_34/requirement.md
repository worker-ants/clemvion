- **[INFO] 확인됨 — `readErrorBody` + `STATE_MISMATCH` 판정 catch 는 요구 동작을 정확히 충족**
  - 위치: `codebase/backend/src/modules/hooks/hooks.service.ts:727-741`(`forwardToInteractionService` catch), `:757-769`(`readErrorBody`)
  - 상세: 프로덕션에서 `STATE_MISMATCH` 409 를 만드는 두 지점(`interaction.service.ts` `dispatchContinuation:415-417`, `assertWaiting:442-447`)이 모두 `new ConflictException({ error: { code: 'STATE_MISMATCH', message } })` shape 을 던지는 것을 직접 추적 확인했다. NestJS `HttpException.initMessage()` 는 응답 객체에 top-level `message` 키가 없으면 `err.message` 를 클래스 기본 문구(`'Conflict Exception'`)로 고정하므로, `readErrorBody` 가 `err.getResponse()` 를 캐스팅해 `error.code`/`error.message` 를 직접 읽는 방식은 정확하다. catch 로직은 `err instanceof ConflictException` 이고 `readErrorBody(err)?.code === 'STATE_MISMATCH'` 인 경우에만 `return`(삼킴)하고, 그 외 모든 경로(다른 409 코드, `ConflictException` 이지만 body 파싱 실패, `ConflictException` 이 아닌 임의 Error)는 `throw err` 로 그대로 전파한다. `hooks.service.spec.ts` 의 3개 신규 테스트(표면 불일치 삼킴+실제 message 로깅 / `IDEMPOTENCY_KEY_CONFLICT` 전파 / 일반 `Error('redis down')` 전파)가 이 세 분기를 각각 비-vacuous 하게 실증한다. 로그 문자열도 `conflict.message ?? '(상세 없음)'` 로 실제 서버측 진단 메시지를 담아 이전 라운드 WARNING #1/#2 를 정확히 해소했다.
  - 참고(비-결함, 문서화 목적): JSDoc(`:670-675`)이 스스로 밝히듯 `STATE_MISMATCH` 는 "표면 불일치" 전용 코드가 아니라 0건/다중-row race·이미 waiting 을 벗어난 경우까지 공유하는 단일 코드다. 따라서 이 catch 는 리터럴하게는 "표면 불일치만" 이 아니라 "STATE_MISMATCH 전체"를 삼킨다 — 다만 이는 의도된 설계(어느 원인이든 provider 재시도로 해결 안 됨)이고 JSDoc/RESOLUTION 이 이미 명시적으로 그 범위를 인정하므로 결함이 아니다.

- **[WARNING] JSONB `COALESCE` 투영이 `readPersistedInteractionType` 의 타입 가드(문자열 검증)를 재현하지 않는다 — 비-문자열 `meta.interactionType` 에서 두 구현이 갈린다**
  - 위치: `execution-engine.service.ts:5199-5204`(`COALESCE(ne.output_data -> 'meta' ->> 'interactionType', ne.output_data ->> 'interactionType')`) vs `waiting-surface-guard.ts:392-405`(`readPersistedInteractionType`)
  - 상세: `readPersistedInteractionType` 은 `meta.interactionType` 이 **문자열일 때만** 채택하고, 아니면(숫자/불리언/객체/배열 등) legacy flat root 로 폴백한다 — 이는 `waiting-surface-guard.spec.ts` 의 `readPersistedInteractionType({ meta: { interactionType: 7 } })` → `undefined` 테스트가 명시적으로 고정한 계약이다. 그러나 SQL `->>` 연산자는 대상이 문자열인지 검사하지 않고 **JSON 스칼라를 텍스트로 강제 변환**한다(숫자 `7` → `'7'`, 불리언 `true` → `'true'`) — 즉 `output_data->'meta'->>'interactionType'` 이 non-NULL 이면(문자열이든 아니든) `COALESCE` 는 이 값을 즉시 채택하고 flat root 로 폴백하지 않는다. 예: `outputData = { meta: { interactionType: 7 }, interactionType: 'buttons' }` (구조화 필드가 오염되고 legacy 필드는 유효한 조합) 에 대해 `readPersistedInteractionType` 은 `'buttons'`(정상 표면 인식)를 반환하지만, 신규 SQL 투영은 `COALESCE('7', 'buttons') = '7'` 을 반환해 `resolveWaitingSurface` 가 어떤 표면도 매칭하지 못하고 `INVALID_EXECUTION_STATE` 로 오거부한다. 두 구현이 "정확히 같은 규칙"이라는 코드 주석(`execution-engine.service.ts:5199-5200`, `waiting-surface-guard.ts:390`)의 line-level 주장과 실제 동작이 어긋난다.
    - 실질 영향 범위는 좁다: (1) 이 코드베이스의 handler 들은 항상 `meta.interactionType` 을 유효한 enum 문자열로만 기록하므로 이 분기는 오직 DB 데이터 손상/수동 편집 시나리오에서만 발현된다. (2) 코드가 갈리는 방향은 "그릇된 허용"이 아니라 "불필요한 거부"(fail-closed 방향) 로 수렴하는 경우가 대부분이다 — coerced 텍스트가 우연히 `'buttons'`/`'ai_conversation'`/`'ai_form_render'` 와 정확히 일치할 확률은 사실상 0이다. (3) 유닛 테스트 harness (`execution-engine.service.spec.ts:1060-1067` `armSlowPathResume`) 도 "프로덕션의 JSONB path 투영과 동일 규칙"이라고 주석에 적어 두고 실제로는 `typeof meta?.interactionType === 'string'` 타입 가드를 재구현한 JS mock 을 쓴다 — 즉 SQL `COALESCE` 의 실제 텍스트-강제변환 동작을 재현하는 테스트가 unit/e2e 어디에도 없다(e2e `execution-park-resume.e2e-spec.ts:409-417` 의 buttons 표면 검증도 정상 문자열 케이스만 다룬다). 결과적으로 이 divergence 는 코드에도 테스트에도 문서화되지 않은 채 남아 있다.
  - 제안: (a) 최소한 SQL 을 `jsonb_typeof(...) = 'string'` 가드로 보강해 비-문자열 값을 명시적으로 NULL 처리하고 flat root 로 폴백하게 하거나, (b) 현재 동작이 의도적으로 "받아들일 수 있는 fail-closed 방향의 근사"임을 주석에서 "동일 규칙"이 아니라 "실무상 동치(정상 데이터 한정)"로 정정 + 회귀 테스트(비-문자열 `meta.interactionType` + 유효한 flat root 병존 입력에 대한 실 Postgres e2e 또는 SQL fragment 단위 테스트)를 추가할 것.

## 요약

집중 지시 4항목 중 (1) hooks catch(`readErrorBody`+`STATE_MISMATCH`)는 요구 동작(표면 불일치만 정확히 좁혀 삼키고 나머지는 전파, 실제 진단 메시지 로깅)을 코드·테스트 양면에서 정확히 충족한다. (4) CHANGELOG 서술도 코드·spec 인용 모두 line-level 로 검증되어 과장·오기가 없다. (3) spec 정합은 §7.5.1/§5.1 본문이 신규 3번째 거부 케이스를 아직 반영하지 않는 SPEC-DRIFT 이나, 이는 이전 라운드가 이미 잡아 plan 에 project-planner 위임으로 명시적으로 이연된 항목이라 이번 fix 커밋의 결함은 아니다(재확인만). 가장 실질적인 신규 발견은 (2) JSONB `COALESCE` 투영이 `readPersistedInteractionType` 의 문자열 타입 가드를 재현하지 않아, 손상된/비정상 `meta.interactionType` 값(정상 handler 는 절대 쓰지 않는 값)에 대해서만 두 "동일 규칙" 구현이 실제로 갈린다는 점이다 — 보안·데이터 무결성 상 위험(그릇된 허용)은 사실상 없고 오거부 방향으로만 수렴하지만, "정확히 같은 규칙"이라는 코드 주석의 line-level 주장이 실제 동작과 어긋나고 이 특정 divergence 를 검증하는 테스트가 어디에도 없다는 점에서 WARNING 으로 기록한다.
