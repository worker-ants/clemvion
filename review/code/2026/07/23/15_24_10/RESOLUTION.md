## Warning (2)

### W1 (performance) — 또 하나의 super-linear 구석. 이번엔 **클래스 전체를 닫았다**

`_commit_heredoc_spans` 는 한 줄에 `<<` 마커가 여럿이고 소유권 판정이 매번 **실패**하면
마커마다 누적 prefix 를 재스캔했다.

**재현**: 첫 시도는 선형으로 나왔다 — 내 PoC 가 소유권 판정에 **성공**해 루프가 한 번에
끝나버렸기 때문이다. 판정이 매번 실패하는 형태로 다시 만들자 재현됐다:
0.167s(13KB) → 0.694s(28KB) → 2.88s(57KB) → **11.59s(118KB)**. 입력 2배 → 시간 4배.
(리뷰어 주장을 못 믿어서가 아니라, **재현 실패가 곧 반증은 아니라는 것**을 확인한 셈이다.)

**원인은 하나가 아니라 둘**이었다:
1. 소유권 검사에 넘기는 슬라이스가 줄 맨 앞부터 자라남,
2. `text.rfind("\n", 0, m.start())` — 개행이 없으면 매 마커마다 prefix **전체를 역스캔**.

1번만 고쳤을 때 56배 빨라졌지만(11.6s→0.205s) **여전히 quadratic** 이었다(×4/2배). 2번까지
`_OWNER_WINDOW`(512B)로 묶고 나서야 선형이 됐다 — 246KB×2=504KB 구간에서 ×2.00, 0.038s.

**그리고 개별 hot spot 추적을 멈췄다.** 3라운드 연속 같은 클래스(모호한 alternation → 누적
슬라이스 → 누적 rfind)가 나왔으므로, 다음 인스턴스를 기다리는 대신 **입력 크기 상한**을 뒀다:

```
_MAX_REDACTION_INPUT = 16_384   # 초과 시 redaction 생략 + 즉시 차단
```

초과 입력은 **차단**(= allowlist 이전 동작 = 안전 방향)이라 우회를 만들지 않는다. 미발견
quadratic 이 남아 있어도 16KB 에서 유계다(관측된 최악 상수 기준 <0.2s). 2라운드 리뷰어가
"최소 방어책" 으로 제안했던 것이기도 하다.

**회귀 테스트**: 상한이 `_is_git_push` 경로를 먼저 막으므로, 스캔 자체의 선형성은
`_commit_heredoc_spans` 를 **직접** 겨냥해 검증한다(24,000 오프너 / 246KB — 수정 전 ~29s,
수정 후 0.02s). 상한 자체도 별도 테스트(초과→차단 / 미만→해제 / 현실 크기 여유).
