- **[WARNING]** 락 제거로 수용한 "첫 cold-install 동시 경쟁" 잔여 리스크가, 설계 노트의 서술("조용히 방치, 수동 복구")보다 나쁜 결과 — **fail-open 약속이 깨져 커밋이 잘못된 메시지로 차단됨** — 를 낳을 수 있음. 직접 재현으로 확인.
  - 위치: `.claude/tools/bootstrap-session.sh` L87-97(마커는 `npm install` 자기 exit code 만 보고 무조건 기록, L91 `: > "$marker" ...`) · `.githooks/pre-commit` L50-55(신규 주석, `is_ready` 가 "half-written node_modules... blocking the commit" 문제를 해소했다고 서술) L70-74(비-zero exit → 무조건 "malformed mermaid block(s)"로 커밋 차단, `exit 1`). 소비측 실제 동작은 이 리뷰 페이로드 밖의 `.claude/tools/mermaid-lint/lint-mermaid.mjs:93`(`const { default: mermaid } = await import("mermaid");`, try/catch 없는 최상위 await)와 `.claude/hooks/lint_mermaid_posttooluse.py:135-141`(`proc.returncode != 0` 이면 무조건 "mermaid syntax error"로 취급, `return 2`)에서 확정됨.
  - 상세: `bootstrap-session.sh` 의 설계 노트(L65-72)는 락 제거로 수용한 잔여 리스크를 "여러 세션이 첫 cold install 을 동시에 치면 npm install 이 동시 실행돼 드물게 corrupt-but-marked node_modules 가 남고, `rm -rf node_modules` 로 수동 복구"라고 서술한다 — "조용히 방치되는" 결과처럼 읽힌다. 그러나 마커는 **이 세션 자신의 npm install 프로세스가 exit 0 을 반환했는가**만 보고 무조건 기록되며 node_modules 트리 내용의 무결성은 전혀 검증하지 않는다. `is_ready()`(`.claude/hooks/_lib/mermaid_lint_ready.py`)는 그 마커를 그대로 신뢰한다. 즉 동시 설치 중 **자신의 npm install 은 성공했지만 형제 프로세스의 동시 쓰기로 인해 실제 트리는 손상된** 상태가 "ready"로 오판될 수 있다. 그 상태에서 `lint-mermaid.mjs` 가 실행되면 try/catch 없는 `await import("mermaid")` 가 `ERR_MODULE_NOT_FOUND` 로 즉시 크래시하고, 이 크래시의 종료 코드(non-zero)는 `.githooks/pre-commit`(L70)과 `lint_mermaid_posttooluse.py` 양쪽 모두에서 "진짜 mermaid 파싱 에러를 찾았다"와 **구분되지 않는다** — 그 결과 매 markdown 커밋마다 "Fix the malformed mermaid block(s) above and re-stage."라는 **사실과 다른** 메시지로 커밋이 차단되고, PostToolUse 쪽은 Claude 에게 "mermaid 블록을 고치라"고 잘못 지시한다. 이는 두 파일이 최상단에 명시한 설계 불변식("Failure mode: every guard fails open... a broken guard must not block routine work")과 정확히 반대되는 결과다. 또한 `.githooks/pre-commit` 의 새 주석(L50-55)은 `is_ready()` 도입으로 "partial deps... blocking the commit" 문제가 해소된 것처럼 서술하지만, 이는 "마커가 아예 없는" 경우만 닫을 뿐 "마커는 있지만 형제 프로세스가 오염시킨" 경우(바로 이 diff 가 락을 빼며 수용한 경우)는 닫지 않는다 — 같은 diff 안에서 두 파일의 서술이 서로를 반영하지 못해 이 gap 이 눈에 띄지 않게 됐다.
  - **실측 검증**: 실제 저장소 main 체크아웃에 이미 설치돼 있는 `mermaid`/`jsdom` 패키지를 스크래치 디렉토리에 심볼릭 링크로 격리 복제 → 기준선(정상 트리)에서 `node lint-mermaid.mjs valid.md` 는 exit 0 확인 → `mermaid` 패키지의 `package.json` 만 남기고 엔트리포인트(`exports["."].import` = `./dist/mermaid.core.mjs`)가 가리키는 `dist/` 를 제거해 "동시 쓰기로 일부만 반영된 부분 설치"를 시뮬레이션 → 동일 명령이 **미처리 `ERR_MODULE_NOT_FOUND` 예외로 크래시, exit 1**. 이 exit 코드는 pre-commit·PostToolUse 양쪽 코드 경로 상 "파싱 에러 발견"과 동일하게 처리됨을 소스로 확인했다. 참고로 `mermaid` 패키지 자체가 75MB(전체 tool_dir node_modules 163MB 중 최대 비중)로, "매니페스트는 먼저, 큰 파일은 늦게 쓰인다"는 일반적 npm 언팩 순서상 동시 설치 시 다른 작은 패키지보다 이 패키지가 오히려 더 자주 "부분 상태"로 걸릴 가능성이 있다(실제 repo 데이터로 확인한 사실, 반증되지 않은 추정).
  - 제안: (1) `lint-mermaid.mjs` 의 `await import("mermaid")`/`await import("jsdom")` 를 try/catch 로 감싸 로드 실패를 "파싱 에러"와 다른 종료 코드(예: 3)로 분리하고, 양쪽 소비처(pre-commit bash, PostToolUse python)가 그 코드를 "툴링 깨짐 → fail open"으로 명시 처리하도록 배선. (2) 또는 `is_ready()` 가 마커만이 아니라 최소한의 스모크 체크(예: 핵심 엔트리포인트 파일 존재 확인 1회, 실패 시 not-ready)를 추가. (3) 최소한 `bootstrap-session.sh` 의 설계 노트가 실제 최악의 경우("조용히 방치"가 아니라 "fail-open 약속이 깨져 매 markdown 커밋이 차단됨")를 정확히 반영하도록 갱신 — 이 프로젝트가 다른 잔여 리스크들에 이미 보여준 정직한 서술 수준에 맞춰서.

## 요약

이번 변경은 직전 리뷰(02_06_42 C1, CRITICAL)에서 실측 재현된 `mkdir` 락의 stale-lock 탈취 TOCTOU 를 락 자체를 통째로 제거하는 방식으로 근본 해소한다 — 락이 없으니 훔칠 대상도 없고, 이는 코드 diff 로 명확히 확인된다. 사용자 결정에 따라 수용한 잔여 리스크(첫 cold install 시 동시 `npm install`)는 설계 노트에 상세히 문서화돼 있고, 그 자체는 dev-tooling 린터 범위에서 합리적인 트레이드오프다. 다만 이번 리뷰에서 그 잔여 리스크의 **실제 최악 결과**가 설계 노트의 서술("조용히 방치, 수동 `rm -rf` 복구")보다 나쁘다는 점을 실제 저장소의 mermaid 패키지를 이용한 격리 재현으로 직접 확인했다 — 마커는 트리 무결성이 아니라 자기 자신의 npm exit code 만 보므로, 동시 쓰기로 손상됐지만 "ready"로 오판된 트리에서 `lint-mermaid.mjs` 의 가드 없는 최상위 `await import("mermaid")` 가 크래시하고, 그 크래시의 종료 코드는 pre-commit·PostToolUse 양쪽에서 "진짜 mermaid 문법 에러"와 구분되지 않아 — 두 파일이 명시한 "fail open" 불변식과 반대로 매 markdown 커밋이 잘못된 메시지("malformed mermaid block(s)")로 차단될 수 있다. 이는 이미 수용된 좁은 확률의 1차 경쟁(첫 cold-install 동시성)에 종속된 2차 결과라 CRITICAL 로 보기는 어렵지만, 이 diff 의 accepted-risk 서술이 다루지 않는 실재하는 gap 이므로 WARNING 으로 표시한다. 부수적으로, 락 제거가 기존에 알려졌던 "npm install 무제한 대기" 한계의 blast radius 를 (한 세션→모든 레이싱 세션으로) 넓혔다는 점도 어디에도 트래킹돼 있지 않아 INFO 로 남긴다. 그 외에는 마커의 원자적 쓰기, throttle 의 동시-삭제 안전성, git config 갱신의 idempotency, GC 의 동시-삭제 안전성, 그리고 신규 동시성 테스트의 구성(진짜 동시 기동·DEVNULL·수렴만 단언)이 모두 견고하게 확인됐다.
