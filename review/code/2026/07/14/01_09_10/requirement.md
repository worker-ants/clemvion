# 요구사항(Requirement) Review — eia-command-waiting-surface-guard F-1 (nodeId 일치 검사)

## 발견사항

- **[CRITICAL]** `spec/5-system/4-execution-engine.md` §7.5.1 새 "nodeId 불일치" 행/exemption 문단이 WS 도 nodeId 를 지정해 검사받는다고 서술하지만, 실제 코드는 WS 채널에 이 검사를 전혀 배선하지 않았고 이는 같은 PR 의 다른 문서(코드 JSDoc, `spec/5-system/6-websocket-protocol.md`)와도 정면으로 모순된다.
  - 위치: `spec/5-system/4-execution-engine.md` §7.5.1 (diff 상 `+| nodeId 불일치 | ... **caller 가 nodeId 를 지정할 때만** 적용 — 외부 EIA `/interact`(`InteractDto.nodeId`)**·WS 는 지정하고**, ...` 및 바로 아래 `> in_process_trusted nodeId 면제` 문단) / `codebase/backend/src/modules/websocket/websocket.gateway.ts` (`handleSubmitForm`/`handleClickButton`/`handleSubmitMessage`/`handleEndConversation`, 각각 `continueExecution`/`continueButtonClick`/`continueAiConversation`/`endAiConversation` 호출) / `codebase/backend/src/modules/execution-engine/execution-engine.service.ts` `resolveWaitingNodeExecutionId` JSDoc (`@param expectedNodeId ... 외부 EIA \`/interact\` 진입점만 전달하고 ...`) / `spec/5-system/6-websocket-protocol.md` L205, L206, L260, L1050 (Rationale)
  - 상세: 이번 diff 는 `resolveWaitingNodeExecutionId(executionId, expectedCommand, expectedNodeId?)` 에 nodeId 불일치 거부 로직을 추가하고, 이를 실제로 호출부에 배선한 곳은 `InteractionService.interact` (EIA REST `/interact`, `isInternalCtx` 로 chat-channel `in_process_trusted` 만 면제) 뿐이다. 그런데 새로 쓴 spec 문장은 "외부 EIA `/interact`(`InteractDto.nodeId`)**·WS 는 지정**하고 ... `in_process_trusted` 는 면제한다" 고 서술해, WS 도 nodeId 를 지정해 이 검사를 받는 것처럼 규정한다. 실제로 `websocket.gateway.ts` 의 4개 continuation handler 는 `data.nodeId`(`click_button`/`submit_message`/`end_conversation` 는 메시지 타입에 필드가 있음)를 **엔진 메서드 호출 시 3번째 인자로 전혀 전달하지 않는다** (`continueExecution(data.executionId, data.formData)`, `continueButtonClick(data.executionId, data.buttonId)`, `continueAiConversation(data.executionId, data.message)`, `endAiConversation(data.executionId)` — 전부 2-arg 이하). 즉 WS 경로는 `expectedNodeId` 가 항상 `undefined` 라 nodeId 불일치 검사가 원천적으로 스킵된다. 게다가 같은 diff 로 추가된 `resolveWaitingNodeExecutionId` 의 JSDoc 자체가 "외부 EIA `/interact` 진입점만 전달"이라고 명시해 spec 새 문장과 자기모순이며, 이번 diff 가 건드리지 않은 기존 spec `6-websocket-protocol.md` 는 이미 "`nodeId` 는 선택 수신되나 서버가 사용하지 않는다 (대기 노드 식별은 server lookup)"(L206), "ack data 에 `nodeId` 는 포함되지 않는다 ... 클라이언트는 `executionId` 만으로 매칭"(L260), 그리고 명시적 Rationale(L1050) "양단 구현(frontend, `websocket.gateway.ts`) 모두 `nodeId` 를 보내지도 받지도 않는다" 라고 WS 는 nodeId 를 원래부터 쓰지 않는 설계임을 못박고 있다. `plan/in-progress/eia-command-waiting-surface-guard.md` 의 F-1 항목 배경/체크리스트도 `interaction.service.ts`/`hooks.service.ts` 만 언급하며 WS 확장은 전혀 계획에 없었다 — 즉 새 spec 문장의 "WS 는 지정하고" 는 이번 spec 동기화 과정에서 들어간 오기(overclaim)로 보인다.
  - 영향: F-1 의 취지("stale/오지정 nodeId 제출을 현재 대기 노드로 오적용하지 않고 거부")가 WS 채널에는 실제로 적용되지 않은 채, spec 문서만 "적용된다"고 주장한다. 향후 이 spec 문장을 근거로 WS 가 안전하다고 판단하는 감사/후속 작업(예: `/consistency-check`, `/spec-coverage`)이 실제로 열려 있는 gap 을 놓칠 위험이 있다.
  - 제안: `project-planner` 에 위임해 spec 정정 — §7.5.1 "nodeId 불일치" 행/exemption 문단에서 "WS 는 지정하고" 부분을 제거하고 "현재 nodeId 검사는 EIA REST `/interact` 진입점에만 적용되며, WS 는 프로토콜 설계상(§6-websocket-protocol.md) nodeId 를 서버에 전달하지 않아 이 검사의 적용 대상이 아니다" 로 정정하거나, 혹은 (보안 강화가 실제 의도라면) WS 4개 handler 가 `data.nodeId` 를 `expectedNodeId` 로 forward 하도록 별도 후속 작업으로 명시 — 어느 쪽이든 spec 과 코드가 같은 사실을 말하도록 결정·반영 필요.

- **[WARNING]** `InteractionService` 클래스 JSDoc 의 "dispatch 매핑" 표가 새 `expectedNodeId` 3번째 인자를 반영하지 않음 (의도-구현 괴리, 문서 drift).
  - 위치: `codebase/backend/src/modules/external-interaction/interaction.service.ts` 클래스 JSDoc (`* dispatch 매핑 ([Spec EIA §11]): * submit_form → ExecutionEngineService.continueExecution(executionId, data) * ...`)
  - 상세: 실제 호출부는 `continueExecution(ctx.executionId, dto.data, expectedNodeId)` 등 3-arg 로 바뀌었으나, 클래스 상단 요약 주석은 여전히 2-arg 시그니처를 나열한다. 기능상 문제는 없으나 이번 F-1 변경의 핵심(추가 파라미터)이 요약 문서에서 누락돼 있어 향후 리더가 오래된 정보를 신뢰할 수 있다.
  - 제안: 각 매핑 라인 끝에 `, expectedNodeId` 를 덧붙여 최신 시그니처와 동기화.

- **[INFO]** 새 nodeId 불일치 케이스의 클라이언트 노출 메시지는 안전하게 설계되어 있음(검증 완료, 이슈 아님) — 참고용으로 기록.
  - 위치: `execution-engine.service.ts` 의 `throw new InvalidExecutionStateError(\`command nodeId=${expectedNodeId} does not match waiting node=${rows[0].nodeId} ...\`)` / `workflow-errors.ts` `InvalidExecutionStateError constructor(detail?: string) { super('Execution is not waiting for input.', detail); }`
  - 상세: `InvalidExecutionStateError` 의 단일 생성자 인자는 항상 `serverDetail` 로만 대입되고 `message` 는 고정 문자열 `'Execution is not waiting for input.'` 로 유지되므로(§7.5.2 client-safe 계약 준수), 새 nodeId 불일치 상세 문자열(실제 대기 nodeId 노출)은 `err.message` 를 그대로 forward 하는 `interaction.service.ts` `dispatchContinuation` 경로를 통해서도 client 에 새지 않는다. 기존 표면-불일치 케이스의 동일 패턴(`거부 메시지는 client-safe 고정 문자열, 상세는 serverDetail 에만` 테스트, `execution-engine.service.spec.ts` L2380 부근)과 일관되게 잘 지켜졌다. Finding 으로 남기지 않고 확인 사실만 기록.

## 요약

핵심 변경(EIA REST `/interact` 진입점에 nodeId-대기노드 일치 검사를 추가하고 `in_process_trusted`(chat-channel 고정 매핑)를 명시적으로 면제)은 unit(`execution-engine.service.spec.ts`, `interaction.service.spec.ts`)과 e2e(`external-interaction.e2e-spec.ts` G-2)로 잘 검증되며, `isInternalCtx` 판정·`assertNodeId` 우회·`InvalidExecutionStateError` 의 client-safe/serverDetail 분리 등 구현 디테일도 견고하다. 그러나 이번 diff 가 함께 갱신한 spec 문장(`4-execution-engine.md` §7.5.1)이 "WS 도 nodeId 를 지정해 검사받는다"고 서술하는데, 실제 WS gateway 코드는 (기존부터, 그리고 이번 PR 로도) nodeId 를 엔진 호출에 전달하지 않으며, 이는 같은 diff 의 코드 JSDoc 및 이미 존재하는 `6-websocket-protocol.md` 의 명시적 서술("WS 는 nodeId 를 보내지도 받지도 않는다")과 정면으로 모순된다. F-1 의 원래 계획 문서(`plan/in-progress/eia-command-waiting-surface-guard.md`)도 WS 확장을 전혀 언급하지 않아, 이는 spec 동기화 과정의 오기로 보인다. 이 한 가지를 제외하면 기능 완전성·에러 시나리오·반환값·비즈니스 로직 반영은 스코프(EIA REST + chat-channel 면제) 내에서 충실하다.
