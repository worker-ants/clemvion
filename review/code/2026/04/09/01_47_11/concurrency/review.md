### 발견사항

이 변경사항은 모두 **마크다운 스펙/PRD 문서**입니다. 실행 가능한 코드가 아니므로 직접적인 동시성 버그는 없습니다. 다만, 문서에 기술된 설계 결정 중 구현 시 동시성 문제를 유발할 수 있는 부분을 분석합니다.

---

- **[WARNING]** Multi Turn 모드의 `_turnDebugHistory` 누적 상태 관리
  - 위치: `spec/4-nodes/3-ai-nodes.md` — `디버그 데이터 (_turnDebugHistory)` 섹션
  - 상세: 새로 추가된 `_turnDebugHistory`는 턴마다 `llmCalls` 배열에 항목이 누적됩니다. Multi Turn 모드는 WebSocket을 통해 비동기적으로 사용자 메시지를 수신하므로, 구현 시 동일 실행 컨텍스트에 복수 요청이 동시에 접근하면 배열 append에 경쟁 조건이 발생할 수 있습니다. 스펙에 상태 변경의 직렬화 보장 여부가 명시되어 있지 않습니다.
  - 제안: 스펙에 "각 턴의 `_turnDebugHistory` 업데이트는 직렬화된 컨텍스트에서 수행되며, 동시 메시지 수신은 큐잉 처리한다"는 제약을 명시하세요.

- **[WARNING]** 복수 조건 동시 호출 결정 로직과 상태 전환의 원자성
  - 위치: `spec/4-nodes/3-ai-nodes.md` — `조건 도구 호출 감지 및 처리` 섹션 (스텝 2a, 2b)
  - 상세: "복수의 조건 도구가 호출된 경우 `conditions` 배열에서 인덱스가 가장 작은 조건을 선택 → AI Agent를 **즉시 종료**"하는 로직에서, 조건 선택과 Agent 종료(상태 전환)가 원자적으로 처리되어야 합니다. 구현에서 이 두 연산 사이에 다른 툴콜 처리가 끼어들면 이중 종료 또는 잘못된 포트 라우팅이 발생할 수 있습니다.
  - 제안: 스펙에 "조건 도구 감지 → 선택 → 종료 전환은 단일 동기 블록에서 처리하며 중간에 다른 tool call 처리가 개입될 수 없다"고 명시하세요.

- **[INFO]** `waiting_for_input` 상태 전환 시 타임아웃 처리의 에러 포트 통합
  - 위치: `spec/4-nodes/3-ai-nodes.md` — Multi Turn 종료 조건 3d항
  - 상세: `turnTimeout` 초과 시 `timeout` 포트 → `error` 포트로 통합하는 변경이 이루어졌습니다. 그런데 `endReason` enum에 `timeout` 값이 여전히 존재하며 스펙에서도 "여전히 유효한 값"이라고 명시합니다. 구현에서 타임아웃 이벤트와 LLM 오류 이벤트가 거의 동시에 발생하면, 둘 다 `error` 포트로 라우팅을 시도하는 이중 종료 상황이 생길 수 있습니다. 스펙에 "최초 종료 사유만 유효하고, 이후 종료 시도는 무시(idempotent)"라는 언급이 없습니다.
  - 제안: "AI Agent 종료는 first-wins 방식으로 처리하며, 종료 상태 전환 후 수신되는 모든 종료 이벤트는 무시한다"는 규칙을 스펙에 추가하세요.

- **[INFO]** 조건 + 일반 도구 혼재 시 "일반 도구 우선 실행" 중 조건 상태 보류(pending) 처리
  - 위치: `spec/4-nodes/3-ai-nodes.md` — 스텝 3b
  - 상세: 조건 도구를 "보류"하고 일반 도구를 먼저 실행하는 로직에서, 일반 도구 실행이 비동기(HTTP 호출 등)이므로 보류 중인 조건 상태가 메모리에 유지됩니다. 이 기간 동안 외부 요청(사용자 종료 명령 등)이 수신되면 보류 조건과 충돌할 수 있습니다. 스펙에 이 케이스의 처리 방식이 정의되어 있지 않습니다.
  - 제안: "일반 도구 실행 중 `end_conversation` 수신 시: 일반 도구 완료 후 LLM 재평가를 건너뛰고 `user_ended`로 즉시 종료한다" 등의 인터럽트 규칙을 명시하세요.

---

### 요약

변경된 파일은 모두 설계 문서이므로 코드 수준의 동시성 버그는 없습니다. 그러나 Multi Turn AI Agent의 특성상 — WebSocket 기반 비동기 메시지 수신, 턴 간 상태 누적, 다중 종료 사유 통합 — 구현 단계에서 경쟁 조건과 이중 종료 문제가 발생할 소지가 있습니다. 특히 `_turnDebugHistory` 누적, 복수 조건 동시 호출 처리, `timeout`/`error` 통합에 따른 이중 종료 시나리오에 대해 스펙 레벨에서 직렬화 및 first-wins 규칙을 명시적으로 보완하면 구현의 안전성을 높일 수 있습니다.

### 위험도

**LOW** — 문서 변경으로 인한 직접적 동시성 버그는 없으나, 스펙의 모호한 부분이 구현 시 동시성 결함으로 이어질 가능성이 있습니다.
