## 발견사항

---

### **[CRITICAL]** Merge 노드 포트 ID 변경 — 기존 워크플로우 데이터 파손 위험
- **위치**: `frontend/src/lib/node-definitions/index.ts` — merge 항목
- **상세**: `in_0`, `in_1` → `in` 으로 포트 ID가 변경되었습니다. 이 정의는 새 노드를 캔버스에 추가할 때 초기 포트 구성을 제공하지만, 이미 저장된 워크플로우 JSON에는 `in_0` / `in_1` ID로 연결된 엣지가 남아있습니다. 해당 엣지들은 존재하지 않는 포트를 참조하게 되어 실행 시 dangling edge / silent drop 이 발생합니다. DB 마이그레이션 없이 이 변경만 배포하면 기존 Merge 노드를 포함한 모든 워크플로우가 손상됩니다.
- **제안**: 
  1. 배포 전 기존 워크플로우 데이터에서 `in_0` / `in_1` 엣지를 `in`으로 마이그레이션하는 DB 스크립트 작성 필요
  2. 또는 캔버스 로드 시 레거시 포트 ID를 자동 변환하는 호환성 레이어 추가

---

### **[WARNING]** `inputCount` config 값의 잔존 — 무해하나 쓰레기 데이터 축적
- **위치**: `logic-configs.tsx` MergeConfig, `spec/4-nodes/1-logic-nodes.md`
- **상세**: UI에서 `inputCount` 필드가 제거되었고 스펙에서도 삭제되었습니다. 그러나 기존에 저장된 워크플로우 config에 `inputCount` 값이 남아있어도 이를 제거하는 코드가 없습니다. 백엔드 핸들러(`MergeHandler`)가 이 값을 무시하는 한 런타임 오류는 없지만, config 객체에 의미없는 필드가 영구적으로 잔존합니다.
- **제안**: 백엔드 `MergeHandler.validate()` 또는 실행 전처리 단계에서 `inputCount` 무시 처리를 명시적으로 문서화하거나, 마이그레이션에서 함께 제거

---

### **[WARNING]** 캔버스 요약 포맷 `{N} inputs` — 산출 로직 불명확
- **위치**: `spec/4-nodes/1-logic-nodes.md` §13 캔버스 요약
- **상세**: Merge 노드 요약 포맷이 `{N} inputs · {strategy}` 인데, 기존에는 `inputCount` config에서 N을 읽었을 것입니다. 이 값이 제거된 이후 N을 어디서 계산하는지 스펙에 명시되지 않았습니다. 런타임에 실제 연결된 엣지 수를 세야 한다면 별도 구현이 필요합니다.
- **제안**: 스펙에 "N = 실제 연결된 엣지 수 (런타임 집계)" 명시, 또는 캔버스 렌더링 컴포넌트에서 엣지 수를 주입하는 방식 정의

---

### **[INFO]** `merge.handler.spec.ts` — object 입력 정렬 의존성
- **위치**: `merge.handler.spec.ts` — `should sort object keys for deterministic ordering`
- **상세**: 테스트가 JavaScript 객체 키의 알파벳 정렬에 의존합니다. 실제 멀티엣지 입력이 실행 엔진에서 어떤 키 형식으로 전달되는지(`nodeA`, `a_node` 형식)가 확정되지 않으면, 실행 순서 보장이 테스트와 프로덕션 간에 달라질 수 있습니다. `strategy: first`의 "첫 번째" 결정도 이 정렬에 의존합니다.
- **제안**: 실행 엔진이 Merge 핸들러에 전달하는 입력 객체의 키 포맷과 순서 보장을 스펙에 명시

---

## 요약

이번 변경의 핵심은 Merge 노드를 동적 다중 입력 포트(`in_0`, `in_1`, ...) 방식에서 단일 포트 다중 엣지(`in`) 방식으로 아키텍처를 전환한 것입니다. UI, 스펙, 테스트는 일관되게 수정되었으나, **기존에 저장된 워크플로우 데이터의 포트 ID(`in_0`/`in_1`)가 새 정의(`in`)와 불일치하는 파괴적 변경(breaking change)**이 DB 마이그레이션 없이 존재합니다. 이는 배포 시 즉각적인 데이터 파손을 유발하므로 반드시 마이그레이션 스크립트 또는 호환성 레이어가 선행되어야 합니다.

## 위험도

**HIGH** — 데이터 마이그레이션 없이 배포 시 기존 워크플로우 파손
