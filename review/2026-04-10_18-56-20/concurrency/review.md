### 발견사항

- **[WARNING]** `first` 전략의 비결정적 동시 도착 순서 처리 불완전
  - 위치: `merge.handler.spec.ts` — `strategy: first` describe 블록
  - 상세: 테스트에서 `a_node` < `b_node` 알파벳 정렬을 통해 "first arrived"를 시뮬레이션하지만, 실제 병렬 브랜치 실행 시 입력 도착 순서는 키 이름이 아닌 실행 완료 시점에 따라 결정된다. 테스트가 결정론적 정렬로 커버하는 케이스와 실제 비결정적 도착 순서 사이의 갭이 존재한다. 실제 구현에서 `first` 전략이 진정한 "가장 먼저 도착한 입력"을 기준으로 동작하는지, 아니면 단순 배열 첫 번째 요소를 기준으로 동작하는지 테스트로는 검증되지 않는다.
  - 제안: 타임스탬프 또는 도착 순서 메타데이터를 기반으로 한 `first` 전략 테스트 추가. 정렬 기반 결정론적 케이스와 별도로, 실제 "도착 순서"를 의미하는 계약을 스펙/테스트에 명시

- **[WARNING]** `wait_all`/`append` 전략에서 부분 도착 상태 누적의 스레드 안전성 미검증
  - 위치: `merge.handler.spec.ts` 전체 `execute` 블록
  - 상세: 모든 테스트가 입력을 미리 집계된 객체(`{ nodeA: ..., nodeB: ... }`)로 한 번에 전달한다. 그러나 실제 워크플로우 실행에서는 병렬 브랜치의 입력이 비동기적으로 순차 도착한다. 핸들러가 실행 컨텍스트별 대기 상태(partial inputs buffer)를 내부에서 유지한다면, 동일 노드로의 동시 다중 호출 시 해당 버퍼에 대한 경쟁 조건이 발생할 수 있다. 테스트는 이 시나리오를 전혀 다루지 않는다.
  - 제안: 실행 엔진이 모든 입력을 집계한 후 핸들러를 단일 호출하는 계약을 문서화하거나, 핸들러 내부에 상태가 있다면 원자적 누적 로직에 대한 테스트 추가

- **[WARNING]** `timeout` 설정에 대한 테스트 완전 누락
  - 위치: `merge.handler.spec.ts` — `validate` 및 `execute` 전체
  - 상세: 스펙(`1-logic-nodes.md`)에 `timeout` (기본 300초)과 `partialOnTimeout` 필드가 정의되어 있고, 타임아웃 시 `MERGE_TIMEOUT` 에러 또는 부분 병합을 수행해야 한다. 동시성 관점에서 타임아웃은 Merge 노드의 핵심 안전장치임에도 불구하고, `validate` 테스트에서 `timeout` 필드 검증도 없고 `execute` 테스트에서도 타임아웃 동작 검증이 전혀 없다.
  - 제안: `partialOnTimeout=false` 시 `MERGE_TIMEOUT` 에러 발생, `partialOnTimeout=true` 시 도착한 입력만으로 병합 수행하는 테스트 추가

- **[INFO]** 포트 모델 변경(`in_0`/`in_1` → 단일 `in` 다중 엣지)에 따른 동시성 책임 이동
  - 위치: `index.ts` line 39, `spec/1-logic-nodes.md` 포트 섹션
  - 상세: 기존 다중 포트 모델은 각 입력 소스가 명확히 구분된 포트로 도달했다. 단일 `in` 포트에 다중 엣지를 수신하는 모델로 변경됨에 따라, 여러 엣지로부터 오는 입력의 수집·동기화 책임이 핸들러에서 실행 엔진으로 이동한다. 실행 엔진이 모든 엣지의 결과를 집계하는 시점과 방식에 따라 `wait_all` 전략의 정확성이 결정된다.
  - 제안: 실행 엔진의 다중 엣지 집계 로직에 대한 통합 테스트 또는 명시적 계약 문서화 필요

### 요약

변경된 코드는 직접적인 공유 메모리 동기화나 락 관련 문제는 없으나, Merge 노드의 본질적인 역할(병렬 브랜치 결과 수집)과 관련하여 동시성 관점의 테스트 공백이 존재한다. 특히 `first` 전략의 비결정적 도착 순서 처리, `wait_all`/`append` 전략에서의 부분 도착 상태 누적 안전성, 타임아웃 동작 검증이 테스트에 반영되지 않았다. 포트 모델 변경(`in_0`/`in_1` → 단일 `in` 다중 엣지)은 아키텍처적으로 타당하지만, 이로 인해 실행 엔진 레벨의 동시성 집계 계약이 명확히 정의되고 검증되어야 한다.
