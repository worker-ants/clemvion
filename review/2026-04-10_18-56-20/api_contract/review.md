### 발견사항

- **[WARNING]** Merge 노드 포트 구조 변경으로 인한 기존 워크플로우 데이터 하위 호환성 파손
  - 위치: `index.ts` diff, `1-logic-nodes.md` diff
  - 상세: 기존에 `in_0`, `in_1`, ... 형태의 동적 다중 포트를 사용하던 저장된 워크플로우 데이터가 있다면, 단일 `in` 포트로 변경 후 해당 엣지 연결 정보(`sourceHandle`/`targetHandle` = `in_0`, `in_1`)가 유효하지 않은 포트 ID를 참조하게 된다. DB에 저장된 워크플로우 정의의 엣지 데이터가 깨진다.
  - 제안: 워크플로우 저장/불러오기 API에서 마이그레이션 로직 또는 포트 ID 정규화 처리 필요. 또는 API 레벨에서 `in_0`/`in_1` → `in`으로 변환하는 하위 호환 레이어 추가.

- **[WARNING]** 백엔드 핸들러(`merge.handler.spec.ts`)의 입력 모델과 포트 변경 간 정합성 확인 필요
  - 위치: `merge.handler.spec.ts` - `input normalization` 섹션
  - 상세: 스펙 변경에서 입력이 "단일 포트에 다중 엣지 수신"으로 바뀌었는데, 테스트 코드는 여전히 `{ nodeA: ..., nodeB: ... }` 형태의 객체 입력(소스 노드 ID를 키로 사용)을 기대한다. 이는 실행 엔진이 다중 엣지를 어떻게 집계하여 핸들러에 전달하는지에 대한 내부 API 계약이다. 단일 `in` 포트로의 변경이 실행 엔진의 입력 집계 방식에 영향을 주는지, 핸들러가 받는 입력 구조가 동일한지 명시적인 문서/테스트가 없다.
  - 제안: 실행 엔진이 단일 포트 다중 엣지를 핸들러에 전달하는 방식(예: `{ sourceNodeId: value }` 객체 유지 여부)을 스펙 또는 코드 주석으로 명시. 핸들러 스펙 파일에 "단일 `in` 포트 + 다중 엣지" 시나리오 테스트 케이스 추가.

- **[INFO]** `inputCount` config 필드 제거에 따른 기존 저장 설정 처리
  - 위치: `logic-configs.tsx` diff, `1-logic-nodes.md` diff
  - 상세: 기존에 `inputCount`가 포함된 노드 config를 가진 워크플로우가 로드될 때, 해당 필드는 단순히 무시된다. 실행 엔진 핸들러의 `validate()`가 `inputCount`를 검증하지 않으므로 에러는 발생하지 않으나, 오래된 데이터에 불필요한 필드가 잔류하게 된다.
  - 제안: 워크플로우 저장 API에서 `merge` 노드 config의 `inputCount` 필드를 자동으로 정리하는 마이그레이션 스크립트 또는 저장 시 정규화 처리를 고려.

### 요약

이번 변경은 REST API 엔드포인트 자체를 수정하는 것이 아니라, **워크플로우 노드 정의 스키마(포트 구조)와 노드 실행 핸들러의 내부 API 계약**을 변경하는 것이다. 핵심 위험은 Merge 노드의 포트 ID가 `in_0`/`in_1` → `in`으로 변경되면서 **DB에 저장된 기존 워크플로우의 엣지 데이터가 파손될 수 있다는 하위 호환성 문제**다. 프런트엔드 UI와 스펙은 일관되게 변경되었고, 백엔드 핸들러의 입력 정규화 로직(`{ nodeId: value }` 객체를 배열로 변환)은 새로운 단일 포트 다중 엣지 모델과 개념적으로 호환되나, 실행 엔진이 이 입력을 핸들러에 전달하는 방식이 변경 없이 유지됨을 명시적으로 검증하는 테스트가 부재하다. 저장된 워크플로우 데이터에 대한 마이그레이션 전략이 없다면 운영 환경에서 기존 워크플로우가 손상될 수 있다.
