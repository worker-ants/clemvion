### 발견사항

**[INFO]** Merge 노드 입력 모델의 아키텍처적 단순화 — 전략적으로 올바른 방향
- 위치: `index.ts:39`, `spec/4-nodes/1-logic-nodes.md:486`
- 상세: `in_0`, `in_1`, ...의 동적 다중 포트 모델에서 단일 `in` 포트 + 다중 엣지 수신 모델로 전환했다. 이 접근은 포트 수 관리 책임을 노드 설정에서 엣지 연결 레이어로 이전하며, 포트 ID 안정성을 높인다. 그러나 이 변경은 런타임에 "다중 엣지 → 객체/배열 입력 정규화" 책임이 실행 엔진에 암묵적으로 부과됨을 의미한다.
- 제안: 실행 엔진 스펙(`spec/5-system/4-execution-engine.md`)에 단일 포트 다중 엣지 입력의 정규화 계약(입력이 `{ [sourceNodeId]: data }` 객체로 전달된다는 것)을 명시적으로 문서화해야 한다.

---

**[WARNING]** `merge.handler.spec.ts`의 입력 정규화 계약이 스펙 문서와 불일치
- 위치: `merge.handler.spec.ts:72-78`, `spec/4-nodes/1-logic-nodes.md:486`
- 상세: 테스트는 입력이 `{ nodeA: data, nodeB: data }` 형태의 소스 노드 ID 키 객체라고 가정한다. 그러나 스펙에는 "단일 `in` 포트, 다중 엣지 수신 가능"만 명시되어 있고, 실행 엔진이 MergeHandler에 어떤 포맷으로 데이터를 전달하는지가 정의되어 있지 않다. 스펙과 구현 사이의 계약 공백이다.
- 제안: 스펙의 Merge 노드 섹션 또는 실행 엔진 스펙에 "다중 엣지 입력은 `{ [sourceNodeId]: outputData }` 형태로 핸들러에 전달된다"는 인터페이스 계약을 명시하고, `ExecutionContext` 또는 핸들러 인터페이스에 이를 타입으로 반영해야 한다.

---

**[WARNING]** `MergeConfig` 컴포넌트에서 `inputCount` 필드 제거 시 기존 저장된 워크플로우 데이터와의 하위 호환성 미처리
- 위치: `logic-configs.tsx:550`
- 상세: 기존 사용자 워크플로우에 `inputCount` 필드가 포함된 노드 설정 데이터가 존재할 수 있다. 프레젠테이션 레이어(Config UI)에서 필드를 제거했지만, 실행 엔진 핸들러(`merge.handler.ts`)나 워크플로우 저장/로드 레이어에서 이 레거시 필드를 무시하거나 마이그레이션하는 처리가 있는지 확인이 필요하다.
- 제안: `MergeHandler.validate()`에서 `inputCount` 필드가 존재하더라도 오류를 발생시키지 않는지 검증하는 테스트를 추가하라. 또한 데이터 마이그레이션 또는 워크플로우 로드 시 레거시 config 정규화 로직이 필요한지 검토하라.

---

**[INFO]** 테스트에서 `strategy: append`가 `wait_all`과 동일하게 처리되는 점이 스펙과 일치하지 않음
- 위치: `merge.handler.spec.ts:106-112`, `spec/4-nodes/1-logic-nodes.md:477`
- 상세: 스펙은 `append` 전략을 "도착 순서대로 배열에 추가, 모든 입력 도착 후 출력"으로 정의하고 있어 `wait_all`과 의미적으로 다르다(`append`는 순서가 보장되어야 함). 테스트 주석 "should return all inputs same as wait_all"이 이 차이를 무시하고 있다. 단일 포트 모델에서 도착 순서를 추적할 수 없으므로 두 전략의 차별점이 사실상 사라졌다면 스펙에서 `append`를 제거하거나 재정의해야 한다.
- 제안: 단일 포트 모델에서 `append`의 의미를 명확히 재정의하거나, 실질적 차이가 없다면 스펙에서 전략을 통합하여 불필요한 복잡도를 제거하라.

---

**[INFO]** `outputFormat: indexed` 테스트의 키 명명이 구현의 암묵적 결정을 노출
- 위치: `merge.handler.spec.ts:135-140`
- 상세: 테스트는 `{ in_0: 'first', in_1: 'second' }` 형태를 기대한다. `in_N` 키 명명 규칙이 스펙 문서(`spec`)에 정의되어 있으나(`{ "0": input0, "1": input1 }` → 스펙은 숫자 키인 반면 테스트는 `in_` 접두사를 포함), 스펙과 테스트 간에 불일치가 있다.
- 제안: 스펙의 indexed 출력 형식 예시(`{ "0": input0, "1": input1 }`)를 `{ "in_0": input0, "in_1": input1 }`로 수정하거나, 테스트의 기대값을 스펙에 맞게 수정하여 단일 진실의 원천(single source of truth)을 유지하라.

### 요약

이번 변경은 Merge 노드의 입력 포트 모델을 동적 다중 포트에서 단일 포트 + 다중 엣지 수신 방식으로 전환한 아키텍처 결정으로, 포트 ID 안정성 확보와 설정 복잡도 감소 측면에서 올바른 방향이다. 그러나 이 변경은 "다중 엣지 입력이 핸들러에 어떤 형태로 전달되는가"라는 실행 엔진 레이어의 핵심 계약을 스펙 문서에서 누락하고 있으며, 테스트 코드가 이 암묵적 계약(`{ [sourceNodeId]: data }` 객체)에 의존하고 있어 명시적 문서화가 필요하다. 또한 `append` 전략의 스펙 재정의, `indexed` 포맷의 키 명명 불일치, 레거시 `inputCount` 데이터 하위 호환성 처리 등 스펙-구현-테스트 삼각형의 일관성 확보가 요구된다.
