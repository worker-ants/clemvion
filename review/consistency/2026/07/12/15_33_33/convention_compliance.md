### 발견사항

- **[WARNING]** `locale` EN 활성화 기능의 "구현 상태" 고지가 3개 대상 문서에서 불일치 — 두 곳은 미구현임을 숨긴 현재형 서술
  - target 위치: `spec/7-channel-web-chat/2-sdk.md` §4 boot config 스키마 하단 note("위젯이 boot 시 1회 해석해 chrome 문자열을... 그 언어로 **렌더한다**") 및 Rationale `### R6`("위젯이 `locale` 을 boot 시 1회 해석... chrome 문자열을... **렌더한다**") · `spec/7-channel-web-chat/5-admin-console.md` §4("운영자가 `en` 을 고르면 위젯 chrome 이 EN 으로 **렌더된다**")
  - 위반 규약: `spec/conventions/spec-impl-evidence.md` §3 (`status` 라이프사이클 — `implemented` = "모든 약속 구현 완료") 및 §2.1 `pending_plans` 의무("status: partial 시 ✓") — 넓게는 본 컨벤션 Overview 가 명시한 "spec 약속 vs 구현 부재 갭 차단"이라는 목적 취지
  - 상세: 같은 커밋 배치(오늘 2026-07-12, `plan/in-progress/spec-draft-webchat-en-i18n.md` §4 Edit B/D)로 함께 수정된 `1-widget-app.md` §4 는 스스로 "본 §4 는 spec 확정분이며 위젯 로컬 catalog·`resolveLocale`·chrome 키 치환·parity 테스트의 **코드 구현은 착수 예정**"이라고 명시하고, `spec/0-overview.md` §6.1 도 "위젯 chrome 문자열 EN 다국어화(`locale` 활성)는 2026-07-12 spec 확정·**구현 착수 예정**인 신규 마일스톤"이라고 동일하게 고지한다. 반면 `2-sdk.md`·`5-admin-console.md` 두 곳은 캐치되지 않은 현재형("렌더한다"/"렌더된다")으로 서술해, 이 두 문서만 읽는 독자(또는 그 부분만 구현 착수하는 개발자)는 EN chrome 렌더링이 이미 동작한다고 오인할 수 있다. 실제로 `codebase/channel-web-chat/src` 에는 `resolveLocale`/로컬 i18n catalog 구현이 전혀 없다(grep 결과 0건). 6개 문서 모두 frontmatter 는 `status: implemented` 이고 `pending_plans:` 필드가 없어, spec-impl-evidence.md §3 의 엄격한 정의(`implemented`="모든 약속 구현 완료", 미완이면 `partial`+`pending_plans:` 의무)와 문언상 어긋난다. 다만 이 "coarse-grained status + 본문 캐비엇" 패턴은 이미 `3-auth-session.md` §3.1("v1 구현 현황(부분)")에서도 선례가 있고, `spec-impl-evidence.md` R-1 Rationale 도 "글로브 stale 은 본 가드로 검출 불가 — `/spec-coverage` 가 보완"이라고 이 한계를 인지하고 있어, 완전히 새로운 위반이라기보다는 **캐비엇 전파가 6개 문서 중 4곳(0-overview·1-widget-app·plan 자신)에는 있고 2곳(2-sdk·5-admin-console)에는 빠진 불일치**가 핵심 문제다.
  - 제안: (a) 2-sdk.md §4 note·R6, 5-admin-console.md §4 note 에도 1-widget-app.md §4 와 동일한 "구현 상태(착수 예정)" 캐비엇을 추가해 6개 피영향 문서 전체의 서술 시제를 일치시키거나, (b) `pending_plans: [plan/in-progress/spec-draft-webchat-en-i18n.md]` 를 `1-widget-app.md`(및 필요 시 `2-sdk.md`)에 추가하고 구현 기간 동안 `status: partial` 로 낮췄다가 developer 구현 완료 커밋에서 `implemented` 로 재승격(§3.1 전이 규칙)하는 편이 spec-impl-evidence.md 문언에 더 부합한다. 최소한 (a) 는 이번 `--impl-prep` turn 안에서 바로 반영 가능한 저비용 수정이다.

### 요약
`spec/7-channel-web-chat/**` 은 정식 규약 대비 전반적으로 잘 정렬돼 있다 — 파일 네이밍(`0-` prefix·`_product-overview.md`·`id:` kebab-case + 영역 접두 충돌 회피)과 frontmatter `code:` 글로브(전량 실존 확인)는 `spec-impl-evidence.md` 를, 응답 DTO 위치(`dto/responses/*-response.dto.ts`)·`{ data }` 래핑 서술은 `swagger.md` §5-1·§2-5 를, `interactionType`/`waitingNodeId` 등 SSE wire 필드 매핑은 `interaction-type-registry.md` §1.1 의 4→3 통합 규칙을, `WEBCHAT_IDLE_TIMEOUT` 같은 에러 코드 명명은 `error-codes.md` §1 의 도메인 prefix 원칙을, 위젯 chrome i18n 스코프(메인 앱 dict 밖 + 위젯 로컬 catalog + ko/en parity)는 `i18n-userguide.md` 적용 범위 절을 정확히 따른다(사실 이 컨벤션 문서 자체가 오늘 같은 배치로 갱신되어 완전히 동기화돼 있다). 유일하게 발견한 실질적 이슈는 오늘 신설된 `locale` EN 활성화 관련 서술에서, 6개 피영향 문서 중 `1-widget-app.md`·`0-overview.md`(및 plan 자신)는 "구현 착수 예정"임을 명시하는 반면 `2-sdk.md`·`5-admin-console.md` 는 동일 미구현 동작을 현재형으로 서술해 문서 간 "구현 상태" 고지가 불일치하는 것이며, 이는 `spec-impl-evidence.md` 가 막으려는 "spec 약속 vs 구현 부재" 갭의 축소판이다.
