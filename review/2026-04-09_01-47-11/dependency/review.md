### 발견사항

- **[INFO]** 문서 간 내부 의존성은 일관성 있게 동기화됨
  - 위치: `prd/3-node-system.md`, `prd/6-phase2-ai.md`, `spec/4-nodes/3-ai-nodes.md`
  - 상세: ND-AG-17/19/23/24 변경사항이 세 문서 모두에 동일하게 반영됨. `timeout` 포트 제거, `cond_`/`tool_` 접두사 규칙, 포트 구조 변경이 PRD → Spec 방향으로 일관됨
  - 제안: 없음

- **[WARNING]** 구현 코드의 `timeout` 포트 참조 잔존 가능성
  - 위치: `spec/4-nodes/3-ai-nodes.md` — `endReason` enum에 `timeout`이 여전히 유효한 값으로 남아 있음
  - 상세: 문서는 "프론트엔드에서는 별도 포트 없이 `error` 포트로 통합 라우팅됨"이라 명시했지만, 기존 구현 코드(backend 실행 엔진, frontend 포트 렌더링 로직)에서 `timeout` 포트를 하드코딩하여 참조하는 코드가 있다면 런타임 라우팅 오류 발생 가능
  - 제안: 구현 코드에서 `'timeout'` 포트 참조를 검색하여 `'error'`로 마이그레이션 필요 여부 확인

- **[WARNING]** Multi Turn 조건 0개 시 하위 호환 정책의 의존성 불일치 가능성
  - 위치: `spec/4-nodes/3-ai-nodes.md:138` — "Multi Turn: `out` (기본 출력만 제공하여 기존 엣지 유지)"
  - 상세: 조건이 0개인 Multi Turn 모드에서만 `out` 포트를 제공하는 것은 기존 워크플로우 데이터(DB에 저장된 엣지)와의 하위 호환을 위한 것임. 그러나 이 예외 케이스가 포트 생성 로직, 실행 엔진, 프론트엔드 포트 렌더러 3곳 모두에 구현되어야 하며, 어느 한 곳이라도 누락 시 기존 워크플로우가 깨짐
  - 제안: 포트 생성 로직과 실행 엔진에 해당 분기가 명시적으로 구현되었는지 확인

- **[INFO]** `cond_`/`tool_` 접두사 규칙과 LLM API 호환성
  - 위치: `spec/4-nodes/3-ai-nodes.md` — "정제(sanitize): UUID 내 `-` 등 비영숫자 문자를 `_`로 치환"
  - 상세: OpenAI, Anthropic 등 LLM API는 도구 이름에 대해 `/^[a-zA-Z0-9_-]{1,64}$/` 형태의 제약을 두는 경우가 있음. `cond_` + 정제된 UUID 조합은 최대 길이(`5 + 32 + 4 = 41자`)로 64자 제한 내에 안전하게 들어옴. 문서 내 명시는 적절함
  - 제안: 없음

- **[INFO]** `_turnDebugHistory` 신규 출력 필드 추가
  - 위치: `spec/4-nodes/3-ai-nodes.md` 신규 섹션
  - 상세: 실행 결과에 `_turnDebugHistory`가 추가되었으나, 이 필드를 소비하는 프론트엔드 Conversation Inspector와의 계약(인터페이스)이 문서에만 정의됨. 기존 출력 스키마 검증 로직이 있다면 새 필드를 허용하도록 업데이트 필요
  - 제안: `_turnDebugHistory`가 하위 노드로 pass-through 될 경우 불필요한 데이터 노출 방지를 위해 출력 시 해당 필드를 제거하는 정책을 스펙에 명시할 것을 권장
