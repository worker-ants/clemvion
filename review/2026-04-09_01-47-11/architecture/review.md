## 아키텍처 코드 리뷰 결과

### 발견사항

---

**[INFO]** 도구 이름 네임스페이스 분리 (`cond_` / `tool_`) 도입
- 위치: spec/4-nodes/3-ai-nodes.md — Tool Area 연동, 조건 도구 등록 섹션
- 상세: UUID 원시값 그대로 사용하던 것에서 접두사 기반 네임스페이스로 전환. LLM API 레이어에서 `cond_` / `tool_` 구분으로 역할 식별이 명확해졌다.
- 제안: 향후 새 도구 유형이 추가될 때(예: `kb_`, `flow_`) 동일 패턴으로 확장 가능하므로, 접두사 레지스트리를 스펙 내 별도 표로 관리하면 좋다.

---

**[WARNING]** `endReason` enum과 라우팅 포트 간 개념 불일치
- 위치: spec/4-nodes/3-ai-nodes.md:395 — `endReason` enum 정의 및 주석
- 상세: `endReason`에 `timeout`이 여전히 유효한 값으로 존재하지만 포트는 `error`로 통합되었다. 이로 인해 데이터 레이어(출력 페이로드)와 라우팅 레이어(포트 결정 로직)의 의미론적 모델이 분기된다. 구현 시 `endReason === 'timeout'` 케이스를 `error` 포트로 매핑하는 별도 변환 로직이 필요해지고, 이 변환이 어느 레이어에서 수행되어야 하는지 스펙이 명시하지 않는다.
- 제안: 두 가지 방향 중 하나를 선택해야 한다. (A) `endReason` enum에서도 `timeout`을 제거하고 `error`로 통합 — 일관성 극대화. (B) `endReason`은 내부 진단 코드로 유지하고, 별도 `routeTo` 필드로 포트 결정을 분리 — 진단 정보와 라우팅 결정의 책임 분리. 현재처럼 한 필드가 두 역할(진단 + 라우팅)을 겸하는 것은 단일 책임 위반이다.

---

**[WARNING]** Multi Turn 조건 0개 케이스의 하위 호환 정책이 비대칭
- 위치: spec/4-nodes/3-ai-nodes.md — 포트 / 공통 섹션, prd/3-node-system.md ND-AG-24
- 상세: 조건 0개일 때 Single Turn은 `out + error`(기존과 동일), Multi Turn은 `out`만 제공한다. Multi Turn에서 `error` 포트가 생략된 근거가 스펙에 없다. 노드 실행 중 LLM 오류가 발생하면 라우팅 불가 상태가 되어 워크플로우가 비정상 종료될 수 있다. 하위 호환을 위해 `out`만 남긴 것이라면 `error` 포트도 함께 유지해야 한다.
- 제안: Multi Turn 조건 0개 케이스도 `out + error`로 통일하거나, 조건 0개를 지원하지 않는 방향으로 정책을 명확히 한다.

---

**[INFO]** `_turnDebugHistory` 필드의 레이어 책임 위치
- 위치: spec/4-nodes/3-ai-nodes.md — 디버그 데이터 섹션
- 상세: 디버그 데이터가 정상 출력 페이로드에 `_`접두사 필드로 포함된다. 디버깅 목적 데이터가 런타임 출력 스키마에 혼재하면, 다운스트림 노드에서 이 필드를 의도치 않게 참조하거나 전달하는 오용이 발생할 수 있다.
- 제안: 디버그 데이터를 별도 사이드 채널(예: 실행 로그 레코드, 실행 컨텍스트 메타데이터)로 분리하고, 노드 출력 페이로드에는 포함하지 않는 구조를 권장한다.

---

**[INFO]** 조건 유효성 검증 규칙의 위치가 실행 스펙에 혼재
- 위치: spec/4-nodes/3-ai-nodes.md — 유효성 검증 규칙 섹션
- 상세: 새로 추가된 유효성 검증 규칙(최대 20개, 예약 ID 충돌 불가, prompt 최대 2,000자 등)이 실행 로직 섹션 중간에 삽입되어 있다. 검증은 입력 경계에서 처리되어야 하며, 실행 흐름 설명과 분리하여 명시하는 것이 가독성과 구현 추적성에 유리하다.
- 제안: 설정(config) 섹션 또는 별도 `### 검증 규칙` 섹션으로 분리한다.

---

**[INFO]** `reason` 필드 잘림 처리(500자)의 책임 레이어 미명시
- 위치: spec/4-nodes/3-ai-nodes.md — 유효성 검증 규칙, `reason` 응답
- 상세: LLM이 반환한 `reason` 값을 500자로 잘리는 처리가 어느 레이어에서 수행되는지(백엔드 실행 엔진, LLM 응답 파서, 출력 직렬화) 명시되지 않았다.
- 제안: 실행 로직 섹션의 tool call 처리 단계에 "조건 도구 응답에서 `reason` 필드를 500자로 truncate" 항목을 명시한다.

---

### 요약

이번 변경은 `timeout` 포트를 `error`로 통합하고, 도구 이름에 `cond_`/`tool_` 접두사 네임스페이스를 도입하여 포트 구조를 단순화한 설계 개선이다. 전반적인 방향성은 올바르나, `endReason` enum이 포트 라우팅과 진단 두 역할을 겸하는 단일 책임 위반이 구현 복잡도를 높일 수 있다. Multi Turn 조건 0개 케이스에서 `error` 포트 누락도 런타임 안전성 관점에서 보완이 필요하다. 디버그 데이터를 출력 페이로드에 혼재시키는 패턴은 레이어 경계를 약화시키므로 사이드 채널로 분리를 권장한다.
